# Функции. Структуры и методы. Интерфейсы


## Что сложного в функциях

Всё это время мы использовали функции как единицы логического построения своих программ. Использовали аргументы в качестве входных параметров, иногда возвращали из функции результаты её работы. И для тебя сигнатура функций выглядела довольно обычно (напомним, что сигнатура — это наименование функции, её входные и выходные параметры). Но и такой сущностью, как функция, можно проделывать интересные вещи.

Начнём с того, что функции в Go — это объекты первого класса. Что это означает для нас? А то, что ты с функцией можешь обращаться как с любой переменной: переприсваивать, передавать в качестве аргумента, возвращать из функции. Простейший пример:

```go
func main() {
    p := prints    // Не указываем аргументы функции - тогда она присваивается переменной
    fmt.Println(p) // Увидим адрес
}

func prints(str string) {
    fmt.Println(str)
}
```

Пример даёт нам понять, что функции своего рода являются ссылочными типами. И мы можем присвоить функцию любой переменной в любом месте и далее передавать её в зависимости от потребности, не вызывая её до необходимого момента. Но что, если нам необходимо объявить функцию в конкретном месте и ее жизнь отдельной сущностью нам не требуется? Тут на помощь приходят **анонимные функции**.

**Анонимные функции** потому и называются анонимными, что мы их никак не обзываем — они рождаются в нужном месте и именно там выполняют свою работу. Но, несмотря на то что они рождаются без имени, их необходимо присвоить какой-то переменной. Выглядят анонимные функции так:

```go
f := func(a int, b int) int {
    return a + b
}

result := f(2, 3)
fmt.Println(result) // 5

```

Как мы видим, тут есть два существенных отличия от обычных функций:
- нет имени;
- объявление прямо в коде.

А ещё мы можем вызвать такую функцию сразу и присвоить её результат переменной:

```go
result := func(x int) int {
    return x * x
}(4)

fmt.Println(result) // 16

```

Но зачем нам функции подобного плана? По большей части для трех вещей: замыканий, колбеков и локальной логики при построении конкурентного кода. Поговорим сейчас только о первой.

### Замыкания

**Замыкания** или *closure* — это анонимная (зачастую) или всё-таки именованная функция, которая использует переменные из внешней области видимости (то есть те переменные, которые не находятся внутри самого тела функции). В случае замыкания функция *не копирует* значение переменной, а захватывает её, воздействуя в течение своей жизни на состояние этой переменной.
Чтобы было легче, давай снова посмотрим на пример:

```go
func counter() func() int {
    i := 0

    return func() int {
        i++
        return i
    }
}
```

Что здесь происходит? Мы объявили анонимную функцию и вернули её из `counter` (функция — объект первого класса, помнишь?); но тело анонимной функции содержит переменную `i`, которая находится вне её тела, однако анонимная функция всё равно видит эту переменную, так как они сами находятся в одной области видимости. Как будет работать анонимная функция, когда мы будем ей пользоваться? Она вернёт нам одни нули? Или одни единицы?

```go
c := counter()

fmt.Println(c()) // 1
fmt.Println(c()) // 2
fmt.Println(c()) // 3

```

Используя литерал функции `c`, ты сможешь вызывать её столько раз, сколько надо. А переменная `i` будет ровно столько раз увеличивать свое значение. Таким образом, здесь мы можем взаимодействовать через замыкание с данными, которые уже находятся вне нашего внимания!

Но тут нужно спросить: *могу ли я вызвать функцию `counter` несколько раз подряд? Я получу одно и то же замыкание?* Ответ: да, ты можешь несколько раз вызвать `counter` и получить несколько замыканий; и нет, ты получишь разные замыкания, которые будут работать независимо друг от друга.

Ещё важно уточнить, что замыкания захватывают не только переменные, но и параметры функции, как в примере:

```go
func multiplier(factor int) func(int) int {
    return func(x int) int {
        return x * factor
    }
}

double := multiplier(2)
triple := multiplier(3)

fmt.Println(double(5)) // 10
fmt.Println(triple(5)) // 15

```

В этом случае замыкание захватило параметр `factor`, увеличивая его на то значение, которое потом передаётся самому замыканию. Вот такие вот дела с обычными функциями!

### Именованный возврат

Кстати, не обязательно создавать переменные, которые хочешь вернуть, в теле функции. Ты можешь сделать это прямо в сигнатуре, и тогда переменные уже будут созданы на этапе компиляции.
Сравни:

```go
func calc(first int, second int) (int, int) {
  sum := first + second
  product := first * second

  return sum, product
}


func calc(first int, second int) (sum int, product int) {
  sum = first + second
  product = first * second

  return
}

```

В первом случае ты создаёшь результат и возвращаешь его явно. Во втором — контейнеры для данных у тебя уже готовы. Когда стоит выбрать один способ, а когда второй? **Главное правило**: *твой код должен быть понятен тем, кто его будет читать*. Поэтому именованный возврат хорош для маленьких функций, но плох в случаях, когда в функциях накручено много логики или же функция уже не умещается в экран.
Тебе придётся читать свой же код и держать в уме тот факт, что в сигнатуре есть какие-то возвращаемые значения. И, читая код, ты будешь постоянно о них забывать и возвращаться к сигнатуре. Так что проявляй внимательность при выборе способа возврата значений.


## Структуры: группировка данных и никакого ООП

Мы уже видели структуры. Видели в исходном коде языка. Но что же это такое?

**Структуры** — составной тип данных, который позволяет объединять связанные переменные в одну единицу данных. Структура состоит из именованных полей одного или разных типов и помогает организовывать связанные данные в одном контейнере.

```go
type Employee struct {
    Name     string
    Age      int
    Position string
    Salary   int
}
```

В данном случае объявлена структура типа `Employee`. Важно помнить, что при объявлении структуры её имя является частью типа.

Создавать структуры мы можем тоже разными способами. Например, через присваивание переменной с чётким именованием полей (на самом деле какое-то из полей можно опустить, и оно заполнится `zero value` своего типа или же `nil` для ссылочных типов):

```go
emp1 := Employee{
    Name:     "Анна Иванова",
    Age:      28,
    Position: "Разработчик",
    Salary:   120000,
}
```

Или же можно создать структуру в более кратком виде — без уточнения имени полей. **Но** поля должны идти по порядку:

```go
emp2 := Employee{"Петр Петров", 32, "Тестировщик", 100000}
```

И ещё ты можешь создать пустую структуру (все её поля заполнятся `zero value`), а затем заполнять поля по мере надобности:

```go
var emp3 Employee
emp3.Name = "Мария Сидорова"
emp3.Age = 25
emp3.Position = "Аналитик"
emp3.Salary = 110000

fmt.Println(emp3.Name)  // Мария Сидорова
```

Именно так — через точку — ты можешь обращаться к полям для того, чтобы прочесть или записать данные.

## Важные особенности структур

1. Структуры, как и остальные типы, можно передавать в функции и возвращать их оттуда. Причём, как и для всех типов, для структур действует правило, что они передаются в функцию по значению, то есть полностью копируются, и изменения, выполненные над структурой, окажутся не видны вне тела функции. Так что, если захочешь что-то изменить в структуре внутри функции, **явно возвращай её**. Или передавай указатель на структуру:

```go
// Эта функция НЕ изменит оригинальную структуру!
func tryToIncreaseSalary(emp Employee, amount int) {
    emp.Salary += amount // Изменяем только копию

    fmt.Printf("Внутри функции зарплата: %d\n", emp.Salary)
}

// Эта функция изменит оригинальную структуру
func increaseSalary(emp *Employee, amount int) {
    emp.Salary += amount // Изменяем оригинал через указатель — изменения будут видны вне функции
}
```

И тут мы натыкаемся на некий *trade-off*: хоть идиоматически верно передавать параметры в функцию по значению, большие структуры (примерно >10 MB) копировать будет дорого, поэтому **лучше** передавать их уже по указателю; но передача по значению гарантирует, что функция не изменит без надобности нашу структуру. Так что здесь стоит хорошо подумать, прежде чем выбирать какой-то из путей.

2. Структуры можно вкладывать друг в друга (так называемые nested-структуры), так как их поля могут быть любыми типами, даже другими структурами. Такой механизм называется **композицией структур**:

```go
type Address struct {
    Street string
    City   string
    Zip    string
}

type Person struct {
    Name    string
    Age     int
    Address Address // Вложенная структура
}
```

Таким образом ты можешь составлять иерархию структур, данных, характеристик.

Частным случаем композиции является **встраивание (embedding)**, когда поле не имеет имени. В таком случае все поля и методы (о которых ниже) встроенной структуры автоматически подтягиваются к родительской структуре, от чего их можно вызывать напрямую, без обращения к встроенной структуре:

```go
type Contact struct {
    Email string
    Phone string
}

type Employee struct {
    Name     string
    Position string
    Contact  // Встроенная структура (анонимное поле)
}

func main() {
    emp := Employee{
        Name:     "Анна Иванова",
        Position: "Разработчик",
        Contact: Contact{
            Email: "anna@company.com",
            Phone: "+7-123-456-7890",
        },
    }

    // Доступ к полям встроенной структуры (короткий синтаксис)
    fmt.Println(emp.Email) // anna@company.com
    fmt.Println(emp.Phone) // +7-123-456-7890
    
    // Доступ к полям встроенной структуры (полный синтаксис) - мы все ещё можем использовать тип структуры как явное обращение к ней
    fmt.Println(emp.Contact.Email) // anna@company.com - также работает
    fmt.Println(emp.Contact.Phone) // +7-123-456-7890
}
```

Здесь возникает соблазн обозвать такой механизм *наследованием*, как в ООП, но это **не наследование** — вместо отношения «является» (is-a) мы создаём отношение «содержит» (has-a) с удобным синтаксисом доступа.

Но тут сходу возникает важная проблема — *конфликт имён*. Что, если обе структуры будут иметь поля с одинаковым именем? В таком случае поле родительской структуры будет **скрывать** поле встроенной. Но есть ещё несколько корнер-кейсов в правилах разрешения конфликтов имён:

- Имена полей, объявленные в самой структуре, имеют приоритет над именами полей встраиваемых структур.
- Если одинаковое имя встречается только у встроенных структур, выбирается поле/метод с меньшей глубиной (ближе к внешней структуре).
- Если одинаковые имена приходят из нескольких встраиваний на одной глубине, обращение не по полному пути приведёт к ошибке компиляции из-за неоднозначности. Надо явно указывать, от какой именно структуры вам нужно поле.

3. Структуры **сравниваемы**! Но только если все поля конкретной структуры сравниваемы и одинаковы по типам у разных структур. Если у одной из структур есть поле типа слайс, такая структура не может быть сравниваема через операторы `==` и `!=`.

## Кастомные типы

Пока мы далеко не ушли от структур, стоит сказать, что ты можешь создавать свои типы поверх уже существующих в Golang. Это так называемые кастомные типы:

```go
type Stringz []string
type Address string
type FullAddress Address
type EveryDay int
```

Ты можешь обозвать новый тип поверх существующего встроенного (даже массива или слайса), а можешь создать кастомный тип от кастомного типа. Это бывает удобно, когда тебе необходимо отличать конкретные бизнес-сущности от уже существующих встроенных типов или же когда тебе требуется прикрутить к типу конкретное поведение, о чём поговорим ниже.


## Методы структур

Теперь мы готовы объединить знания по функциям и типам в единую сущность — методы. **Метод** — это функция, которая принадлежит конкретному типу и реализует логику, связанную только с ним. Этим типом может быть структура или любой кастомный тип. Метод отличается от функции именно этим — принадлежностью к чему-то конкретному. И эта принадлежность обозначается существованием так называемого ресивера (получателя) между `func` и наименованием функции:

```go
type Status string  // Объявим кастомный тип

// Добавим к типу поведение — возвращать представление типа в виде базового типа (строки)
func (s Status) String() string { // s - ресивер типа Status, который связывает метод и тип
    return string(s)  // Приведение кастомного типа к базовому
}
```

Или вот пример со структурой:

```go
type Rectangle struct {
    Width  float64
    Height float64
}

// Метод с получателем по значению
func (r Rectangle) Area() float64 {
    return r.Width * r.Height
}

// Метод с получателем по указателю
func (r *Rectangle) Scale(factor float64) {
    r.Width *= factor
    r.Height *= factor
}

func main() {
    rect := Rectangle{Width: 10, Height: 5}

    fmt.Println("Площадь:", rect.Area())
    fmt.Println(rect) // cтруктура ДО вызова Scale

    rect.Scale(2)
    fmt.Println("После масштабирования:", rect)
    fmt.Println("Новая площадь:", rect.Area())
}
```

### Получатель по значению vs по указателю

Обрати внимание, что в примере со структурой ресивер используется в двух состояниях: он существует как параметр, переданный по значению и по указателю. Наверное, ты уже можешь самостоятельно предположить, что же это значит, но давай проговорим.

**Получатель по значению** — метод получает копию структуры:
- Используется когда методу не требуется изменять исходные поля структуры.
- Безопасен для конкурентного доступа.
- Может быть дороже для больших структур из-за копирования.

**Получатель по указателю** — метод получает указатель на структуру:
- Используется когда методу требуется изменять исходные поля структуры.
- Избегает копирования, эффективен для передачи больших структур.
- Требует осторожности при конкурентном доступе.

Поэтому использовать получатели по указателю стоит только тогда, *когда тебе надо изменить состояние этого получателя*. Но существует одно важное правило: как только у одного из методов твоего типа появляется метод с ресивером по указателю, **стоит** поменять ресиверы в остальных методах так же на передачу по указателю.
Этого правила стоит придерживаться для того, чтобы не допустить смешения типов, приводящего к ошибкам:
- Теряется согласованность (консистентность) — указатели во всех методах гарантируют, что ты будешь работать с одним и тем же экземпляром данных, без его копирования.
- Получаем ожидаемую изменяемость — последовательное использование указателей предотвращает работу с копиями, что может быть критично для производительности.
- Набор методов с одним типом ресивера для реализации интерфейсов (мы почти добрались до этой темы тоже) позволяет интерфейсам корректно работать со всем набором методов.

### Публичные и приватные методы

Как и обычные функции, методы структур могут быть **публичными** (доступными из других пакетов) или **приватными** (доступными только внутри текущего пакета). В терминологии Go это называют **экспортируемыми** и **неэкспортируемыми** идентификаторами. Правило простое: если имя метода начинается с заглавной буквы — метод **экспортируемый**, если с маленькой — **неэкспортируемый**.

**Практическая польза**
- **Экспортируемые методы** формируют публичный API твоего типа — то, как другие пакеты могут взаимодействовать с твоей структурой.
- **Неэкспортируемые методы** инкапсулируют внутреннюю логику, которую не нужно выставлять наружу.
- Это позволяет изменять внутреннюю реализацию, не ломая код, который использует экспортируемые методы вашей структуры.

### Конфликт методов при embedding

Как и с полями, при встраивании структур могут возникать конфликты методов. Go применяет те же правила приоритета, что и для полей:

1. **Продвижение методов**: методы встроенной структуры автоматически становятся доступны у внешней (они «продвигаются»).
2. **Приоритет внешней**: метод, объявленный у внешнего типа, скрывает одноимённый метод встроенной структуры.
3. **Множественное встраивание и неоднозначность**: если два встроенных типа на одной глубине имеют одинаковый метод, обращение `fh.Write()` без уточнения — ошибка компиляции. Используй явный путь.


## Интерфейсы: определение поведения

**Интерфейс** в Go — это абстрактный тип, который определяет набор конкретных методов. Иными словами интерфейсы описывают поведение, а не данные. А у Golang, в отличие от других популярных языков, существует правило *утиной типизации*: если что-то ходит как утка, крякает как утка, летает как утка, то это утка. Простое и забавное правило, которое в контексте языка говорит нам о следующем: если какой-то тип имеет набор методов, указанных в некоем интерфейсе, то этот тип будет автоматически удовлетворять интерфейсу. Без явного указания, например, через `impl`.

```go
type Shape interface { // Объявление интерфейса, содержащего два метода
    Area() float64
    Perimeter() float64
}

type Rectangle struct {  // Структура, которая реализует интерфейс, даже несмотря на то, что имеет три, а не два метода
    Width  float64
    Height float64
}

func (r Rectangle) Area() float64 {
    return r.Width * r.Height
}

func (r Rectangle) Perimeter() float64 {
    return 2 * (r.Width + r.Height)
}

func (r Rectangle) Info() {
    fmt.Println("Width: ", r.Width)
    fmt.Println("Height: ", r.Height)
}
```

### Использование интерфейсов

С практической точки зрения интерфейсы очень важны по крайней мере для двух вещей:
- избавление кода от множества конкретных зависимостей;
- как следствие, написание тестов.

В примере ниже функция `printShapeInfo` принимает параметр типа интерфейс, а это сразу даёт нам понять, что функция зависит не от конкретной реализации, а от поведения. Если какой-то тип удовлетворяет этому поведению, то он может быть использован в этой функции, при этом самой функции всё равно, как именно это поведение реализовано внутри — сигнатура одна и та же для каждого типа, который удовлетворяет интерфейсу.

```go
func printShapeInfo(s Shape) {
    fmt.Printf("Площадь: %.2f\n", s.Area())
    fmt.Printf("Периметр: %.2f\n", s.Perimeter())
}

func main() {
    rect := Rectangle{Width: 10, Height: 5}
    circle := Circle{Radius: 3}

    // Один и тот же код работает с разными типами
    fmt.Println("Прямоугольник:")
    printShapeInfo(rect)

    fmt.Println("\nКруг:")
    printShapeInfo(circle)
}
```

### Внутреннее устройство интерфейса

Может быть неожиданно, но интерфейс в исходном коде Go — это снова структура ([ссылка на GitHub](https://github.com/golang/go/blob/go1.25.6/src/runtime/runtime2.go#L178))!

```go
type iface struct {
    tab  *itab           
    data unsafe.Pointer
}
```

Оба поля — это указатели. `itab` — это структура, которая хранит информацию о типах и методах, а именно метаданные: интерфейсный тип, реальный тип и таблицу указателей на методы, которая позволяет объединить одно с другим. `data` указывает на конкретные, реальные данные (значение), которые в данном случае реализует этот интерфейс. Предлагаем тебе поглубже познакомиться с внутренним устройством интерфейсов в источниках в конце лонгрида.

### Пустой интерфейс

**Пустой интерфейс** `interface{}` (или с версии Go 1.18+ `any`) не содержит ни одного метода в своём определении. Поскольку любой тип в Go может иметь ноль методов (это вполне законно), **любой тип удовлетворяет пустому интерфейсу**. Это делает `interface{}/any` универсальным типом, способным хранить значения любого типа!

```go
func printAnything(value interface{}) {
	fmt.Printf("Значение: %v, Тип: %T\n", value, value)
}

func main() {
	printAnything(42)                              // int
	printAnything("Hello")                         // string
	printAnything([]int{1, 2, 3})                  // []int
	printAnything(Rectangle{Width: 10, Height: 5}) // Rectangle
	printAnything(nil)                             // <nil>
}
```

В исходном коде пустой интерфейс тоже присутствует отдельной структурой ([ссылка на GitHub](https://github.com/golang/go/blob/go1.25.6/src/runtime/runtime2.go#L183)):

```go
type eface struct {
    _type *_type
    data  unsafe.Pointer
}
```

Как ты видишь, в ней нет таблицы интерфейсов `itab`, а только тип и конкретное значение, так как у пустого интерфейса нет необходимости проверять его соответствие какому-то набору методов. Но вытащить значение из интерфейса как-то можно? Да.

#### Type Assertion

Для извлечения конкретного типа из значения интерфейса используется **type assertion**, при котором мы пытаемся привести переменную типа интерфейса к интересующему нам типу и проверяем приведение через булевое `ok`:

```go
func processValue(value interface{}) {
    // Безопасная проверка через идиому "comma, ok"
    if str, ok := value.(string); ok {
        fmt.Printf("Это строка: %s (длина: %d)\n", str, len(str))
    } else if num, ok := value.(int); ok {
        fmt.Printf("Это число: %d (квадрат: %d)\n", num, num*num)
    } else {
        fmt.Println("Неизвестный тип")
    }
}

func main() {
    processValue("Hello")
    processValue(42)
    processValue(3.14)
}
```

#### Type Switch

Более элегантный способ работы с разными типами через **type switch**:

```go
func describe(value interface{}) {
    switch v := value.(type) {
    case string:
        fmt.Printf("Строка длиной %d: %s\n", len(v), v)
    case int:
        fmt.Printf("Целое число: %d\n", v)
    case float64:
        fmt.Printf("Число с плавающей точкой: %.2f\n", v)
    case Rectangle:
        fmt.Printf("Прямоугольник: площадь = %.2f\n", v.Area())
    default:
        fmt.Printf("Неизвестный тип: %T\n", v)
    }
}
```

### Ограничения и особенности пустого интерфейса

**1. Потеря статической типизации**
```go
func demonstrateTypeLoss() {
    var x any = 42
    
    fmt.Println(x + 10) // Ошибка компиляции! Go не знает, что x — это число
    
    // Нужно приведение типа через type assertion
    if num, ok := x.(int); ok {
        fmt.Println(num + 10) // 52
    }
}
```

**2. Производительность**
```go
// Медленнее - нужны проверка типов и приведения
func slowSum(values []any) int {
    sum := 0
    for _, v := range values {
        if num, ok := v.(int); ok {
            sum += num
        }
    }
    return sum
}

// Быстрее - работа с конкретным типом
func fastSum(values []int) int {
    sum := 0
    for _, v := range values {
        sum += v
    }
    return sum
}
```

**3. Отсутствие проверки типов на этапе компиляции**
```go
func riskyCode() {
    var data any = "строка"
    
    // Код ниже скомпилируется, но упадёт в runtime!
    number := data.(int) // panic: interface conversion: interface {} is string, not int
    
    // Безопасный вариант
    if number, ok := data.(int); ok {
        fmt.Println("Число:", number)
    } else {
        fmt.Println("Это не число")
    }
}
```

### Пустой и nil-интерфейс

На самом деле поначалу многие путают пустой и nil-интерфейсы. Тут есть важная оговорка: это не одно и то же. О пустом интерфейсе мы уже поговорили: он не имеет набора методов, но не будет иметь значение `nil`. `nil` означает, что интерфейс инициализирован как тип, но не имеет внутри никакого значения, чтобы мы могли сказать, чему удовлетворяет этот интерфейс:

```go
func main() {
    var p fmt.Stringer
    var i any

    fmt.Println(p == nil) // true
    fmt.Println(i == nil) // true

    i = 42

    fmt.Println(i == nil) // false
}
```

### Method set и правила реализации интерфейсов

**Method set** — это набор методов, доступных у значения определённого типа. Понимание method set критически важно для работы с интерфейсами в Go, особенно когда речь идёт о методах с получателями-указателями и получателями-значениями. Существует набор определённых правил.

**Для типа `T` (значение):**
- Method set содержит методы с получателем `(t T)`.
- НЕ содержит методы с получателем `(t *T)`.

**Для типа `*T` (указатель):**
- Method set содержит методы с получателем `(t T)` и `(t *T)`.

```go
type Document struct {
    title string
    content string
}

// Метод с получателем по значению
func (d Document) GetTitle() string {
    return d.title
}

// Метод с получателем по указателю
func (d *Document) SetTitle(title string) {
    d.title = title
}

// Интерфейс только с методом по значению
type Reader interface {
    GetTitle() string
}

// Интерфейс с методами по значению и указателю
type Writer interface {
    GetTitle() string
    SetTitle(string)
}

func main() {
    doc := Document{title: "Заголовок", content: "Содержимое"}
    docPtr := &Document{title: "Заголовок", content: "Содержимое"}
    
    // Document (значение) удовлетворяет Reader
    var r1 Reader = doc    // OK
    var r2 Reader = docPtr // OK (указатель содержит методы значения)
    
    // Writer требует SetTitle (указательный метод)
    var w1 Writer = doc    // Ошибка компиляции: method set T не включает SetTitle
    var w2 Writer = docPtr // OK
}
```

### Автоматическое разыменование

Go автоматически преобразует между указателями и значениями при вызове методов:

```go
type Counter struct {
    value int
}

func (c *Counter) Increment() {
    c.value++
}

func (c Counter) GetValue() int {
    return c.value
}

func main() {
    // Значение
    counter1 := Counter{value: 0}
    counter1.Increment() // Компилятор автоматически подставит (&counter1).Increment()
    
    // Указатель  
    counter2 := &Counter{value: 0}
    value := counter2.GetValue() // Компилятор автоматически подставит (*counter2).GetValue()
    
    fmt.Println(counter1.GetValue(), value)
}
```
Авторазыменование работает только в момент вызова метода, но **не изменяет method set типа**. Поэтому при присваивании интерфейсу учитывается именно method set. Но это **не** работает при присваивании интерфейсам!

### Композиция интерфейсов

Интерфейсы можно комбинировать для создания более сложных контрактов, прямо как у структур:

```go
type Reader interface {
    Read([]byte) (int, error)
}

type Writer interface {
    Write([]byte) (int, error)
}

type Closer interface {
    Close() error
}

// Композиция интерфейсов
type ReadWriter interface {
    Reader
    Writer
}

type ReadWriteCloser interface {
    Reader
    Writer
    Closer
}
```

### Идиомы проектирования интерфейсов

1. **Держи интерфейсы маленькими**. Лучше много маленьких интерфейсов, чем один большой.
2. **Определяй интерфейсы там, где используешь**, а не в пакете с реализацией. Так делается, потому что интерфейс отражает потребности клиента, а не возможности провайдера.
3. **Используй осмысленные имена** — `Writer`, `Reader`, `Validator`.
4. **Принимай интерфейсы — возвращай конкретные типы**. Вызывающему полезны реальные поля/методы результата: абстракция нужна на входе, где ты «потребляешь» поведение.
5. **Не вводи интерфейс «на всякий случай»** — если у тебя единственная реализация и нет места, где нужна подмена, начни с конкретного типа. Интерфейс добавь, когда появится нужда в вариативности.
