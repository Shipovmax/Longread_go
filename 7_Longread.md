# Контекст и примитивы синхронизации. Основные паттерны конкурентного кода

Go силён конкурентностью под капотом. А ещё и теми абстракциями, которые появились в Go из-за этой самой конкурентности. Сегодня поговорим про контекст и примитивы синхронизации, которые помогают управлять горутинами, то есть выполнением асинхронного кода. С одним из примитивов синхронизации мы уже знакомы — это каналы.

## Контекст

**Контекст** — это ещё одна изюминка Go, которая помогает дирижировать программами, имеющими множество горутин. 

О самом контексте можно думать в рамках трех уровней:
- Первый уровень мы используем в повседневной жизни, думая о контексте как об условиях, окружении, которые определяют точный смысл слова или события. В этом плане контекст в Go делает то же самое, помогая формировать вокруг нашего кода некое окружение, которое помогает точно определить рамки выполнения кода.
- На втором уровне мы можем говорить о контексте как о специальном типе данных, который призван управлять жизненным циклом горутин, а значит, и всей программы целиком.
- На третьем уровне контекст — это [интерфейс](https://github.com/golang/go/blob/go1.25.7/src/context/context.go#L71) с определённым перечнем методов, а значит, контекст может быть любым типом, который удовлетворит этому самому интерфейсу.

Контекст позволяет делать в коде три важные вещи:
1. Передавать сигнал отмены между горутинами.
2. Управлять дедлайнами для выполнения блоков кода.
3. Нести в себе метаданные сквозь все слои кодовой базы.

Именно поэтому использование контекста в Go — это не просто принятая практика. Это необходимость, потому что без контекста и трёх его основополагающих функций мы не могли бы красиво передавать метаданные при http- и grpc-запросах в иные сервисы; не могли бы контролировать время запросов в сервисы и базы данных; теряли бы ресурсы и процессорное время на долгих фоновых задачах.

А ведь интерфейс контекста довольно скромный — всего четыре метода:

```go
type Context interface {
    Deadline() (deadline time.Time, ok bool) // Возвращает конкретное время, в которое контекст должен быть отменён, и флаг, который говорит, установлен ли дедлайн или нет
    Done() <-chan struct{} // Возвращает канал, который закрывается при отмене контекста
    Err() error // Возвращает ошибку как причину отмены контекста
    Value(key interface{}) interface{} // Позволяет получить значение из контекста по ключу
}
```

То есть контекст как некоторая сущность связан со временем, а также с сигналами окончания некоторого действия. 

### Типы контекста

Не слишком ли абстрактно говорить о контексте как просто об интерфейсе? На самом деле если посмотреть в пакет контекста глубже, то можно увидеть методы, создающие разные типы контекста, которые мы можем использовать в своём коде, создавая их на лету:

1. `ctx := context.Background()` — создаёт пустой контекст.
2. `ctx := context.TODO()` — тоже создаёт пустой контекст, но используемый в тех случаях, когда мы ещё не уверены, а нужен ли нам тут контекст и если нужен, то какого типа.

Обе эти сущности под капотом представляют собой [структуру](https://github.com/golang/go/blob/go1.25.7/src/context/context.go#L181), которая удовлетворяет интерфейсу `context.Context`. **И здесь нас встречает первое правило работы с контекстом**: *Background-контекст* — начальный контекст, на основе которого мы можем выстраивать целые цепочки из контекстов; *TODO* нельзя использовать в продакшене. Никогда. Этот контекст — заглушка, которая просто помогает нам разрабатывать сервисы и помечать места, в которые мы хотим в дальнейшем подставлять работоспособные контексты.

3. `ctx, cancel := context.WithCancel(ctx)` — возвращает два параметра: контекст и функцию отмены, связанную с ним. Эту функцию можно вызвать в любом участке кода в любой момент времени, и контекст в таком случае отменится, [закрывая](https://github.com/golang/go/blob/go1.25.7/src/context/context.go#L435) канал `Done`. Этот контекст очень полезен в случаях, когда хочется отменить некоторую операцию по выполнении некоторых указанных тобой условий.

4. `ctx, cancel := context.WithTimeout(ctx, 5*time.Second)` — возвращает контекст, который автоматом после истечения некоторого [тайм-аута](https://github.com/golang/go/blob/go1.25.7/src/context/context.go#L664) (5 секунд, 11 минут и любой другой таймер) закрывается и тем самым закрывает и канал `Done`. А ещё мы снова обладаем функцией отмены `cancel`, которой можем закрыть канал до истечения тайм-аута по необходимости. И кстати, под капотом эта функция обращается к функции из пятого пункта. Отличный [пример](https://github.com/golang/go/blob/go1.25.7/src/context/context.go#L703) переиспользования кода.

5. `ctx, cancel := context.WithDeadline(ctx, time.Now().Add(10*time.Minute))` — возвращает контекст, который закрывается после достижения конкретного времени (14:14 МСК). И тут мы тоже имеем возможность завершить его раньше времени через функцию `cancel`.

Из дополнительных возможностей можно использовать функцию `context.AfterFunc(ctx, func() {})`, которая выполнит переданное ей замыкание после завершения контекста.

Также существуют методы с постфиксом `Cause`, которые позволяют добавлять бизнес-причину вызова `cancel`:

```go
ctx, cancel := context.WithCancelCause(context.Background())

go func() {
	time.Sleep(200 * time.Millisecond)
	cancel(errors.New("client closed browser tab"))
}()

<-ctx.Done()
fmt.Println("Err:", ctx.Err())              // context canceled
fmt.Println("Cause:", context.Cause(ctx))   // client closed browser tab
```

### Иерархии контекстов

Прелесть контекстов заключается в том, что из них можно создавать целые иерархии, тем самым дробя выполнение кода через продолжительность операций. Наверное, заметно, что некоторые функции выше принимают в себя контекст. Именно таким образом и выстраивается цепочка из контекстов. Стандартный код — когда сначала создаётся пустой `Background`, а потом от него наследуются разные нужные нам контексты.

```go
ctx := context.Background()

ctx, cancel = context.WithCancel(ctx)

ctx, cancel = context.WithTimeout(ctx, 5*time.Second)
```

Мы именно переиспользуем переменную контекста, потому что под капотом пакета `context` у нас работают структуры-обёртки, как, например, [контекст](https://github.com/golang/go/blob/go1.25.7/src/context/context.go#L432) с отменой, который внутри себя содержит интерфейс контекста, а это значит, что он может содержать любую нашу кастомную структуру, которая удовлетворяет интерфейсу.

Здесь всплывает закономерный вопрос — если у нас имеется иерархия контекстов, то как же влияют отмены контекстов разной степени вложенности друг на друга? **Второе правило работы с контекстом**: если отменился родительский контекст, то отменяются дочерние; если отменился дочерний, то родительский не отменяется.

Но всё-таки есть одно исключение, при котором отмена родительского контекста не влияет на жизненный цикл дочернего контекста. Мы можем использовать `context.WithoutCancel(ctx)`, который вернёт нам дочерний контекст, который не будет отменён вместе с родительскими. Это бывает полезно в случае, когда некоторой операции надо выполниться независимо от того, отменился ли контекст выше или нет.

### Передача значений в контексте

Да, контекст может быть некоторым контейнером, в котором мы можем передавать данные между слоями кода, от сервиса к сервису. Это делается благодаря `ctx := context.WithValue(ctx, key, value)`, который складывает значение по ключу внутрь контекста. И ключ, и значение — это пустой интерфейс. Но как происходит это складывание? 

Если посмотреть на нужную [структуру](https://github.com/golang/go/blob/go1.25.7/src/context/context.go#L742), то станет понятно, что мы вновь складываем структуру в структуру, образуя вложенную иерархию значений по ключам. И мы можем спокойно передавать в контексте метаданные, информацию по аутентификации и прочие сопровождающие запрос данные. 

**Но никогда не передавай в контексте параметры для выполнения бизнес-логики!** Это очень важное правило, которое постулируется с невероятной силой. Почему? Во-первых, при такой передаче важных данных ты невольно создаёшь неявный контракт. Знает ли принимающая сторона о том, что в контексте лежат важные для вычисления данные? А если знает, то будут ли они там в принципе? Во-вторых, падает производительность, потому что вложенные контексты с параметрами ищут данные в иерархии за O(n), что означает очень медленный поиск по ключу в худшем случае по всей цепочке данных. В-третьих, метод `ctx.Value`, который ищет и возвращает значение, если оно найдено, работает с приведением типа из интерфейса, что вновь означает потерю производительности. А ведь значение по ключу может и не быть нужного нам типа:

```go
if id, ok := ctx.Value(requestIDKey).(string); ok {
	fmt.Println(id)
} else {
	log.Fatal("fatal!!!")
}
```

Также код с такими вложенными данными сложно тестировать и дебажить, что ведёт к снижению прогнозируемости работы твоего кода.

Если вдруг действительно что-то надо передать в контексте (метаданные), то хорошим тоном является объявление ключа кастомным типом в виде константы, чтобы быть уверенным в том, что если данные придут, то ожидаемые:

```go
type contextKey string // кастомный тип ключа на основе строк

const userIDKey contextKey = "userID"

ctx := context.WithValue(context.Background(), userIDKey, 123)
```

### Работа с контекстом

Предлагаем проанализировать следующий пример:

```go
package main

import (
    "context"
    "fmt"
    "time"
)

func worker(ctx context.Context, id int) {
    for {
        select {
        case <-ctx.Done():
            fmt.Printf("Worker %d: остановка, причина: %v\n", id, ctx.Err())
            return
        default:
            fmt.Printf("Worker %d: работает...\n", id)
            time.Sleep(500 * time.Millisecond)
        }
    }
}

func main() {
    ctx, cancel := context.WithCancel(context.Background())

    // Запускаем несколько горутин
    for i := 1; i <= 3; i++ {
        go worker(ctx, i)
    }

    // Даем им поработать две секунды
    time.Sleep(2 * time.Second)

    // Отменяем контекст
    fmt.Println("main: отправляем сигнал отмены...")
    cancel()

    // Даём время на завершение
    time.Sleep(1 * time.Second)
    fmt.Println("main: программа завершена")
}
```

Здесь собраны наши знакомые с прошлой лекции. Что происходит?
1. На основе базового контекста (`Background`) создаётся контекст с явной отменой.
2. Далее запускаются три горутины, в которые происходит передача контекста для управления их жизненным циклом.
3. Внутри горутин-воркеров в бесконечном цикле происходит прослушивание канала `Done` у контекста. Сначала `select` пытается понять, закрыт ли канал через `case` (напомним, что из закрытого канала происходит бесконечное чтение дефолтных значений). Если контекст ещё не закрыт, то выполняет блок `default` и так до того момента, пока не последует сигнал от контекста.
4. Код некоторое время ждёт, имитируя бурную деятельность воркеров, а затем явно вызывает `cancel`, после чего все воркеры получают сигнал о том, что контекст завершён через канал `Done` и код ещё даёт немного времени горутинам на завершение.

Подобные примеры можно сделать и с тайм-аутами, и с дедлайнами. Но важно тут ещё следующее: **для контекстов с отменой (с функцией `cancel`), эту функцию надо вызывать всегда — зачастую через `defer`**. Это делается для гарантии завершения контекста в любой ситуации. Так как иначе можно получить утечку памяти и горутин.

```go
ctx, cancel := context.WithCancel(context.Background())
defer cancel()
```

Также мы можем явно проверять ошибки из пакета `context` через вызов:

```go
if err := ctx.Err(); err != nil {
	if errors.Is(err, context.Canceled) {
		// ...
	}

	if errors.Is(err, context.DeadlineExceeded) {
		// ...
	}
}
```

`context.Canceled` и `context.DeadlineExceeded` — ошибки, которые порой можно и нужно проверять.

### Идиоматика при работе с контекстом

Несколько правил работы с контекстом ты уже знаешь. Поговорим про другие:

1. Передавать контекст надо всегда первым параметром в функции. Это единый стандарт, при котором сразу видно, что функция может уметь в отмену или временное ограничение.
2. Контекст не надо хранить в структурах, так как они не имеют отношения к жизненному циклу функций и времени выполнения кода.
3. В долгих операциях надо проверять контекст через закрытие канала `Done` в `select` с неблокирующим вызовом `default`.
4. Надо внимательно следить за тайм-аутами. Если дочерний контекст будет иметь больший тайм-аут, чем родительский, то это может повлиять на некорректное поведение цепочки вызовов.
5. Использовать `context.TODO` в рабочем коде не нужно. Контекст ничего не делает, и его существование — причина отвлечения от того, что на самом деле делает код.
6. `context.Background` создаётся только на верхнем уровне вызовов. Не прерывай цепочку контекстов вновь созданным контекстом.

## Graceful shutdown

Что происходит, когда ты нажимаешь Ctrl + C во время выполнения программы? Она мгновенно завершается. Но что, если нам необходимо завершиться не сразу, а подождать некоторое время, чтобы определённые операции успели завершить свою работу? Например, закрыть соединения к базам данных или завершить тяжеловесные вычисления и успеть записать их на диск? Для этого необходимо реализовывать **graceful shutdown**.

**Graceful shutdown** — это способность приложения корректно завершать работу, сохраняя целостность данных и не прерывая критические операции.

Как реализовать такое поведение? ОС, Docker, Kubernetes общаются с нашими программами посредством *сигналов* — способов уведомления процесса о каком-то событии. В Go слушать эти сигналы можно с помощью пакета `os/signal`.
Какие основные сигналы можно ловить:
- SIGINT (2) — сигнал прерывания, обычно посылается при нажатии Ctrl + C.
- SIGTERM (15) — сигнал завершения. Это «вежливый» запрос на завершение, который дает приложению время на «уборку».
- SIGKILL (9) — сигнал принудительного завершения. Не может быть перехвачен или проигнорирован. Используется как крайняя мера.
- SIGHUP (1) — сигнал «повесить трубку». Исторически использовался для уведомления об обрыве соединения с терминалом. Сегодня часто используется для перезагрузки конфигурации.

Корректное завершение программ использует каналы:

```go
package main

import (
    "fmt"
    "log"
    "os"
    "os/signal"
    "syscall"
    "time"
)

func main() {
    // Создаём канал для приёма сигналов. Почему буферизированный?
    sigChan := make(chan os.Signal, 1)

    // Указываем, какие сигналы мы хотим получать и куда будем писать при их появлении
    signal.Notify(sigChan,
        syscall.SIGINT,  // Ctrl+C
        syscall.SIGTERM, // Сигнал завершения от системы/Kubernetes
        syscall.SIGHUP,  // Сигнал перезапуска конфигурации
    )

    fmt.Println("Приложение запущено. Ожидаю сигналы...")

    // Запускаем фоновую работу
    go func() {
        for i := 1; i <= 10; i++ {
            fmt.Printf("Выполняю работу %d/10...\n", i)
            time.Sleep(2 * time.Second)
        }
    }()

    // Блокируемся до получения сигнала
    sig := <-sigChan
    log.Printf("Получен сигнал: %v (%v)\n", sig, sig)
}
```

Ниже прикрепляем пример правильного graceful shutdown для http-сервера (мы будем говорить о http позже, но пока нам важно использование контекста и некоторой функции, которая будет корректно завершать выполнение программы):

```go
func gracefulServer() {
    server := &http.Server{
        Addr:    ":8080",
        Handler: createRouter(),
    }

    // Запускаем сервер в горутине
    go func() {
        if err := server.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Printf("Ошибка сервера: %v", err)
        }
    }()

    // Ждём сигнала
    sig := make(chan os.Signal, 1)
    signal.Notify(sig, syscall.SIGINT, syscall.SIGTERM)
    <-sig

    log.Println("Получен сигнал завершения, начинаем graceful shutdown...")

    // Создаём контекст с тайм-аутом для shutdown
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()

    // Корректно завершаем работу
    if err := server.Shutdown(ctx); err != nil {
        log.Printf("Ошибка при shutdown: %v", err)
        server.Close() // Вынужденное завершение
    }

    log.Println("Сервер корректно завершён")
}
```

## Примитивы синхронизации

Для того чтобы асинхронный, многопоточный код выполнялся, необходимо синхронизировать разные его ветви между собой. Или же защитить данные от множественного доступа. Тебе уже знакомы *каналы* — тип, который позволяет синхронизировать горутины между собой, пересылать уведомления и данные из одной подпрограммы в другую.

Каналы — это пример **примитивов синхронизации**. Это такие низкоуровневые инструменты, которые обеспечивают потокобезопасный доступ к общим ресурсам и предотвращают состояния гонки при одновременной работе потоков, корутин, горутин и так далее. В Golang примитивы синхронизации (помимо каналов), лежат в двух пакетах: [`sync`](https://github.com/golang/go/tree/go1.25.7/src/sync) и [`atomic`](https://github.com/golang/go/tree/go1.25.7/src/sync/atomic) (внутри `sync`). Пройдёмся по ним и постараемся понять, с чем мы можем иметь дело и когда.

1. `Атомики`

**Атомики** — максимально низкоуровневые примитивы синхронизации, которые по своей сути являются обёрткой над ассемблер-кодом в Golang. Функции, которые мы можем использовать, лежат в [одном](https://github.com/golang/go/blob/go1.25.7/src/sync/atomic/doc.go) месте, а их [реализация](https://github.com/golang/go/blob/go1.25.7/src/sync/atomic/asm.s) — в другом, и это правда ассемблер. Атомики позволяют получить доступ к переменной с гарантией, что ни одна другая горутина в момент доступа не владеет ею: ни читает значение, ни пишет новое. То есть атомарные операции гарантированно выполняются без прерываний и обеспечивают синхронизацию доступа к данным на уровне инструкций процессора.

На самом деле это самое важное, что мы должны знать об атомиках. Их зачастую используют, как потокобезопасные счётчики. Методов у пакета много, но в качестве примера можно рассмотреть работу с некоторой переменной, в которую мы добавляем значение, а затем читаем его, без страха получить неконсистентные данные:

```go
func main() {
    var v int64 = 0

    go atomic.AddInt64(&v, 1)
    go atomic.AddInt64(&v, -2)

    time.Sleep(1 * time.Second)

    val := atomic.LoadInt64(&v)
    fmt.Println(val)
}
```

В этом примере мы без гарантии порядка выполнения горутин и без гарантии того, что они не выполнятся одновременно, имеем гарантию того, что всегда получим корректный и потокобезопасный результат.

2. `Мьютексы`

**Мьютексы** — это примитив синхронизации, который позволяет защищать блок кода от неконсистентных данных при доступе к ним из нескольких горутин. Ведь при таком сценарии, если, например, одна горутина сначала пишет в мапу, а потом, спустя время, читает из неё, надеясь получить в одной операции те данные, что уже были сложены в мапу, а в этот момент другая горутина получит к этой мапе доступ и перепишет значение, то получится неожидаемое поведение, которое сломает нашу логику. Как повелось, мьютексы в Go — это [структуры](https://github.com/golang/go/blob/go1.25.7/src/sync/mutex.go#L30).

Поэтому мьютексы очень часто защищают большие блоки кода от конкурентного доступа. В том числе в связке с мапами, делая их потокобезопасными. Каким образом? При вызове мьютекса он блокирует участок кода для текущей горутины, не давая другим горутинам выполнять его. Зачастую горутины определяются в структурах для того, чтобы использоваться в связанных с ними методах. Но ты можешь вызвать мьютекс напрямую:

```go
func exampleOne() {
    mu := sync.Mutex{}

    mu.Lock()
    fmt.Println("Mutex locked")
    mu.Unlock()
}
```

Двумя вызванными у мьютекса методами `Lock` и `Unlock` мы заблокировали участок кода для других горутин. Причём и операции чтения, и операции записи. Но всё же в структурах с мапами мьютексы используются чаще. Здесь стоит сказать о важном: **поле с мьютексом располагается в блоке над объектом, который он защищает.** Это чисто визуальная идиоматическая договорённость.

```go
type Cache struct {
    // Мы показываем, что мьютекс защищает мапу
    mu sync.Mutex
    m map[int]int

    sl []int // Никто не мешает вызвать мьютекс со слайсом, однако мы предполагаем, что всё-таки слайс не под защитой мьютекса
}

func (c *Cache) Add(key int, val int) {
    c.mu.Lock
    defer c.mu.Unlock

    m[key] = val
}
```

В этом примере использована хитрость: да, мы можем наглядно закрыть обращение к мапе методами мьютекса и снизу, и сверху. Однако `defer` во многих случаях помогает быть уверенными, что мьютекс точно будет разблокирован.

Но помимо обычного мьютекса в Go существует [*RW-мьютекс*](https://github.com/golang/go/blob/go1.25.7/src/sync/rwmutex.go#L39), который имеет методы `RLock` и `RUnlock`, позволяющие не блокировать горутины с операциями чтения. Но в случае появления горутины, которая намерена мутировать данные, мьютекст заблокирует код для всех прочих горутин, даже тех, кто пришёл читать данные. Таким образом обычный мьютекс является полностью блокирующим, а RW-мьютекс блокируется только на операциях записи. RW-мьютексу стоит отдавать предпочтение, когда операций чтения у защищаемого кода намного больше, чем операций записи.

Таким образом мьютексы синхронизируют доступ к общей памяти.

3. `WaitGroup`

**WaitGroup** — очень важный примитив синхронизации, который позволяет ожидать выполнения конечного, заранее известного числа горутин. Это снова [структура](https://github.com/golang/go/blob/go1.25.7/src/sync/waitgroup.go#L48) в исходном коде Go, реализованная с помощью атомиков. Этот примитив имеет всего три метода:
- `Add(n)` — добавляет в счётчик ожидания горутин их общее количество (или обычно просто используется `Add(1)`) при объявлении новой горутины;
- `Done()` — производит декремент счётчика горутин по окончании одной из них;
- `Wait()` — блокирует выполнение программы для ожидания окончания работы всех n горутин.

```go
func main() {
    var wg sync.WaitGroup

    wg.Add(2)

    for range 2 {
        go func() {
            defer wg.Done()

            fmt.Println("Hello World")
        }()
    }

    wg.Wait()
}
```

В примере выше мы создаём счётчик через **waitGroup**; запускаем именно столько горутин, сколько у нас их в счётчике (иначе можем попасть на *deadlock*); в `defer` сбрасываем счётчик (каждый раз −1) — можно вызывать `Done()` и не в `defer`, но использование этого слова — гарантия срабатывания счётчика в любой ситуации завершения горутины; далее блокируем горутину `main`, пока все дочерние горутины не отработают.

**WaitGroup** — невероятно полезный инструмент при работе с пулом горутин. Однако, у него имеется недостаток - обработка ошибок при использовании этого примитива превращается в головную боль, так как придется ждать выполнения всех горутин и вдобавок грамотно обрабатывать возникающие ошибки. В случае возникновения подобных проблем стоит воспользоваться [пакетом](https://pkg.go.dev/golang.org/x/sync/errgroup) `errGroup`, который помогает отслеживать возникновение ошибок и корректно завершать горутины.

4. `Once`

`Once` — это тоже структура с методом `Do`, которая позволяет гарантировать, что некоторая функция, доступ к которой имеется у множества горутин, выполнится **только один раз** самой первой горутиной. Последующие её вызовы не приведут к срабатыванию функции.

```go
var (
    once sync.Once
)

func initialize() {
    fmt.Println("Инициализация ресурсов (выполняется один раз)...")
}

func main() {
    var wg sync.WaitGroup

    // Запускаем 10 горутин, которые хотят выполнить инициализацию
    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
			
            once.Do(initialize) 
            fmt.Printf("Горутина %d завершила работу\n", id)
        }(i)
    }

    wg.Wait()
    fmt.Println("Программа завершена.")
}
```

5. `sync.Map`

Помимо потоконебезопасной мапы и мапы с мьютексами существует ещё и [мапа](https://github.com/golang/go/blob/go1.25.7/src/sync/map.go#L40) в пакете `sync`, являющаяся потокобезопасной за счёт использования под капотом мьютекса и двух обычных мап для актуальных и изменённых данных. Возникает вопрос: зачем тогда трюк с мапой и мьютексами, когда можно использовать `sync.Map`? Ответ прост — она довольно медленная и неуклюжая, а профит от неё получается только при наличии большого количества свободных ядер и невероятно большого процента операций чтения. Подробнее о ней читай в дополнительных источниках.

## Паттерны конкурентности

В целом логично, что при существовании отдельного стиля программирования в виде многопоточки стали выделяться отдельные классы задачи и обобщённые способы их решения — паттерны. Давай взглянем на несколько полезных и часто используемых паттернов конкурентного программирования, реализованных с помощью концепций языка.

1. `Singleflight` позволяет в случае запроса к какому-то сервису или части бизнес-логики группировать запросы, ожидающие одной и той же долгой операции. В таком случае у нас появляется лидерский запрос, выполняющий вычисления и получающий их напрямую, и пул запросов, которые падают в ожидании ответа лидерского запроса, который, завершаясь, делится результатами со всем пулом, завершаясь сам и давая возможность ожидающим запросам разблокироваться. Подобное поведение невероятно полезно при долгих вычислениях, которыми можно поделиться с запросами, приходящими во время этих самых вычислений. Таким образом экономятся ресурсы сервера. Зачастую используется во время ожидания долгих вычислений, когда во время них нам проще собрать приходящие запросы в кучу и поделиться со всеми одними и теми же результатами.

`Singleflight` [реализован](https://pkg.go.dev/golang.org/x/sync/singleflight) в расширенной стандартной библиотеке языка.

```go
// call — структура, которая описывает вызовы некоторой функции. Содержит поля с ошибкой и значением, которые мы пошарим на весь пул запросов, а также канал, закрывающийся при окончании работы запроса/функции
type call struct {
    err   error
    value any
    done  chan struct{}
}

// SingleFlight реализует логику паттерна, используя список вызовов в мапе по конкретным ключам, чтобы группировать вызовы, относящиеся к одному и тому же ресурсу, друг с другом. В мапе вызов хранится указателем, чтобы имелась возможность разделять этот объект между разными горутинами
type SingleFlight struct {
    mutex sync.Mutex
    calls map[string]*call
}

func NewSingleFlight() *SingleFlight {
    return &SingleFlight{
        calls: make(map[string]*call),
    }
}

// Do реализует действие вычисления некоторой функции action
func (s *SingleFlight) Do(ctx context.Context, key string, action func(context.Context) (any, error)) (any, error) {
    // Большой блок под мьютексом, в котором его закрытие происходит дважды, так как defer тут не поможет программе завершиться корректно - после второго Unlock у нас происходит выполнение функции, которая с defer s.mutex.Unlock просто заблокирует этот код на всё время своей работы
    // Здесь мы проверяем существование вызова, и если он есть, то просто вешаем его в ожидание через мапу. Иначе создаём новый вызов
    s.mutex.Lock()
    if call, found := s.calls[key]; found {
        s.mutex.Unlock()
        return s.wait(ctx, call)
    }

    call := &call{
        done: make(chan struct{}),
    }

    s.calls[key] = call
    s.mutex.Unlock()

    // Запускаем функцию в горутине, чтобы реализовать ожидание данных и их получение через разделяемую структуру вызова call. В случае паники мы её обрабатываем, не роняя программу, просто подчищая за собой в списке вызовов
    go func() {
        defer func() {
            if v := recover(); v != nil {
                call.err = errors.New("error from single flight")
            }

            close(call.done)

            s.mutex.Lock()
            delete(s.calls, key)
            s.mutex.Unlock()
		}()

        call.value, call.err = action(ctx)
    }()

    return s.wait(ctx, call)
}

// wait ожидает либо завершения контекста для всего запроса, чтобы завершить все запросы, либо сигнала от лидерского запроса, что он завершился и поделился данными с ожидающими вызовами
func (s *SingleFlight) wait(ctx context.Context, call *call) (any, error) {
    select {
    case <-ctx.Done():
        return nil, ctx.Err()
    case <-call.done:
        return call.value, call.err
    }
}
```

2. `Генератор` используется в случаях, когда необходимо создать некоторое количество данных, поделиться ими с другой горутиной, а затем закрыть канал, тем самым владея его полным жизненным циклом:

```go
func generator() <- chan int {
    ch := make(chan int)

    go func(){
        for i := range 10 {
            ch <- i + 1
        }

        close(ch)
    }()

    return ch
}
```

3. `Fan-in` помогает собирать данные из нескольких каналов в один общий. Подходит для случаев объединения результатов расчёта от нескольких горутин в единой точке сбора данных.

```go
func fanIn(input1, input2 <-chan string) <-chan string {
    ch := make(chan string)

    go func(){
        for {
            select {
                case s := <-input1: 
                    ch <- s
				case s := <-input2: 
                    ch <- s
            }
        }
    }()

    return ch
}
```

4. `Fan-out` делает обратное прошлому паттерну: он, наоборот, распределяет данные из одного канала в несколько. Полезно, например, использовать в связке с генератором данных:

```go
func worker(ch <-chan int, wg *sync.WaitGroup) {  
    defer wg.Done()

    for v := range ch {  
       fmt.Println(v)  
       time.Sleep(1 * time.Second)  
    }  
}  
  
func sender() {  
    ch := make(chan int)  
    var wg sync.WaitGroup 

    for i := 0; i < 2; i++ {  
       wg.Add(1)  
       go worker(ch, &wg)  
    }  
  
    for i := range 10 {  
       ch <- i  
    }  
  
    close(ch)  
  
    wg.Wait()  
    fmt.Println("done")  
}
```

5. `Pipeline` выстраивает цепочку обработки данных, что необходимо в случае запуска некоего конвейера для, например, последовательной обработки файлов:

```go
func producer() <-chan int {  
    c := make(chan int)  
  
    go func() {  
       for i := range 10 {  
          c <- i + 1  
       }  

       close(c)  
    }()  
  
    return c  
}  
  
func producerConsumer(c <-chan int) <-chan int {  
    out := make(chan int)  
  
    go func() {  
       for v := range c {  
          out <- v * 2  
       }  

       close(out)  
    }()  
  
    return out  
}  
  
func consumer(ch <-chan int) {  
    for v := range ch {  
       fmt.Println(v)  
    }  
}
```

6. `Worker pool` — паттерн, помогающий организовать работу над данными в виде параллельных воркеров. Обычно используется при большом количестве однотипной работы над крупными данными:

```go
func worker(jobs <-chan int, results chan<- int, wg *sync.WaitGroup) {  
    defer wg.Done()  
  
    for j := range jobs {  
       time.Sleep(1 * time.Second)  
       fmt.Println("job", j)  
       results <- j * j  
    }  
  
}  
  
func main() {  
    jobs := make(chan int)  
    results := make(chan int)  
  
    var wg sync.WaitGroup  
  
    for i := range 3 {  
       wg.Add(1)  
       go worker(jobs, results, &wg)  
    }  
  
    go func() {  
       for i := range 10 {  
          jobs <- i  
       }  
       close(jobs)  
    }()  
  
    go func() {  
       wg.Wait()  
       close(results)  
    }()  
  
    for result := range results {  
       fmt.Println(result)  
    }  
}
```
