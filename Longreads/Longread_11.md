# HTTP протокол и работа с ним в REST-архитектуре в Golang

Мы, как разработчики, создаем не только консольные утилиты, которые можно запустить через исполняемый бинарный файл. Если мы выходим за рамки системы, в которой работаем, то сталкиваемся с межсервисным взаимодействием. Конечно, его можно организовать в нескольких вариантах, но в большинстве своём подобное взаимодействие сетевое, то есть с использованием сети Интернет либо внутренних сетей компаний.

Для того чтобы сервисы общались друг с другом, необходимо использовать некоторый протокол — набор правил и договоренностей о формате запросов и ответов. Как правило, запросы и ответы происходят и между отдельными сервисами, и между браузером и каким-то сервером, хранящим нужную для отображения данных информацию.

В любом случае в таком варианте мы сталкиваемся с понятием **клиент-серверной архитектуры** — архитектуры взаимодействия в сети, при которой кто-то отсылает запрос (клиент), желая либо получить данные в ответе, либо изменить их в месте хранения, а кто-то этот запрос принимает, обрабатывает и отдаёт подходящий ответ (сервер). Так вот, набор правил и договоренностей, который управляет общением клиентов и серверов в сети (между прочим, почти любой сервис может быть и клиентом, и сервером одновременно), в нашем случае **HTTP**.

## HTTP-протокол

Вообще, для описания взаимодействия в сети существует две модели: [OSI](https://neerc.ifmo.ru/wiki/index.php?title=OSI_Model) и [TCP/IP](https://ru.wikipedia.org/wiki/TCP/IP), которые тебе нужно будет понимать в будущем. Сейчас важно понять, что HTTP-протокол в обеих моделях относится к прикладному уровню. Это означает, что протокол отвечает именно за передачу данных в сети. Вообще HTTP — *HyperText Transfer Protocol*, то есть протокол, основанный на разметке страниц в интернете.

И в его основные задачи входят:

- передача веб-страниц, изображений и других файлов;
- обработка форм и пользовательского ввода;
- управление состоянием соединения (метаданные, сессии, авторизация);
- поддержка разных типов данных для передачи информации (JSON, XML, текст, бинарные файлы).

Иными словами, протокол использует определённые форматы данных для их передачи определёнными сообщениями, сохраняя в соединении между клиентом и сервером информацию об этом соединении. При этом HTTP — протокол без сохранения состояния. То есть каждый запрос клиента независим от других, а сервер не помнит предыдущие запросы этого же клиента. Да, для сохранения состояния можно использовать вспомогательные метаданные (cookie, данные сессии), но эти данные в большинстве своём живут только в рамках одного соединения.

Протокол передаёт данные между клиентом и сервером через сообщения, которые так и называются «запрос» и «ответ». Они оба передаются между клиентом и сервером в виде открытого и простого текста в кодировке ASCII или UTF-8.

### HTTP-запрос

Запрос — это описанное в протоколе сетевое сообщение определённого вида, которое позволяет понять серверу, что от него хочет клиент:

```text
POST /api/users/123 HTTP/1.1           ← Стартовая строка
Host: example.com                      ← Заголовки
User-Agent: Mozilla/5.0
Accept: application/json
Authorization: Bearer abc123
                                       ← Пустая строка
{"key": 123}                           ← Тело запроса (для POST/PUT, для GET тело запроса не используется!)
```

Обсудим это сообщение по порядку его составляющих.

1. *Стартовая строка* определяет то, какое действие клиент ожидает от сервера, к какому эндпоинту (хендлеру, ручке) сервера идёт обращение и какая версия протокола используется (об этом немного потом).
Технически вернее сказать, что эндпоинт — это **URI** (Uniform Resource Identifier), то есть компактная строка, которая однозначно идентифицирует ресурс, к которому происходит обращение. В нашем случае *URI* можно прочитать так: из API (списка эндпоинтов) сервера, мы хотим получить данные по пользователю 123. Почему получить? Потому что в стартовой строке мы видим метод `POST`, который отвечает создание и обновление данных. Но, помимо него, существует ещё целый [набор](https://developer.mozilla.org/ru/docs/Web/HTTP/Reference/Methods) HTTP-методов.
Вот самые часто используемые:

- `GET` — говорит о том, что клиент хочет получить данные/ресурс: файлы, веб-страницу, данные в виде JSON/XML и так далее;
- `POST` — просит сервер создать данные на сервере: загрузить файл, создать запись в базе данных;
- `PUT` — используется для обновления ресурса/данных на сервере: поменять имя пользователя, заменить файл;
- `DELETE` — просит сервер что-то удалить у себя из данных.

Из этого видно, что только один запрос из перечня (а именно `GET`) — это запрос, который не меняет состояние данных. Это так называемая `идемпотентность` - свойство операций или действий, при котором многократное повторение приводит к такому результату, что и однократное.

Важно отметить, что и *URI* может быть не простым. Даже в текущем примере он имеет важный параметр, который мы можем изменять при составлении HTTP-запроса. В строке `/api/users/123` число `123` — так называемый *path-параметр* (параметр пути), и в общем виде строка должна выглядеть примерно так: `/api/users/{userID}`. Подобные параметры помогают определять конечный ресурс в иерархии, указывая на него через уникальный параметр. Значение этого параметра при запросе указывать обязательно, если, конечно, в *URI* он имеется. Но в строке могут быть и необязательные параметры. Они называются *query-параметрами* (параметрами запроса), и перечисляются после знака `?` прямо в строке *URI*: `/api/users/{userID}?offset=20&limit=2`. В данном конкретном примере у нас есть два query-параметра, которые передаются серверу и могут быть либо заполнены (разделителем параметров является `&`), либо опущены. Во втором случае разработчику **необходимо** обработать ситуацию отсутствия параметра/параметров через использование дефолтных значений из кода.

2. *Заголовки* в HTTP-запросе помогают передавать метаданные конкретного запроса серверу в виде ключа — значения. То есть это именно техническая информация, которая бывает довольно полезна, потому что определяет контекст:

- `Authorization` и `Cookie` позволяют реализовать аутентификацию и определение пользователя для ресурса.
- `Host` указывает, к какому ресурсу идёт обращение.
- `Accept` определяет содержание тела запроса: тип, кодировка.
- `User-Agent` и, например, `Accept-Language` говорят о контексте клиента, то есть какой у него браузер, ОС или язык системы.

Исчерпывающий список заголовков (хедеров) можно отыскать [тут](https://developer.mozilla.org/ru/docs/Web/HTTP/Reference/Headers).

3. *Тело запроса* может быть JSON (зачастую), XML, бинарный формат. Это те данные, которые передаются от клиента к серверу и обратно. **Важно**: в `GET`-запросах принято не добавлять тела запроса. Ведь с помощью этого метода мы просим данные. Поэтому в этом случае обходятся параметрами пути и запроса.

### HTTP-ответ

Ответ протокола тоже текст определённого вида:

```text
HTTP/1.1 200 OK                        ← Стартовая строка ответа
Content-Type: application/json         ← Заголовки ответа
Content-Length: 1234
Date: Mon, 23 Oct 2023 12:00:00 GMT
Server: nginx/1.18.0
                                       ← Пустая строка
{                                      ← Тело ответа
  "id": 123,
  "name": "John Doe",
  "email": "john@example.com"
}
```

Здесь пройдём чуть быстрее по составу сообщения.

1. *Стартовая строка* теперь имеет только три элемента: версию протокола, код ответа и статус ответа. Код и статус взаимосвязаны. Код определяет то, ответ какого вида получен клиентом, а статус позволяет более детально понять, что именно за результат получил клиент. Например, если сервер не нашел данные, то он ответит, что искомых данных он честно не нашел: `HTTP/1.1 404 Not Found`. В случае возникшей ошибки кода сервера клиент спокойно может получить: `HTTP/1.1 500 Internal Error`.

В целом коды ответа делятся на группы:
- `2xx` — успешные ответы;
- `3xx` — перенаправление запроса;
- `4xx` — ошибки клиента;
- `5xx` — ошибки сервера.

Подробнее стоит почитать [здесь](https://developer.mozilla.org/ru/docs/Web/HTTP/Reference/Status). Ради интереса найди ошибку с кодом `418`.

2. Заголовки ответа могут содержать дополнительную информацию о сервере и данных. То есть выполняют ровно ту функцию, что и в запросе.

3. Тело ответа тоже может быть тех же самых форматов, что и тело запросов.

### Версии HTTP-протокола

HTTP-протокол довольно устойчив во времени. С момента его создания в 1991 году он получил лишь пять версий, одна из которых в глобальном вебе всё ещё основная:

- HTTP/0.9 (1991) — очень простой протокол: только GET-запросы, только HTML-разметка.
- HTTP/1.0 (1996) — получил POST, HEAD, статусы ответов, заголовки. Неудачная версия с проблемами безопасности.
- HTTP/1.1 (1997) — стабильная версия с keep-alive, pipelining, хостами. Самая распространённая и используемая.
- HTTP/2 (2015) — мультиплексирование, сжатие заголовков, приоритеты. На его основе реализован GRPC.
- HTTP/3 (2022) — работает поверх QUIC (UDP), быстрее и надёжнее прошлых версий.

Версии 2 и 3 являются бинарными и их нельзя прочитать обычными средствами.

## REST

Раз у нас существуют клиенты и серверы, то необходимо обсудить, как выстраивать грамотное взаимодействие между ними. Здесь мы упираемся в разговор о контрактах и архитектуре. В целом клиент в этой истории выступает ведомым, так как именно ему нужны данные сервера. Это означает и то, что контракт общения клиента и сервера должен лежать на стороне последнего (в том числе потому, что клиентов может быть больше одного — как пример, несколько пользователей, которые одновременно пользуются сайтом). Подобное накладывает на сервер определённую ответственность. Как минимум, в удобном API (наборе эндпоинтов) для клиентов. Этот набор можно конструировать. И именно здесь возникает проблема архитектуры именно HTTP-сервера.

К счастью, до нас уже придумали довольно универсальный способ архитектурного построения веб-серверов. Это `REST` — архитектурный стиль, набор принципов и ограничений для создания веб-систем. Да, можно писать всё по-своему, но в индустрии принято всё-таки соответствовать этому стилю. Важно, что `REST` именно что набор принципов, а не какой-то протокол. Очень часто на собеседованиях путают архитектурный стиль с протоколом.

Ниже представлены принципы `REST`:

- *Клиент-серверная архитектура* — реализуется через разделение обязанностей между клиентом (отвечает за использование предоставленного API) и сервером (отвечает за данные и набор эндпоинтов в API), которые развиваются независимо друг от друга.
- *Отсутствие состояния* (stateless) — сервер не хранит данные о сессии клиента, а каждый запрос независим от остальных и содержит всю информацию, необходимую для обработки.
- *Кеширование* — ответы от сервера должны быть помечены как кешируемые или нет, чтобы клиент мог повторно использовать полученные данные.
- *Единообразие интерфейса* — использование унифицированного способа взаимодействия для упрощения архитектуры за счёт HTTP-методов, понятия ресурсов и URL.
- *Многослойная система* — разделение компонентов на отдельные уровни (балансировщик нагрузки, HTTP-обработчик), которые не могут видеть дальше какого-то другого уровня.
- *Код по требованию* (необязательный пункт) — возможность расширять функциональность клиентов за счёт загрузки кода с сервера (например, передача им клиенту каких-то скриптов).

При выполнении хотя бы пяти первых пунктов систему можно назвать соответствующей `REST`-архитектуре. Или же подобные приложения обычно называют `RESTful`, то есть подчеркивая, что конкретное приложение ориентировано на соблюдение всех этих правил.

Нужно подметить, что `REST API` реализуют методы четырех операций `CRUD`: create, read, update, delete. То есть `REST API` у серверов используют только четыре метода:

- `Get` для получения ресурса;
- `Post` для создания нового ресурса;
- `Put` для обновления или полной замены существующего ресурса;
- `Delete` для удаления ресурса с сервера.

Пора обсудить, что такое `ресурс` в контексте `HTTP` и `REST`, потому что в `REST` всё вертится вокруг ресурсов. А *ресурс* — это любая сущность, которая имеет уникальный идентификатор в виде *URI*, к которой можно получить доступ. Или который можно изменить, заменить, удалить. Например, документ, изображение, товар, пользователь, информация о состоянии системы. С такими ресурсами клиент работает не напрямую, а через их представления в виде `JSON`, `XML`, бинарный формат. Таким образом обращаясь по `URI` в виде `https://api.example.com/users/123` клиент имеет дело с ресурсом в формате `JSON`. А при обращении к ресурсу по адресу `https://api.example.com/users` клиент уже будет иметь дело с массивом ресурсов.

Таким образом понятие «ресурс» в подобном контексте отражает именно данные и доступ к ним по уникальному идентификатору. И этот уникальный идентификатор также имеет определённые правила хорошего дизайна:

```text
Хорошо: ресурсно-ориентированные URI

GET    /api/users          — Получить всех пользователей
GET    /api/users/123      — Получить пользователя с ID=123
POST   /api/users          — Создать нового пользователя
PUT    /api/users/123      — Обновить пользователя 123
DELETE /api/users/123      — Удалить пользователя 123

GET    /api/users/123/orders     — Заказы пользователя 123
GET    /api/orders/123/items     — Позиции в заказе 123

ПЛОХО: глаголы в URI

GET    /api/getUsers        — Неправильно!
POST   /api/createUser      — Неправильно!
PUT    /api/updateUser/123  — Неправильно!

```

Закончить стоит тем, что `REST` не строгий стандарт, а скорее философия, которой можно следовать, а можно не следовать. Но повторим ещё раз: в индустрии принято следовать.

## HTTP-серверы в Go

Нам, как разработчикам, необходимо уметь создавать серверы, которые будут обслуживать клиентов и работу с данными. Серверы должны уметь принимать HTTP-запросы, иметь понятный API и отвечать на запросы клиентов (возвращать ресурсы).

В Go уже под капотом есть всё, чтобы написать такой сервер без использования сторонних инструментов. Для этого будем использовать библиотеку `net/http`, которая своим наименованием говорит нам о том, что библиотека работает с сетью и протоколом `HTTP`.

`net/http` позволяет запускать HTTP-сервер за несколько строк:

```go
package main

import (
    "fmt"
    "net/http"
)

// weatherHandler обрабатывает запросы к /weather и возвращает приветственное сообщение
func weatherHandler(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintln(w, "Добро пожаловать в сервис погоды!")
}

func main() {
    http.HandleFunc("/weather", weatherHandler) // Регистрируем обработчик маршрута /weather
    fmt.Println("Сервер запущен на порту 8080...")

    err := http.ListenAndServe(":8080", nil) // Запуск сервера на порту 8080
    if err != nil {  
       fmt.Println("Ошибка запуска сервера:", err)  
    }
}
```

Главная функция здесь `http.ListenAndServe()`, которая принимает порт интерфейса сервера и обработчик запросов. В контексте Go обработчик (или handler) — объект с методом `ServeHTTP(w http.ResponseWriter, r *http.Request)`, который реагирует на входящие HTTP-запросы. Этот метод сервер вызывает для каждого входящего запроса. В качестве обработчика может выступать **любая** функция или структура, реализующая этот интерфейс. В простейшем случае вместо полноценного интерфейса можно использовать функцию с помощью `http.HandleFunc`. Стандартная библиотека сама обернёт её в нужный интерфейс и позволит обработать запрос по указанному адресу (`/weather`).

После запуска сервер будет доступен по ссылке http://localhost:8080/weather и поднимется локальный веб-сервер. Если в `ListenAndServe` будет указан параметр `nil`, обработчик выберется по умолчанию. В `net/http` по умолчанию используется глобальный маршрутизатор (`http.DefaultServeMux` — об этом позже), в который мы зарегистрировали свой обработчик через http.HandleFunc. Теперь, если перейти в браузере по адресу http://localhost:8080/weather, сервер вернёт строку «Добро пожаловать в сервис погоды!».

Обрати внимание, что функция-обработчик `weatherHandler` принимает два параметра:

- `http.ResponseWriter` — это интерфейс для отправки ответа клиенту (через него мы пишем данные и заголовки);
- `http.Request` содержит информацию о запросе (URL, метод, тело).

Однако всё-таки серверы создаются сложнее. Если в примере выше вся регистрация и создание сервера происходит под капотом, где мы не влияем на параметры сервера, то зачастую нам всё же хочется его настраивать под себя.

Поэтому сервер можно создать и таким образом:

```go
package main

import (
	"fmt"
	"log"
	"net/http"
	"time"
)

func handler(w http.ResponseWriter, r *http.Request) {
	switch r.URL.Path {
	case "/":
		fmt.Fprintln(w, "главная")
	case "/about":
		fmt.Fprintln(w, "о проекте")
	default:
		http.NotFound(w, r)
	}
}

func main() {
	srv := &http.Server{
		Addr:           ":8080",
		Handler:        http.HandlerFunc(handler),
		ReadTimeout:    5 * time.Second,
		WriteTimeout:   10 * time.Second,
	}

	log.Fatal(srv.ListenAndServe())
}
```

В новом примере мы уже оперируем не дефолтной структурой, а новым экземпляром сервера, который настраиваем сами под себя. Заметь, что в коде используется один метод `handler`, который будет принимать все запросы к серверу на себя, работая через парсинг `URL` с логикой запросов. Таким образом мы создаём самый простой маршрутизатор. Даже пока не проверяя тип самого метода запроса через `r.Method()`.

Здесь же глаз цепляется за то, что в первом примере мы использовали `http.HandleFunc()`, а в этом используем `http.HandlerFunc()`. Разница в одной букве, но она существенна. `http.HandleFunc` — функция регистрации маршрута на глобальном `DefaultServeMux`: первый аргумент — это шаблон пути (например, "/api/"), второй — функция-хендлер. Она внутри вызывает `DefaultServeMux.HandleFunc`. `http.HandlerFunc` же конкретный тип (`type HandlerFunc func(http.ResponseWriter, *http.Request)`), у которого присутствует метод `ServeHTTP`, позволяющий представлять функцию как хендлер. Иными словами, `HandlerFunc` обёртка над функцией, которая делает её хендлером, а `HandleFunc` вешает эту функцию на конкретный URL-паттерн.

Сервер можно сделать ещё более кастомным, используя голый `tcp`:

```go
package main

import (
	"fmt"
	"log"
	"net"
	"net/http"
)

func handler(w http.ResponseWriter, r *http.Request) {
	fmt.Fprintln(w, "Hello via Serve(listener)")
}

func main() {
	ln, err := net.Listen("tcp", ":8080")
	if err != nil {
		log.Fatal(err)
	}

	srv := &http.Server{Handler: http.HandlerFunc(handler)}
	log.Fatal(srv.Serve(ln))
}
```

Только вот нам часто требуется объединить конкретную логику с единой структурой, которая эту самую логику предлагает. Простой способ сделать это, как обычно написать всё руками (более простой и правильный способ рассмотрим в маршрутизации вопросов ниже):

```go
package main

import (
	"fmt"
	"log"
	"net/http"
)

type App struct{}

func (a *App) home(w http.ResponseWriter, r *http.Request) {
	fmt.Fprintln(w, "главная")
}

func (a *App) about(w http.ResponseWriter, r *http.Request) {
	fmt.Fprintln(w, "о проекте")
}

func (a *App) ServeHTTP(w http.ResponseWriter, r *http.Request) {
	switch r.URL.Path {
	case "/":
		a.home(w, r)
	case "/about":
		a.about(w, r)
	default:
		http.NotFound(w, r)
	}
}

func main() {
	app := &App{}
	srv := &http.Server{Addr: ":8080", Handler: app}
	log.Fatal(srv.ListenAndServe())
}
```

В этом примере мы уже действуем по-взрослому: аккумулируем логику в конкретной сущности и предоставляем эту логику в зависимости от пути. Благодаря методу `ServeHTTP` наш `App` теперь может участвовать в обработке запросов, регистрируясь в нашем сервере.

### Ещё немного про обработчики и маршрутизацию

В реальном сервисе обычно много разных `URL` (маршрутов), и для каждого нужен свой обработчик. Маршрутизация — это как раз процесс выбора подходящего обработчика для каждого запроса в зависимости от его пути и метода. В примере выше мы зарегистрировали обработчик для маршрута `/weather` с помощью `http.HandleFunc("/weather", ...)`. Для более сложных API можно использовать несколько обработчиков для разных путей.

Например, как в остальных примерах, или мы можем написать такое для трекера задач:

```go
http.HandleFunc("/tasks", tasksHandler)    // Например, вывод списка задач,
http.HandleFunc("/add", addTaskHandler)    // Добавление новой задачи
```

Если теперь сервер получит запрос к `/tasks`, вызовется функция `tasksHandler`, а к `/add` — `addTaskHandler`. `net/http` сам сравнивает URL запроса с зарегистрированными маршрутами и вызывает нужную функцию. По умолчанию используется префиксное совпадение путей: например, если обработчик зарегистрирован на `/tasks`, он будет вызван в том числе и для /tasks/subpath. Позже ты узнаешь, как организовать более точную маршрутизацию. Обработчики можно создавать и другим способом — реализуя интерфейс `http.Handler`, как мы и делали.

### Разбор запросов к серверу

Когда клиент делает HTTP-запрос, он отправляет метод (`GET`, `POST`, другие), URL, заголовки и опционально тело (body). В обработчике через объект `*http.Request` можно получить всю эту информацию:

- *Метод запроса*. Доступен как `r.Method` (строка, например `GET` или `POST`). Часто используют условные операторы, чтобы различать логику для разных методов на одном пути.
- *Путь и строка запроса*. `r.URL.Path` содержит путь (например, `/tasks`), а `r.URL.Query()` — объект с query-параметрами (часть URL после `?`). Например, для запроса `/tasks?done=true` вызов `r.URL.Query().Get("done")` вернёт строку `true`. У query-параметров всегда тип string, но при необходимости их нужно конвертировать.
- *Заголовки* (headers). Хранятся в `r.Header` (`map[string][]string`). Можно получить конкретный заголовок, например `r.Header.Get("Content-Type")`.
- *Тело запроса* (body). Для методов вроде `POST` или `PUT` клиент может отправлять данные в теле. В Go тело запроса доступно как `r.Body` (`io.ReadCloser`). Его можно прочитать, например через `io.ReadAll(r.Body)`, или использовать `json.NewDecoder(r.Body)` для разбора `JSON`. Важно: `r.Body` можно читать только один раз. После чтения (или если не предполагается чтение) его обычно закрывают: `defer r.Body.Close()`.

Рассмотрим обработчик, который разбирает параметры запроса. Допустим, для маршрута `/weather` мы хотим запросить погоду по названию города, принимая название как query-параметр:

```go
package main  
  
import (  
    "encoding/json"  
    "fmt"    
    "net/http"
)
  
// weatherHandler теперь анализирует переданный город из query-параметра
func weatherHandler(w http.ResponseWriter, r *http.Request) {
    city := r.URL.Query().Get("city") // Получаем данные из параметра ?city=...
    if city == "" {
        http.Error(w, "Не указан город", http.StatusBadRequest)
        return
    }
    fmt.Fprintf(w, "Запрошена погода для города: %s", city)
}
  
func main() {  
    http.HandleFunc("/weather", weatherHandler)
  
    fmt.Println("Сервер запущен на порту 8080...")  
    err := http.ListenAndServe(":8080", nil)
    if err != nil {  
       fmt.Println("Ошибка запуска сервера:", err)  
    }  
}
```

В этом коде `weatherHandler` проверяет, передано ли название города. Пытаемся получить *city* из query (например, запрос `/weather?city=Moscow`). Если параметр отсутствует, отправляем ошибку 400 (ошибка «Плохой запрос») обратно с помощью http.Error. После этого формируем сообщение и отправляем его через `fmt.Fprintf(w, "Запрошена погода для города: %s", city)`. Обрати внимание на `http.Error(w, msg, code)`. Это удобная функция, которая комбинирует установку статуса ответа и тело с сообщением об ошибке. В данном случае при ошибке разбора JSON мы возвращаем статус 400 Bad Request.

В последних версиях Go стали поддерживаться и path-параметры (раньше стандартная библиотека с ними не могла работать). Они задаются довольно легко в шаблоне URL, где мы можем сразу прописать и метод этого пути:

```go
	http.HandleFunc("GET /users/{id}", func(w http.ResponseWriter, r *http.Request) {
		id := r.PathValue("id") // Строка из сегмента пути
		if id == "" {
			http.Error(w, "missing id", http.StatusBadRequest)

			return
		}
		// ...
	})
```

Но как же читать тела `POST`-запросов? Выше мы уже обсудили кратко, давай взглянем на пример:

```go
package main

import (
	"encoding/json"
	"log"
	"net/http"
)

type createUserReq struct {
	Name  string `json:"name"`
	Email string `json:"email"`
}

func createUser(w http.ResponseWriter, r *http.Request) {
	if r.Method != http.MethodPost {
		http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
		return
	}
	defer r.Body.Close()

	var req createUserReq
	dec := json.NewDecoder(r.Body)
	dec.DisallowUnknownFields() // Опционально: лишние поля в JSON → ошибка

	if err := dec.Decode(&req); err != nil {
		http.Error(w, "invalid json", http.StatusBadRequest)
		return
	}
	// Пустое тело: Decode вернёт EOF — при желании отдельно проверяйте Content-Length

	w.Header().Set("Content-Type", "application/json")
	_ = json.NewEncoder(w).Encode(map[string]string{
		"ok":    "true",
		"saved": req.Name,
	})
}

func main() {
	http.HandleFunc("/users", createUser)
	log.Fatal(http.ListenAndServe(":8080", nil))
}
```

Тут важно несколько вещей. Во-первых, общение происходит посредством JSON, что позволяет нам использовать структуру как контейнер для данных. И всё благодаря тегам. Во-вторых, нам тут очень помогает `defer`, который закрывает соединение, чтобы программа могла его переиспользовать.

## Маршрутизация запросов

Когда количество маршрутов растёт, управлять ими с помощью глобальных `http.HandleFunc` становится труднее. Кроме того, может понадобиться несколько разных групп маршрутов (например, API и веб-страницы) или запуск нескольких серверов. В таких случаях удобно создавать свой маршрутизатор. Про сам процесс маршрутизации мы вкратце поговорили выше.

В Go существует маршрутизатор по умолчанию. Это тип `*http.ServeMux` (multiplexer), который реализует интерфейс `http.Handler`. Этот тип умеет соотнести URL запроса с зарегистрированными шаблонами путей под капотом сам. В предыдущем параграфе мы фактически использовали глобальный экземпляр `ServeMux` через функции `http.HandleFunc`. Теперь создадим с помощью встроенного функционала Go.

```go
func main() {
    mux := http.NewServeMux()

    // Регистрируем обработчик в маршрутизаторе
    mux.HandleFunc("/weather", weatherHandler)

    fmt.Println("Сервер запущен на 8080...")
    http.ListenAndServe(":8080", mux)
}
```

Или для более сложного случая:

```go
mux := http.NewServeMux()

mux.HandleFunc("/tasks", tasksHandler)          // Список задач
mux.HandleFunc("/tasks/add", addTaskHandler)    // Создание новой задачи
mux.HandleFunc("/tasks/", taskNotFoundHandler)  // Обработчик по умолчанию для подмаршрутов

http.ListenAndServe(":8080", mux)
```

В этом примере мы создали новый маршрутизатор `mux` и зарегистрировали на нём три обработчика. В `ListenAndServe` мы передаём mux вместо `nil`, и сервер будет использовать наш локальный маршрутизатор.

У мультиплексора есть правила сопоставления путей `ServeMux`:

- Самое длинное совпадение по пути выигрывает. Например, запрос `/tasks/add` точно пойдёт в `addTaskHandler`, а не в обработчик `/tasks`, потому что `/tasks/add` более точный матч, чем `/tasks`.
- Если маршрут зарегистрирован с завершающим слешем (`/tasks/`), он воспринимается как префикс для более длинных путей. В примере выше `taskNotFoundHandler` с шаблоном `/tasks/` будет вызван для любого запроса, начинающегося с `/tasks/` и не попавшего под более конкретные маршруты. Это удобно для обработки «Не найдено» или общих случаев.
- Маршрут `/` обозначает обработчик по умолчанию для всех запросов, если ничего более подходящего не найдено.

Таким образом, с помощью нескольких вызовов `mux.HandleFunc` мы организуем маршрутизацию. Код становится чище: мы настраиваем маршруты и описываем функции-обработчики отдельно. Можно даже вынести маршрутизацию в функцию `setupRoutes()` для удобства. Кроме того, при необходимости можно иметь несколько `ServeMux` — например, один для API, другой для статического сайта — и выбирать между ними, но обычно хватает одного.

## HTTP-клиенты на Go

Наш сервис умеет обрабатывать входящие запросы. Но иногда приложению самому нужно выступать в роли клиента, например, чтобы обратиться к внешнему API или другому сервису микросервисной архитектуры. В том же пакете `net/http` есть и клиентские функции.

Проще всего делать запросы с помощью высокоуровневых функций: `http.Get`, `http.Post`, `http.Head` и других. Ещё можно создавать свой объект `http.Client` для настроек или даже низкоуровневые запросы через `http.NewRequest` + `Client.Do`.

Рассмотрим пример `GET`-запроса к внешнему сервису:

```go
import (
    "encoding/json"
    "fmt"
    "net/http"
)

type WeatherResponse struct {  
    CurrentWeather struct {  
       Temperature float64 `json:"temperature"`  
    } `json:"current_weather"`  
}  
  
type Coordinates struct {  
    Lat  float64  
    Long float64  
}  
  
var cityCoordinates = map[string]Coordinates{  
    "Москва": {Lat: 55.7558, Long: 37.6173},  
    "Лондон": {Lat: 51.5074, Long: -0.1278},  
    "Нью-Йорк": {Lat: 40.7128, Long: -74.0060},  
}  
  
func getWeather(city string) (float64, error) {  
    coords, exists := cityCoordinates[city]  
    if !exists {  
       return 0, fmt.Errorf("город не найден")  
    }  
  
    baseURL, err := url.Parse("https://api.open-meteo.com/v1/forecast")  
    if err != nil {  
       return 0, err  
    }  
  
    queryParams := url.Values{}  
    queryParams.Set("latitude", fmt.Sprintf("%f", coords.Lat))  
    queryParams.Set("longitude", fmt.Sprintf("%f", coords.Long))  
    queryParams.Set("current_weather", "true")  
    baseURL.RawQuery = queryParams.Encode()  
  
    resp, err := http.Get(baseURL.String())  
    if err != nil {  
       return 0, err  
    }  
    defer resp.Body.Close()  
  
    var data WeatherResponse  
    if err := json.NewDecoder(resp.Body).Decode(&data); err != nil {  
       return 0, err  
    }  
    return data.CurrentWeather.Temperature, nil  
}
```

Ключевые моменты при использовании HTTP-клиента:

- Функция `http.Get(url)` возвращает (`*http.Response`, error). Ошибка `err` будет не `nil`, только если запрос не смог вообще выполниться (нет сети, DNS, тайм-аут). Если же удалённый сервер ответил с кодом ошибки (404, 500 и так далее), `err` всё равно будет `nil`. Поэтому проверяй `resp.StatusCode` отдельно.
- Всегда закрывай `resp.Body` (через `defer` сразу после проверки ошибки). Иначе соединение может зависнуть и не вернуться в пул, особенно если ответ дочитан не до конца.
- Чтение ответа можно делать через `io.ReadAll`, как выше, или построчно, или сразу парсить JSON: `json.NewDecoder(resp.Body).Decode(&obj)`, если ожидается JSON.
- Если нужен POST-запрос, можно использовать `http.Post(url, contentType, bodyReader)`.
  Например:
```go
resp, err := http.Post("https://httpbin.org/post", "application/json", strings.NewReader(`{"x": 5}`))
```
Это отправит JSON {"x":5} на указанный URL. Но часто удобнее вручную собрать запрос, особенно если нужны нестандартные заголовки.
- Для гибкости можно создать запрос вручную:
```go
req, _ := http.NewRequest("PUT", "http://example.com/resource/42", bodyReader)
req.Header.Set("Authorization", "Bearer TOKEN")
resp, err := http.DefaultClient.Do(req)
```
Здесь мы установили заголовок `Authorization` и отправили запрос через `http.DefaultClient` (стандартный клиент). Напомним, что `DefaultClient` — это готовый `&http.Client{}`. Можно создать и свой, особенно если нужны настройки (тайм-ауты, прокси, кастомный `Transport`).
- Обработка ответа обычно включает проверку кода (StatusCode), чтение или декодирование тела, а также возможное использование заголовков ответа.

Теперь у тебя есть все инструменты, чтобы написать, например, функцию, которая из нашего веб-сервиса задач делает внешний запрос (может быть, к какому-то другому API, скажем, переводу текста задачи или авторизации пользователя). Помни про обработку ошибок на каждом этапе — с сетью всегда может что-то пойти не так.

## Middlewares

При работе с веб-серверами часто возникает потребность выполнять некоторый код до или после основного обработчика маршрута — например, логировать запросы, проверять авторизацию, обрабатывать ошибки единообразно или изменять выходные данные (сжатие, кеширование). Такой шаблон называется **middleware** (промежуточное ПО).

В Go нет специального типа middleware в stdlib, но его легко реализовать самостоятельно, ведь обработчики — это интерфейсы и функции. Самый распространённый подход — написать функцию, которая принимает один `http.Handler` и возвращает другой `http.Handler`, оборачивающий логику вокруг исходного вызова.

Например, реализуем middleware для логирования запросов:

```go
func loggingMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        fmt.Printf("=> %s %s\n", r.Method, r.URL.Path)  // Логируем метод и путь
        next.ServeHTTP(w, r)                            // Вызываем следующий обработчик
        fmt.Println("<= отправлен ответ клиенту")
    })
}
```

Здесь `loggingMiddleware` — функция, возвращающая новый обработчик. При каждом запросе он сначала пишет в лог информацию о запросе, затем вызывает `next.ServeHTTP(w, r)`, а после этого может выполнить ещё какие-то действия (в данном случае просто пишет, что ответ отправлен). Когда обработчик вызывает `next.ServeHTTP(w, r)`, он передаёт обработку реальному маршруту.

Мы можем применять эту функцию к нашим маршрутизаторам или отдельным обработчикам. Например, чтобы логировать абсолютно все запросы на сервере, можно обернуть весь `ServeMux`:

```go
mux := http.NewServeMux()
// ... регистрация маршрутов ...
loggedMux := loggingMiddleware(mux)
http.ListenAndServe(":8080", loggedMux)
```

Теперь каждый запрос будет проходить через `loggingMiddleware` перед тем, как попасть к реальным обработчикам. В консоли будут выводиться строки вида:

```text
=> GET /weather
<= отправлен ответ клиенту
```

Таким же образом можно делать и другие «прослойки». Например, проверку авторизации: middleware читает заголовок `Authorization` или `cookie`, решает, можно ли пускать дальше, и либо возвращает 401, либо вызывает `next`. Другая распространённая прослойка — установка общих заголовков ответа (например, `Content-Security-Policy`) для всех запросов. Это тоже можно сделать через middleware, вызывая `w.Header().Set(...)` до `next.ServeHTTP`.

Важно понимать, что middleware может быть много и их можно компоновать (оборачивать одно другим). При этом порядок имеет значение: тот, что обёрнут внешним, выполнится первым при входе и последним при выходе.

Как в примере ниже:

```go
package main

import (
	"log"
	"net/http"
	"time"

	"github.com/google/uuid"
)

func withRequestID(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		id := uuid.NewString()
		w.Header().Set("X-Request-ID", id)
		next.ServeHTTP(w, r)
	})
}

func withLogging(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		start := time.Now()
		next.ServeHTTP(w, r)
		log.Printf("%s %s (%s)", r.Method, r.URL.Path, time.Since(start))
	})
}

func hello(w http.ResponseWriter, r *http.Request) {
	w.Write([]byte("OK"))
}

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("/", hello)

	h := withRequestID(withLogging(http.HandlerFunc(hello)))
    http.ListenAndServe(":8080", h)
}
```

Здесь мы передаём `hello` как объект, который удовлетворяет интерфейсу, а не как функцию. И в этом случае порядок объявления играет роль. Сначала при входящем запросе сработает функция `withRequestID`, потом `withLogging` и только затем основная функция `hello`.

Как всегда, такую запись можно сделать красивее:

```go
package main

import (
	"log"
	"net/http"
)

// chain склеивает middleware слева направо: сначала outer, потом inner, потом h
func chain(h http.Handler, middlewares ...func(http.Handler) http.Handler) http.Handler {
	for i := len(middlewares) - 1; i >= 0; i-- {
		h = middlewares[i](h)
	}
	return h
}

// ...

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("/", hello)
	root := chain(mux, withLogging, withRequestID)
	log.Fatal(http.ListenAndServe(":8080", root))
}
```

Согласись, что в подобной записи выглядит приятнее. Но помимо стандартной библиотеки, у нас есть и классные сторонние решения:

- [chi](https://github.com/go-chi/chi);
- [gorillamux](https://github.com/gorilla/mux);
- [gorm](https://github.com/go-gorm/gorm) — не роутер, но и в нём есть работа с маршрутизацией.

На курсе мы их использовать не будем, но для кругозора о них всё же стоит знать.

Однако middleware можно реализовать не только у сервера, но и у клиента. Делается это не совсем просто через интерфейс `http.RoundTripper`, который отвечает за разделение ролей в пакете. Если `http.Client` отвечает за удобный API, то `http.RoundTripper` контролирует именно отправку запроса в сеть и чтение ответа. То есть это контракт, описывающий поведение: выполни HTTP-запрос и верни ответ. Это точка расширения между высокоуровневым клиентом и низкоуровневым транспортом. Чем мы и воспользуемся:

```go
package main

import (
	"log"
	"net/http"
	"time"
)

// withAPIKey оборачивает транспорт: перед реальным запросом дописывает заголовок
func withAPIKey(key string, next http.RoundTripper) http.RoundTripper {
	return roundTripperFunc(func(req *http.Request) (*http.Response, error) {
		req = req.Clone(req.Context())
		req.Header.Set("X-API-Key", key)
		return next.RoundTrip(req)
	})
}

// roundTripperFunc — маленький адаптер, чтобы не заводить отдельную struct
type roundTripperFunc func(*http.Request) (*http.Response, error)

func (f roundTripperFunc) RoundTrip(req *http.Request) (*http.Response, error) {
	return f(req)
}

func main() {
	client := &http.Client{
		Timeout: 10 * time.Second,
		Transport: withAPIKey("secret", &loggingTransport{
			next: http.DefaultTransport,
		}),
	}

	resp, err := client.Get("https://example.com")
	if err != nil {
		log.Fatal(err)
	}
	defer resp.Body.Close()
	log.Println("status:", resp.Status)
}

// loggingTransport — вторая обёртка только для цепочки
type loggingTransport struct {
	next http.RoundTripper
}

func (t *loggingTransport) RoundTrip(req *http.Request) (*http.Response, error) {
	log.Println("→", req.Method, req.URL.String())
	return t.next.RoundTrip(req)
}
```

Используя поле `Transport`, ты можешь создавать цепочки middleware так же, как и в сервере. Для статистики и логов, а также прокидывания одинаковых хедеров очень удобно.
