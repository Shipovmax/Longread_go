# Структура проекта: пакеты, модули, зависимости и шаблонная архитектура


Любой серьёзный проект состоит из многих файлов и имеет сложную структуру. По мере роста кодовой базы возникают вопросы: как организовать код так, чтобы он оставался понятным и поддерживаемым? Как обеспечить повторное использование компонентов? Как сделать код гибким и типобезопасным одновременно?

Go предлагает следующие ключевые механизмы для решения этих задач:

- **Пакеты** — механизм организации кода и управления областью видимости;
- **Модули** — управление зависимостями, версиями и границами проекта.

Понимание этих концепций помогает не только структурировать код, но и эффективно работать с зависимостями. Модульная система в Go позволяет управлять зависимостями и их версиями, обеспечивая изоляцию разных частей кода, что значительно упрощает тестирование и интеграцию внешних библиотек.

Пакетная структура, в свою очередь, помогает разбить проект на логические блоки, каждый из которых может быть независимо использован в разных частях программы или других проектах. Это позволяет создать гибкую, масштабируемую и удобную в поддержке архитектуру.


## Пакеты, модули и области видимости


**Пакет** в Go — это набор файлов с одинаковым объявлением `package` в одном каталоге и объединенные общим пространством имен: они компилируются вместе. Это основная единица организации и переиспользования кода. Пакеты используются для организации кода, обеспечения инкапсуляции и повторного использования. Например, пакет `math` содержит математические функции, а пакет `fmt` — функции для форматированного ввода-вывода.

**Модуль** — коллекция пакетов, которая публикуется, распространяется и версионируется как единое целое. Модуль определяется наличием файла `go.mod` в корневой директории и, по сути, в подавляющем большинстве случаев является отдельной программой/сервисом/утилитой.

Модуль может содержать в свою очередь множество пакетов:

```
example.com/my-module # модуль, представляющий собой программу
├── go.mod
├── cmd/        # пакет с точкой входа в программу
│   └── app/
│       └── main.go # точка входа в программу
├── internal/   # пакеты, доступные только внутри этого модуля/программы
│   ├── config/ # пакет для работы с конфигурациями
│   └── server/ # пакет для работы с серверной логикой
└── pkg/        # общедоступные пакеты для прочих программ и сервисов
    ├── database/
    └── logger/
```

Эта структура отражает одну из возможных организаций модуля Go. Обрати внимание, как разные компоненты системы разделены по своему назначению: точки входа в `cmd`, внутренние компоненты в `internal`, и публичные пакеты в `pkg`. Такое разделение помогает поддерживать чистоту архитектуры и понятные границы между компонентами. Далее мы с вами ознакомимся с примерами реализации архитектурных подходов для программ на Go.


### Работа с пакетами


Каждый файл в Go должен начинаться с объявления пакета, к которому он принадлежит (это объявление называется _package clause_):

```go
package config
```

В примере выше мы подразумеваем, что создали как минимум один go-файл в пакете `config`. Но в одном пакете может находиться столько файлов, сколько тебе захочется иметь для поддержания его функциональности. Для примера можно обратиться к [исходному коду Golang](https://github.com/golang/go/tree/go1.25.7/src). Особый случай — объявление пакета `main`. Пакеты с таким объявлением служат точкой входа для исполняемых программ и содержат функцию `main`, зачастую, пребывая в единичном экземпляре в одном модуле (но порой возникает необходимость держать несколько main.go файлов в одном модуле - подумай, зачем?):

```go
package main

func main() {
    // Начало выполнения программы
}
```

Вообще, в Go принято называть директории и пакеты внутри них одинаково (в данном случае под пакетом мы имеем в виду не просто папку, а именно _package clause_). Почему мы вдруг об этом упомянули? Никто не запрещает тебе объявить директорию с названием `config`, а внутри создать несколько go-файлов, которые будут иметь отличное наименование у пакета - например, `configuration`. Но по правилам хорошего тона, _package clause_ внутри всех файлов директории `config` нужно обозвать именно `config`! Кстати, если в директории создать хотя бы 2 go-файла с разными наименованиями пакетов, то компилятор будет ругаться, так как **наличие файлов с разными наименованиями пакетов внутри одной директории запрещено!**

Но ведь пакеты, как было сказано выше, создаются не просто так. Они являются обособленными элементами кодовой базы проекта, которые могут друг с другом взаимодействовать. Для использования кода из других пакетов используется оператор `import`:

```go
package main

import (
	"fmt"                        // стандартный пакет
	"my-module/internal/config"  // внутренний пакет
	"github.com/sirupsen/logrus" // внешний пакет
)

func main() {
	cfg, err := config.Load()
	if err != nil {
		logrus.WithError(err).Fatalln("loading config")
	}
	fmt.Printf("Config loaded: %+v\n", cfg)
}
```

Для того, чтобы использовать методы стандартной библиотеки, достаточно просто написать название пакета - `net/http`. Для использования другого внутреннего пакета проекта требуется вставить в импорт **путь пакета**, являющийся строкой, которая идентифицирует пакет и используется для его импорта. Обычно это путь модуля, за которым следует подкаталог, содержащий пакет (относительно корня модуля) - `my-module/internal/config`. Путь пакета может быть абсолютным (например, `golang.org/x/net/html`) или относительным (используется для пакетов из стандартной библиотеки, как видно в примере выше).

В случае, если требуется затянуть зависимость внешнего пакета, используется встроенная в Go команда `go get ...` о которой мы поговорим чуть позже.

При импорте пакетов, **важно знать и помнить**, что Go запрещает циклические импорты пакетов. Например, если пакет `A` импортирует пакет `B`, а пакет `B` импортирует пакет `A`, компилятор выдаст ошибку:

```
import cycle not allowed
```

Это необходимо, чтобы обеспечить быструю компиляцию, создать предсказуемую иерархию зависимостей и предотвратить архитектурный беспорядок. В случае разрешения возможностей циклических импортов, то будет невозможно (или же невероятно сложно) определить порядок компиляции, а пакеты перестают становиться изолированными единицами.

Для разрешения циклических зависимостей обычно используют одну из стратегий:
1. Выделение общего кода в третий пакет;
2. Использование интерфейсов вместо прямых зависимостей;
3. Переосмысление архитектуры компонентов.


#### Области видимости пакетов


Пакеты позволяют не только группировать код по зонам ответственности, но и разделять свой функционал на внешний и внутренний относительно других пакетов. Как мы уже знаем, в Go видимость идентификаторов (переменных, функций, типов) определяется **регистром первой буквы** их наименования. Это напрямую относится к пакетам:

- **Экспортируемые** (видимые из других пакетов) идентификаторы начинаются с **заглавной буквы**;
- **Неэкспортируемые** (видимые только внутри пакета) — с **маленькой буквы**.

```go
package user

// Экспортированный тип (виден извне)
type User struct {
	Name string // экспортированное поле
	age  int    // неэкспортированное поле
}

// Экспортированная функция
func NewUser(name string, age int) User {
    return User{
        Name: name,
        age:  age,
    }
}

// Неэкспортированная функция
func validate(u User) bool {
    return u.Name != "" && u.age > 0
}
```

В этом примере мы видим ключевой принцип инкапсуляции в Go. Структура `User` экспортирована и доступна извне пакета, но поле `age` скрыто. Это создаёт контролируемый API: клиентский код должен использовать функцию `NewUser` для создания объектов, что позволяет гарантировать валидность данных. Функция `validate` не экспортирована, поэтому может использоваться только внутри кода самого пакета.

Код из других пакетов может использовать только экспортированные сущности:

```go
package main

import (
    "fmt"
    "my-module/user"
)

func main() {
	u := user.NewUser("Alice", 30)

	fmt.Println(u.Name) // доступно

	fmt.Println(u.age)  // ошибка компиляции
	user.validate(u)    // ошибка компиляции
}
```

Здесь демонстрируется использование пакета извне. Клиентский код может создать пользователя через экспортированный конструктор и обратиться к экспортированному полю `Name`, но попытки получить доступ к неэкспортированному полю `age` или вызвать внутреннюю функцию `validate` приведут к ошибкам компиляции. Такой механизм защищает внутреннюю реализацию пакета от непредвиденных изменений.


#### Функция `init`


Каждый пакет может содержать функции `init`, которые выполняются на стадии инициализации пакета:

```go
package database

import "log"

var db *Database

func init() {
    // Выполнится один раз при импорте пакета
    log.Println("Initializing database...")
    db = &Database{}
}

type Database struct {
    // ...
}
```

Функция `init` - особая в Go. Она выполняется при инициализации программы, если пакет вошёл в сборку (то есть был импортирован в main или другие пакеты, которые взаимодействуют с main напрямую или через импорт в другие пакеты, работающие с main). Функция часто используется для настройки глобальных переменных или регистрации драйверов. В этом примере мы инициализируем глобальный экземпляр базы данных. Однако будьте осторожны с `init`: код в нём выполняется неявно и может усложнить тестирование. Для сложной инициализации с возможными ошибками лучше использовать явные конструкторы.

Важные особенности `init`:
- Может быть несколько функций `init` в одном пакете;
- Выполняются после инициализации глобальных переменных;
- Порядок выполнения: сначала в импортируемых пакетах, затем в текущем;
- Не принимают параметров и ничего не возвращают.

Но эти функции могут быть очень опасны:
- Если забыть про то, в каком порядке они выполняются при импортах пакетов друг в друга, то можно наткнуться на "необъяснимое" и неявное поведение, которое придется дебажить;
- Такие функции довольно сложно тестировать из-за их природы;
- Зачастую, `init` используется для объявления чего-то глобального (например, переменных), однако, это во многих языках считается плохим тоном;
- Сложная схема очереди срабатывания `init` в одном пакете может запутать и снова привести к "необъяснимому" поведению;
- Откровенно говоря, когда функция ничего не возвращает, то непонятно, как правильно в таком случае обрабатывать ошибки, которые могут возникать внутри. Таким образом `init` зачастую просто паникует, без возможности правильной обработки некорректной ситуации.


#### Именование пакетов


При проектировании пакетов следует избегать "статтеринга" (stuttering) - повторения имени пакета в именах его экспортируемых идентификаторов. При импорте имя пакета уже служит префиксом. Плохо делать так:

```go
package user

func UserCreate() { /* ... */ }
func UserDelete() { /* ... */ }
```

При использовании этих идентификаторов в других пакетах предстоит писать так: `user.UserCreate()` — что является избыточным. Лучше делать иначе:

```go
package user

func Create() { /* ... */ }
func Delete() { /* ... */ }
```

Тогда в других пакетах предстоит писать так: `user.Create()` — что является понятным и не избыточным. Ты обращаешься к пакету `user` и просишь создать пользователя через `Create()` - код читается, как строка в книге.

Также приветствуется избегание обобщенных названий пакетов, вроде `common`, `helpers`, `utils`. При использовании этих наименований в коде ты не будешь понимать, к чему именно относится этот пакет. Наименование пакетов следует выстраивать емко и по возможности коротко, описывая названием зону ответственности пакета (`pricing`, `database`). И снова, ориентиром может служить исходный код Go.


### Модули


Как мы уже поняли, **модули** - это коллекция пакетов, которая по большей части является самодостаточной программой. Благодаря тому, что Go использует систему модулей Go Modules, которая является современной системой управления зависимостями, работа с внешними библиотеками становится простой и безопасной. Давай разберемся, как это всё работает.

Каждый Go-проект начинается с простой команды:

```bash
# Создание нового модуля
go mod init github.com/username/project-name
```

После выполнения этой команды Go создает два важных файла: `go.mod` и `go.sum`.

- **go.mod** — файл описания модуля и зависимостей;
- **go.sum** — файл с криптографическими хешами зависимостей.

Остановимся подробнее на примере файла `go.mod`:

```go
module github.com/username/project-name

go 1.25

require (
    github.com/gin-gonic/gin v1.9.1
    github.com/golang-migrate/migrate/v4 v4.16.2
    github.com/lib/pq v1.10.9
    github.com/spf13/viper v1.16.0
    gorm.io/driver/postgres v1.5.2
    gorm.io/gorm v1.25.4
)

require (
    github.com/bytedance/sonic v1.9.1 // indirect
    github.com/chenzhuoyu/base64x v0.0.0-20221115062448-fe3a3abad311 // indirect
    // ... другие косвенные зависимости
)
```

Файл `go.mod` состоит из списка используемых в проекте/модуле зависимостей, делящихся на нескольких секций, каждая из которых выполняет определённую функцию:

**a)** `module`

Определяет **имя модуля** — уникальный идентификатор проекта в экосистеме Go:

```go
module github.com/username/project-name
```

Go автоматически умеет загружать зависимости напрямую из git-репозиториев по указанному пути. Это работает благодаря интеграции Go с системами контроля версий:

```go
module github.com/username/project-name  // Go будет искать репозиторий на github.com
module gitlab.com/company/team/project   // Go будет искать на gitlab.com
module golang.org/x/text                 // Go будет искать на golang.org
```

Как это работает:
1. Go определяет хостинг-платформу по доменному имени (github.com, gitlab.com и т.д.);
2. Обращается к API платформы для получения информации о репозитории;
3. Скачивает указанную версию или коммит;
4. Кэширует зависимость локально.

Для публикуемых библиотек (которые будут использоваться как зависимости):
```go
module github.com/username/my-library  // Публичная библиотека
module company.com/libraries/utils     // Корпоративная библиотека
```

Для непубликуемых проектов можно использовать простые имена:
```go
module my-private-app        // Приватное приложение
module internal-dashboard    // Внутренний проект
module company-project       // Название без домена тоже работает
```

**Важные моменты:**
- При использовании доменного имени Go ожидает найти действующий git-репозиторий;
- Для приватных проектов можно использовать любое имя, но рекомендуется указывать домен для будущей совместимости;
- Изменение имени модуля потребует обновления всех импортов в проекте.

**b)** `go`

Указывает **минимальную версию Go**, требуемую для сборки проекта:

```go
go 1.21
```

Таким образом:
- Гарантируется совместимость с используемыми возможностями языка;
- Предотвращается сборку на устаревших версиях Go.

**c)** `require`

Перечисляет **прямые и косвенные зависимости** проекта:

```go
require (
    github.com/gin-gonic/gin v1.9.1           // прямая зависимость
    github.com/golang-migrate/migrate/v4 v4.16.2 // прямая зависимость
    github.com/bytedance/sonic v1.9.1 // indirect  // косвенная зависимость
    github.com/chenzhuoyu/base64x v0.0.0-20221115062448-fe3a3abad311 // indirect
)
```

**Прямые зависимости** — пакеты, которые импортируются напрямую в коде проекта:

```go
import "github.com/gin-gonic/gin"  // прямая зависимость
```

**Косвенные зависимости** (`// indirect`) — пакеты, которые требуются прямым зависимостям, но не используются напрямую в коде проекта.

**Почему `// indirect` важно:**
- Позволяет отличать зависимости, которые можно безопасно удалить;
- Упрощает управление зависимостями при рефакторинге;
- Помогает понять дерево зависимостей проекта.

**d)** `replace`

Иногда случаются ситуации, когда стандартная зависимость нам не подходит. Может быть, ты нашел баг в популярной библиотеке и хочешь его исправить, или разрабатываешь новую функциональность и тестируешь её с локальной версией пакета. На помощь приходит секция `replace`!

Она позволяет подменить любую зависимость на другую версию или даже на локальный файл.

```go
replace github.com/old/package => github.com/new/package v1.2.3
replace github.com/local/package => ../local/package
```

Эта секция особенно полезна в нескольких сценариях:
- Когда ты отлаживаешь проблему и хочешь использовать локальную версию пакета для быстрого тестирования исправлений;
- Если ты создал форк популярной библиотеки и добавил в неё свою функциональность;
- Когда нужно временно исправить баг в зависимости, ожидая официального патча от авторов.

**e)** `exclude`

А что если определённая версия зависимости содержит критическую уязвимость или просто "сломана"? Секция `exclude` работает как страж на границе — она говорит Go: "Эту версию загружать не нужно, пропускай!"

```go
exclude github.com/problematic/package v1.5.0
```

Это особенно ценно, когда:
- Ты знаешь о конкретной уязвимости в версии v1.5.0 и хочешь предотвратить её использование;
- Некоторые версии содержат баги, которые ломают твою сборку;
- Ты хочешь гарантировать, что команда случайно не использует проблемную версию.

Но помимо `go.mod`, модуль содержит и файл `go.sum`. Он содержит **криптографические хеши** всех зависимостей проекта и обеспечивает безопасность и воспроизводимость сборок. Структура файла go.sum:

```
github.com/gin-gonic/gin v1.9.1 h1:4idEAncQnU5cB7BeOkPtxjfCSye0AAm1R0RVIqJ+Jmg=
github.com/gin-gonic/gin v1.9.1/go.mod h1:hPrL7Yrp6KXt5YId3A/Tnip5kqbEAP+KLuI3SUcPTeU=
github.com/bytedance/sonic v1.9.1 h1:6iJ6NqdoxCDr6mbY8h18oSO+cShGSMRGCEo7F2+0Y8=
github.com/bytedance/sonic v1.9.1/go.mod h1:i736AoUSYt75HyZLuJZ0SSRlHkmaJ2Kl9N6Fz5rD3k=
```

Каждая строка содержит:
1. **Путь к модулю** — имя пакета;
2. **Версия** — конкретная версия или коммит;
3. **Хеш** — криптографическая подпись пакета.

*Зачем нужен этот файл?*

Представь, что ты заказываешь товары из интернет-магазина. `go.sum` — это как чек с уникальными номерами каждого товара. Когда курьер привозит заказ, ты проверяешь, что номера на товарах совпадают с номерами в чеке. Если кто-то пытается подменить товар, ты сразу это заметишь!

Файл `go.sum` выполняет три важные функции:

1. **Надёжная защита** — каждый раз при сборке проекта Go проверяет, что скачанные пакеты точно соответствуют хешам в `go.sum`. Если злоумышленник попытается подменить какую-то зависимость на вредоносный код, Go немедленно остановит сборку с ошибкой;
2. **Идеальная воспроизводимость** — все разработчики команды получают абсолютно одинаковые версии всех зависимостей. CI/CD системы собирают проект с теми же зависимостями, что и на твоей машине;
3. **Полная история версий** — `go.sum` хранит хеши не только прямых зависимостей, но и всех косвенных, включая конкретные коммиты. Это позволяет отследить, какие именно версии пакетов использовались в любой момент времени.

Тебе не нужно создавать или редактировать этот файл вручную — Go делает это автоматически:
- Когда ты впервые добавляешь зависимость командой `go get`;
- При обновлении существующих зависимостей;
- Во время выполнения `go mod tidy`.

**Важный совет:** Всегда включай `go.sum` в git! Этот файл — твоя страховка от неожиданностей и гарантия безопасности проекта.


#### Работа с зависимостями


Теперь объединим знания о зависимостях в пакетах и модулях. Общий пайплайн может быть таким:

```bash
# Добавление последней версии пакета
go get github.com/gin-gonic/gin

# Добавление конкретной версии
go get github.com/gin-gonic/gin@v1.9.1

# Добавление последней версии пакета
go get github.com/stretchr/testify@latest

# Обновление всех зависимостей
go get -u ./...

# Удаление неиспользуемых зависимостей
go mod tidy
```

`go get` - команда, которая позволяет добавить зависимость в модуль, проставляя ее в `go.mod` и хеш зависимости в `go.sum`. Команда запускает как раз тот самый механизм поиска и подключения зависимостей, который мы обсудили ранее. Как видно в примере выше - `go get` может скачивать зависимости с точностью до версии или коммита. И это всегда более предпочтительный путь использования зависимости, чем использование `last`, который может при вызове `go.mod` неожиданно обновить библиотеку на менее стабильную версию;
`go mod tidy` - команда, которая убирается в зависимостях нашего модуля. Она сортирует зависимости в `go.mod`, обновляет их и удаляет неиспользуемые зависимости из модуля, подчищая `go.mod`.

Не забывай вызывать время от времени `go mod tidy` внутри своего проекта.

Еще существует команда `go install`, которая **собирает и устанавливает исполняемые файлы** (команды) в директорию `$GOBIN` (по умолчанию — `$GOPATH/bin`). Чтобы потом запускать установленные утилиты из терминала, добавь `$GOBIN` (или `$(go env GOPATH)/bin`) в `PATH`.

Самый частый сценарий — поставить **внешние инструменты разработки** (линтеры, генераторы, форматтеры) так, чтобы они **не трогали `go.mod` твоего проекта**. Для этого указывай версию через `@...`:

```bash
# форматтер/инструменты x/tools
go install golang.org/x/tools/cmd/goimports@latest

# линтер (пример — версию можно закрепить)
go install github.com/golangci/golangci-lint/cmd/golangci-lint@v1.55.2
```


### Важная информация о базовых пакетах


Мы узнали, как дробится код в Go на логические блоки на уровне архитектуры. И перед тем, как перейти к самой архитектуре и узнать, как можно комбинировать эти блоки, нам необходимо узнать пару интересных вещей про пакеты `internal` и `pkg`.

Go предоставляет особый механизм для ограничения доступа к пакетам модуля — любой пакет внутри директории `internal` доступен только для пакетов этого модуля:

```
my-module/
├── internal/
│   └── database/  // доступен только внутри my-module
├── api/
│   └── handler.go // может импортировать my-module/internal/database
└── client/
    └── main.go    // может импортировать my-module/internal/database
```

Как видно, внутри самого проекта использование пакета `internal/database` допускается, однако, при импорте модуля `my-module` в любой другой проект, все, что скрыто в `internal` нельзя будет использовать. Таким образом мы можем защитить наш код от неправомерного или же некорректного использования напрямую из разрабатываемого модуля. Тем не менее, код из `internal` можно без проблем использовать во всех пакетах на уровне определения этого пакета.

Здесь есть еще одна уловка: `internal` можно определить не только на верхнем уровне модуля, как в примере, но и внутри вообще любого пакета на любом уровне вложенности! В этом случае остается работать правило: импортировать код из `internal` можно только в рамках родительского пакета и его подпакетов.

```
my-module/
├── internal/
│   └── database/           // доступен только внутри my-module
│       ├── internal/
│       │   └── anypackage/ // доступен только внутри my-module/internal/database
│       └── postgres/       // может импортировать my-module/internal/database/internal/...
├── api/
│   └── handler.go // может импортировать my-module/internal/database, но не my-module/internal/database/internal/...
└── client/
    └── main.go    // может импортировать my-module/internal/database, но не my-module/internal/database/internal/...
```

Но что, если все-таки необходимо поделиться с миром каким-то пакетом из текущего сервиса/модуля? В этом случае можно использовать пакет `pkg`:

```
my-module/
├── internal/
│   └── database/  // доступен только внутри my-module
├── pkg/
│   └── csv_converter/  // доступен и внутри my-module, и вовне его при импорте my-module в другие модули
├── api/
│   └── handler.go // может импортировать my-module/internal/database
└── client/
    └── main.go    // может импортировать my-module/internal/database
```

Go позволяет использовать подпакеты пакета `pkg` в прочих проектах. Но тут стоит быть осторожным: дело в том, что пакет `pkg/` часто превращается в «свалку общих утилит», а также у зависящих модулей могут возникнуть проблемы с версионированием и происходить жесткая привязка к конкретному общему пакету конкретного сервиса. Поэтому использовать `pkg` лучше всего как можно реже и выносить его подпакеты в отдельные общие библиотеки, когда ими начинают пользоваться множество сервисов.


## Структура проекта на Go


Правильная организация проекта — это основа разработки поддерживаемого и масштабируемого кода. Правильная организация проекта помогает разработчикам легко ориентироваться по кодовой базе, снижает когнитивную нагрузку, ускоряет ревью и упрощает онбординг новых людей на проект. Она также поможет различать границы ответственности при использовании пакетов другими разработчиками и поможет им избежать нежелательных зависимостей.

Важным концептом Go является его минималистичность: простота, читаемость и предсказуемость важнее «универсальных» шаблонов. Важно, чтобы выбранная структура проекта (также именуемая как `layout`) была понятна команде и не мешала инструментам экосистемы.

Только вот ключевая мысль по структуре Go-проекта заключается в том, что у Go нет «единственно правильной» структуры проекта. Важно, чтобы команда заранее договорилась о наборе структурных паттернов и последовательно придерживалась их. Универсальные гайды полезны, но как «делать правильно» для твоего проекта ты узнаешь на работе: через практику, код-ревью и локальные соглашения. В Open-Source проектах часто фиксируют соглашения, в том числе по структуре проекта, в файле `CONTRIBUTING.md`. Важно также не бояться эволюционировать структуру по мере роста проекта.

Однако, все же некоторые рекомендации по выстраиванию архитектуры сервисов на Go имеются:

1. Существует официальный гайдлайн по организации кода под названием [«Organizing a Go module»](https://go.dev/doc/modules/layout). Это не строгий стандарт, а рекомендации от авторов Go по организации кода внутри модуля.

Документ описывает несколько типовых паттернов, наиболее часто встречаемыми из которых являются:
- [`Basic command`](https://go.dev/doc/modules/layout#basic-command) - минимальный модуль из одного пакета `main` в корне;
- [`Server project`](https://go.dev/doc/modules/layout#server-project) - модуль с поддержкой нескольких точек входа и серверной логикой в пакетах под директорией `internal`.

**Basic command** - базовая исполняемая программа (CLI-утилита) организуется в зависимости от сложности и объёма кода. Самый простой вариант - один файл Go, в котором определена `func main`. В программах побольше код можно разделить на несколько файлов, все они объявляют `package main`:

```
project-root-directory/
  go.mod
  auth.go
  auth_test.go
  client.go
  main.go
```

Точку входа обычно по конвенции объявляют в файле `main.go`. Данный layout применяют для небольших программ, для которых не требуется разделение логики по пакетам.

**Server project** - часто выбирают для реализации серверов. Обычно серверные проекты не экспортируют пакеты для внешнего использования: сервер - это, как правило, самодостаточная исполняемая программа (или несколько). Поэтому в официальном гайде рекомендуют хранить логику сервера под директорией `internal`. Кроме того, поскольку в репозитории часто будет встречаться много директорий с не-Go кодом, удобно держать все точки входа вместе в каталоге `cmd`. Структура Go кода будет выглядеть следующим образом:

```
project-root-directory/
  go.mod
  internal/
    auth/
      ...
    metrics/
      ...
    model/
      ...
  cmd/
    api-server/
      main.go
    metrics-analyzer/
      main.go
    ...
  ... остальные директории проекта с не-Go кодом
```

Разберём плюсы такой структуры:
- В данной структуре явно понятно, что нет экспортируемых пакетов Go - четко видно, что Go код написан в директориях `internal/` и `cmd/`. Серверные проекты часто замкнутые и поэтому им не требуется экспортировать пакеты;
- Отдельно выделена директория `cmd/` с точками входа;
- Все остальные директории могут использоваться как угодно - различать Go код они никак не помешают. Например, в корень проекта можно добавить `docker/`, `chart/`, `.gitlab/` и прочее.

В связи со всеми удобствами, которые предоставляет данный layout, на самом деле большинство других layout'ов используют за основу его: они добавляют лишь дополнительные директории со своим смыслом.

2. Проект [golang-standards/project-layout](https://github.com/golang-standards/project-layout) представляет собой рекомендации по организации структуры проектов на Go. Это **неофициальный** стандарт, который широко используется сообществом для организации крупных и сложных проектов. Однако, он не является стандартом, как таковым. Расс Кокс [официально закрепил](https://github.com/golang-standards/project-layout/issues/117), что подобные проекты не могут быть стандартами, так как у самого Go нет стандартов, как таковых.

Однако, этот репозиторий является неплохой точкой отсчета для того, чтобы подглядеть, как еще можно организовывать код сервисов и программ на большом масштабе.

3. [Исходный код Go](https://github.com/golang/go/tree/go1.25.7) на самом деле является хорошим ориентиром того, как организовывать свой код, называть пакеты, структурировать их самих и внутренние файлы. Да, когда ты погрузишься в изучение этой структуры, то сможешь найти противоречия идиомам Go и тому, как иногда написан и организован сам исходный код. Но ведь в любых правилах бывают исключения :)

4. В мире программирования существуют более общие подходы к архитектуре кодовой базы:
- DDD;
- onion architecture;
- clean architecture;
- CQRS

и некоторые другие. О них стоит иметь общее представление, потому что их даже поверхностное знание пригодится не только при работе с Go, но и с другими языками программирования.

В нашем курсе мы предлагаем опираться на «Organizing a Go module» и использовать `Basic command` и `Server project` там, где это необходимо.

Однако, стоит подсветить, что многие библиотеки (наборы пакетов, которые предоставляют конкретную функциональность твоей программе просто за счет импорта, например, нужной экспортируемой функции), обходятся вообще без `main.go`, а их основная архитектура, это как раз либо `Basic command`, либо `Server project` (если библиотека предлагает разные варианты функционала (например, оплата картой, купонами, в рассрочку)), но без пакета `cmd` и `main.go` - компилировать библиотеку не требуется. Требуется просто взять от нее функционал.
