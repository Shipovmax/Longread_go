# Файлы конфигурации, работа с json и yaml

Теперь, когда ты умеешь собирать небольшие cli-проекты и работать с кодом в асинхронном режиме, пора переходить к сетевому взаимодействию. Но перед этим важно понять, как передавать начальные условия работы сервисов, другими словами их конфигурацию, правильно.

Да, у нас есть фича в виде флагов, которые мы можем передавать программе при её старте в командной строке. И как будто бы это удобная и валидная история. Однако подобный сценарий передачи конфигурации хорош именно для cli-приложений, потому что в реальной жизни в сервис необходимо передать довольно много данных для его корректной работы.

Это могут быть:

- URL'ы для подключения к сторонним приложениям;
- ключи для подключения к базам данных;
- тайм-ауты;
- настройки inmemory-кешей;
- уровни логирования и так далее.

А ещё не стоит забывать, что в разработке реальных сервисов программисту приходится работать как минимум с тремя окружениями:

- локальным;
- stage, тестовым;
- продовым.

И это означает, что на каждый из этих контуров нам необходимо иметь свои отдельные настройки и конфигурации.

Поэтому в этой теме мы погрузимся в то, как правильно работать с конфигурациями приложений. Нам очень сильно поможет тема про работу с файлами, но для начала вернёмся к структурам и разберёмся с некоторыми их особенностями, которые помогают взаимодействовать с внешними файлами.

## Файлы конфигураций и теги структур

Начнем с самих файлов: какие форматы в большинстве своём мы используем в разработке. Но для начала нам необходимо задать два очень важных определения, которые будут важны не только для лонгрида и Golang, но и для разработки в принципе.

Так как нам приходится гонять данные в разных форматах туда-сюда, для каждого отдельного случая нам требуется выбирать какой-то один формат данных, а затем преобразовывать его в другой формат. Эти операции преобразования называются сериализацией и десериализацией.

**Сериализация** - процесс преобразования объекта или данных в формат, который можно сохранить, передать или восстановить позже (struct → JSON/YAML, `json.Marshal`). А **десериализация** - это обратный процесс, когда из строки или файла снова создаётся объект (JSON/YAML → struct, `json.Unmarshal`).

Вот теперь мы готовы двигаться дальше.

### JSON

Скорее всего, тебе уже встречался `JSON`-формат. Вообще, `JSON` расшифровывается как JavaScript Object Notation, что раскрывает его как выходца из JS, где он используется для обмена данными. Это текстовый формат, использующий удобное разбиение по полям, обёрнутым в фигурные скобки, формирующие `JSON`-объект:

```json
{
	"port": ":8080",
	"log_level": 2
}
```

То есть это множество `ключей:значений`, где ключи — это названия полей/параметров. Значения могут иметь простой тип, вроде строк, целочисленных значений и чисел с плавающей точкой, логические значения и `null`. Также мы можем хранить по ключу массив значений; множество, также являющееся `JSON`-объектом, или же массив `JSON`-объектов. Важно все `ключи:значения` разделять запятой, кроме последней пары.

```json
[
    {
        "name": "Андрей",
        "age": 33,
        "isEmployed": true
    },
    {
        "name": "Вячеслав",
        "age": 21,
        "isEmployed": false
    }
]
```

Файлы такого формата довольно легковесны и являются де-факто стандартом в современной разработке, ими пользуются для структурированного хранения и передачи данных. `JSON` имеет неплохую читаемость для человека и поддерживается практически всеми языками.

Умение работать с таким форматом — это важный и обязательный навык современных разработчиков.

### YAML

`YAML` — это ещё более дружественный человеку текстовый формат сериализации данных. Он подходит для удобного хранения конфигурации больше, чем `JSON`, в целом легко конвертируясь в него и из него. `YAML` характерен тем, что обладает минималистичным синтаксисом, работающим через отступы для отображения иерархии данных, позволяя группировать данные удобнее, чем в том же `JSON`:

```yaml
server:
    port: 8080
    host: localhost
app:
    log_level: 2
    enabled_test: true

test: null
dev: ["local", "stage", "prod"]
```

Как мы видим, тут реализована та же концепция `ключ:значение`, как и в `JSON`, но читается всё приятнее. Теперь мы оперируем списками и работаем с вложенностью через отступы. Этот формат зачастую используется для создания не только файлов конфигурации, но и протоколов автоматизации. В отличие от `JSON` формат `YAML` поддерживает комментарии, что также упрощает работу с ним.

Кстати, заметь, что в примере выше `localhost` не заключён в кавычки, однако это не помешает программам распознать в значении строку. Также можно использовать и явную типизацию типов в файле:

```yaml
this_is_string: !!str 67.43
```

Объявляя `!![тип]`, мы даём понять, что именно ожидаем при парсинге файла.

Также `YAML` имеют возможность создавать внутри себя якоря/алиасы, тем самым позволяя переиспользовать значения.

Якоря начинаются с `&` перед неймингом:

```yaml
default: &default_settings
    port: 8080

local:
    <<: *default_settings
    host: localhost
```

В целом формат позволяет работать с данными намного глубже и вариативнее, чем это позволяет делать `JSON`. Именно поэтому он и используется для конфигурационных файлов больше.

Фан-факт: ты можешь встретить файлы с двумя различными форматами — `.yml` и `.yaml`. Оба правильные, но предпочтительнее использовать второй вариант, так как это современный вариант обозначения формата. `.yml` использовался в те времена, когда формат файла ограничивался тремя символами.

### Теги структур

Как же знание о форматах файлов поможет нам далее? Всё из-за структур. Структуры — самый качественный способ организации данных одного и того же логического порядка, каковые в том числе данные конфигурации. Для того чтобы мы могли без усилий складывать отдельные элементы конфигураций в структуры, их поля должны содержать определённые теги, по которым рантайм поймёт, как и куда складывать данные из файлов.

Например:

```go
type Config struct {
	Port int
	Host string
}
```

Обычная такая структура, которая нам вряд ли чем-то поможет. Делаем небольшое движение рукой и получаем:

```go
type Config struct {
	Port int    `json:"port" yaml:"port"`
	Host string `json:"host" yaml:"host"`
}
```

Теги — это метаданные полей. По факту строковый атрибут, который через [рефлексию](https://go.dev/blog/laws-of-reflection) позволяет обозначать типовую идентичность полей. Теги — общий механизм, а `json`- и `yaml`-теги — частные случаи. И как ты видишь, одно поле может иметь несколько тегов. При этом имя тега не обязательно должно совпадать с именем поля. Они могут быть абсолютно разными, но при этом тег позволяет смаппить это поле с соответствующим файлом.

Если говорить про `JSON`, то пакет `encoding/json` по умолчанию сериализует только **экспортируемые** поля структуры, используя имена полей, как имя json-ключа, если тег этому полю не задан. Если поле в структуре не экспортируемое, то и json-пакет с ним работать **не будет**.

```go
type Config struct {
	Port int    `json:"port" yaml:"port"`
	host string `json:"host" yaml:"host"` // Даже с тегом в этом поле ничего не сериализуется
}
```

Но json-теги могут быть не просто именем ключа/поля. Ты можешь управлять сериализацией более гибко:

```go
type User struct {
    Name     string `json:"name"`             // Обычный кейс
    Surname  string `json:"last_name"`	      // Разные имена у поля и тега
    Age      int    `json:"age,omitempty"`    // Если поля нет в json, то оно будет иметь дефолтное значение типа. Если создавать json-файл, то такого поля в файле может не быть
    Password string `json:"-"`                // Поле полностью исключается из рассмотрения
    Address  string `json:",omitempty"`       // Наименование ключа по названию поля. Если поля нет в файле, то пустая строка в значении. При создании файла и пустой строки в значении поля в файле не будет
    Phone    string `json:",string"`          // Наименование ключа по названию поля. Преобразование данных в json-строку
    HasJob   bool   `json:"has_job,omitzero"` // О механике omitzero и omitempty ниже
    Foo      *int   `json:"foo"`              // Значение, которого 
}
```

Здесь стоит отдельно остановиться на `omitempty` и `omitzero`. Они призваны делать примерно одно и то же, но по-разному. Оба тега работают с нулевыми значениями (`zero value`), но `omitempty` пропускает значения при создании json-объектов, если эти значения как раз таки дефолтные для типа — `zero values`. А вот `omitzero` позволяет убирать именно нулевые значения для всех типов, с которыми не мог работать `omitempty`. Например, с `time.Time{}`.

В целом json-объекты путешествуют в коде Go в виде слайса байт до момента, пока не сериализуются в структуру или не десериализуются из неё. Важный код, которым мы будем часто пользоваться при работе с `JSON`:

```go
response := answer{Key: "value"}

b, err := json.Marshal(response) // Преобразуем go-объект в json-строку (байты)
if err != nil {
	panic(err)
}

fmt.Println(string(b)) // {"key":"value"}

k := answer{}

err = json.Unmarshal(b, &k) // Обратное действие: превращение json-строки в структуру
if err != nil {
	panic(err)
}
```

С yaml-тегами всё немного сложнее, потому что Go из-под капота не работает с `YAML`. Для работы с этим форматом раньше использовался отдельный пакет `go-yaml` — [на github.com](https://github.com/go-yaml/yaml/tree/v3.0.1). На данный момент он находится в архиве, но его разработка продолжилась в [форке](https://github.com/yaml/go-yaml). Также имеется отдельная [альтернатива](https://github.com/goccy/go-yaml).

Но архивирование проекта не значит, что им нельзя пользоваться. Просто библиотека зафиксировала максимально стабильное поведение и версию, после чего разработку можно не продолжать.

В целом теги `YAML` похожи на `JSON`:

```go
type Config struct {
    Port int     `yaml:"port"`
    Host string `yaml:"host,omitempty"`
}
```

Но на сцену выходит новый тег `inline`, который позволяет встраивать структуру в структуру при сериализации, так как не все `YAML`-библиотеки умеют встраивать данные автоматически:

```go
type HTTPConfig struct {
    Host string `yaml:"host"`
    Port int    `yaml:"port"`
}

type Config struct {
    HTTPConfig `yaml:",inline"`
    Debug      bool `yaml:"debug"`
}
```

Но это может работать и не работать в разных библиотеках. В целом прочитать yaml-конфиг можно довольно просто:

```go
package main

import (
    "fmt"
    "os"
    "log"

    "gopkg.in/yaml.v3"
)

type Config struct {
    Server   ServerConfig   `yaml:"server"`
    Database DatabaseConfig `yaml:"database"`
}

type ServerConfig struct {
    Port int    `yaml:"port"`
    Host string `yaml:"host"`
}

type DatabaseConfig struct {
    Driver string `yaml:"driver"`
    Conns  int    `yaml:"connections"`
}

func main() {
    yamlFile, err := os.ReadFile("config.yaml")
    if err != nil {
        log.Fatalf("Ошибка чтения файла: %v", err)
    }

    var config Config
    err = yaml.Unmarshal(yamlFile, &config)
    if err != nil {
        log.Fatalf("Ошибка парсинга: %v", err)
    }

    fmt.Printf("Сервер запущен на %s:%d\n", config.Server.Host, config.Server.Port)
}
```

При работе с тегами стоит обращать внимание на не всегда очевидные вещи. **Во-первых**, структуры с одинаковыми полями, но разными тегами — это разные типы!

Их нельзя сравнить:

```go
a := struct {
	Name string `json:"name"`
}{}
b := struct {
	Name string `json:"names"`
}{}

fmt.Println(a == b) // Не скомпилится, пока у b не поменяется тег на name
```

**Во-вторых**, тег не равен логике приложения и никак в ней не участвует.

### JSON vs YAML

Если сравнивать между собой два формата, то пока предлагаем использовать следующие правила:

- `JSON` стоит выбрать, когда нужен строгий, простой, машинный формат;
- `YAML` требуется при наличии большого конфига и частой редактуры человеком.

## Переменные окружения

Ещё одним источником конфигурации для приложений могут служить переменные окружения. Это набор строковых пар `key=value`, хранящихся в операционной системе вне кода программ, то есть обеспечивающих универсальность параметров в различных средах для набора программ, запускаемых в них.

При этом ещё раз обратим внимание на то, что переменные окружения не относятся к Go-фичам. Они именно что общий механизм ОС и среды запуска. Именно поэтому их очень удобно использовать и для локальной разработки, и в Docker/Kubernetes, и для запуска бинарника вообще.

Как и прочие способы конфигурирования, переменные окружения позволяют программе понять, в каком окружении они находятся без необходимости менять сам код. Но при этом и без необходимости раскладывать данные о конфигурациях по файлам. В [Twelve-Factor App](https://12factor.net/ru/), методологии создания современных облачных SaaS-приложений от Heroku, рекомендуется хранить изменяемую между деплоями конфигурацию в окружении, а не в коде. *Зачем?*

У переменных окружения есть три сильные стороны:

- Они легко меняются без пересборки самой программы.
- Они не привязаны к языку или платформе.
- Их удобно подставлять при запуске, не храня секретные данные в исходном коде.

Twelve-Factor App дополнительно подчёркивает, что такой способ хранения уменьшает риск закоммитить некорректное изменение чувствительных настроек в репозиторий и позволяет проводить масштабирование приложений удобнее.

В Go, кстати, есть отдельная команда `go env`, которая раскрывает значения переменных окружения, относящихся к среде языка. Попробуй ввести команду в своём терминале или терминале IDE.

Переменные окружения можно считывать прямо из кода Go. Например, в MacOS можно добавить свою переменную в `~/.zshrc`, записав в конец:

```bash
export APP_PORT=":8080"
```

После сохранения и выхода из файла используем команду:

```bash
source ~/.zshrc
```

В мини-программе на Go ты можешь обнаружить значение этой переменной:

```go
func main() {
    fmt.Println(os.Getenv("APP_PORT"))
}
```

Но эта функция вернёт пустую строку, если не найдёт переменной в окружении. Для разрешения ситуации, есть переменная или нет, стоит использовать функцию `os.LookupEnv()`, которая вернёт значение, если оно есть, и флаг существования переменной.

**Важно** ещё и то, что переменные окружения — это строки. А значит, после получения значения переменной, его стоит провалидировать и преобразовать в нужный тип. Поэтому ошибки при работе с переменными окружения по большей части — ошибки в контракте между окружением и приложением. То есть это не Go плохой, а возможно нарушение формата значения у переменной. Поэтому все данные конфигурации необходимо сначала проверить перед использованием.

Как использовать эти переменные, если их много и нам необходим какой-то артефакт, чтобы хранить их в нём? На помощь приходит стандартный инструмент в виде *текстового* файла `.env`. Это обычный текстовый файл, который на каждой строке хранит данные в виде `key=value`. И это не стандартный формат Go или какого-то инструмента. Это именно что текстовый файл, который используется для локальной разработки. При этом сам `.env` не окружение! Он только хранит переменные, которые можно загрузить в окружение.

Существует полезная для такого подхода работы библиотека `joho/godotenv`, которая через функцию `godotenv.Load()` загружает переменные из файла `.env` в саму систему, а мы уже через `os.Getenv` или более продвинутые инструменты можем загрузить конфигурации в код.

```go
import "github.com/joho/godotenv"

func main() {
    godotenv.Load() // Загружает переменные из .env в систему
    val := os.Getenv("MY_VAR")
}
```

Но мы часто встречаемся с ситуацией, когда для одного сервиса может существовать один-другой десяток переменных. Что делать с этим? Вызывать на каждый чих `os.Getenv` как-то грустно.

На встречу приходит ещё одно решение в виде библиотеки `caarlos0/env` и тегов полей структуры:

```go
package main

import (
    "fmt"
    "log"

    "github.com/caarlos0/env"
    "github.com/joho/godotenv"
)

// Определение структуры с тегами `env`
type Config struct {
    DBHost  string `env:"DB_HOST"`
    DBPort  int    `env:"DB_PORT"`
    APIKey  string `env:"API_KEY" envDefault:"12345678"` // Значение по умолчанию
    Debug   bool   `env:"DEBUG"`
}

func main() {
    // Загружаем .env-файл (если он есть)
    if err := godotenv.Load(); err != nil {
        log.Println("No .env file found, relying on environment variables")
    }

    // Создаём экземпляр структуры
    cfg := Config{}

    // Парсим переменные окружения в структуру
    if err := env.Parse(&cfg); err != nil {
        log.Fatalf("Error parsing env: %v", err)
    }

    // Теперь переменные в структуре
    fmt.Printf("Config: %+v\n", cfg)
    fmt.Println("Database Host:", cfg.DBHost)
}
```

Нам снова пригодились теги в структурах, благодаря которым библиотека понимает, куда и что определять. Заметь, что используется специальный тег `envDefault`, который говорит о том, что если такой переменной нет, то библиотека будет взаимодействовать с кодом и подставлять указанное значение. Это удобно, когда мы хотим перестраховаться и обозначить параметрами по умолчанию важные для нас характеристики системы.

Когда стоит использовать `.env`? В случае локальной разработки или в CI. Это некоторый способ автоматизации ручных определений переменных на конкретных машинах или для конкретных ситуаций (тесты в gitlab). В остальном переменные либо прописываются в контейнерах Docker, либо в отдельных `.yaml`-файлах, которые выбираются при деплое автоматически, исходя из того, в какой контур (прод или стейдж) происходит разворачивание сервиса.

Также существует набор правил, которые позволяют нам правильно называть переменные окружения:

- всегда в верхнем регистре;
- с понятным префиксом приложения: APP_PORT, APP_DB_URL;
- без слишком общих имён.

И ещё кое-что. В `.env` и прочих файлах лучше не хранить секреты и чувствительные данные конфигурации. Подобные вещи хранятся в отдельных системах вроде `Vault`.

В итоге переменные окружения — это внешний, стандартный и удобный способ передать конфигурацию процессу. `.env` лишь вспомогательный локальный инструмент, который помогает эти переменные подготовить, но не заменяет механизм окружения. В коде же лучше всего считывать переменные при старте приложения и хранить их в едином месте вроде структуры `Config`.

## Дефолтные значения

Немного про значения по умолчанию. Когда мы работаем с различными источниками конфигурации приложения, будь то флаги, файлы или переменные окружения в контейнерах, **всегда** стоит предусмотреть для критичных параметров конфигурации дефолтные значения.

В случае с `.env`, это специальный тег. При работе с флагами мы тоже можем устанавливать значение по умолчанию:

```go
port := flag.Int("port", 8080, "Порт сервера") // Имя, дефолтное значение, описание
```

Для остальных случаев можно использовать валидацию и подстановку объявленных значений в коде. Подобный ход поможет обезопасить выполнение твоего кода.

## Пайплайн сборки конфигурации для проектов

1. Создаём единую структуру для хранения конфигурации

Это главный момент при работе с параметрами из кода. Тебе необходимо где-то хранить данные для корректной работы приложения, и где, если не в единой структуре? Напомним, что именно они являются типами объединения данных по здравому смыслу.

2. Прочитать файл конфига

Если указан файл конфига, то он считается первым источником истины. Данные оттуда стоит внести в созданную структуру.

3. Применить env-переменные

Напомним, что мы можем вынести определённые параметры вне кода. В случае если они больше относятся к системным настройкам, то такие параметры стоит вынести в переменные контейнера/ОС.

4. Применить флаги

Всё-таки порой один-два флага при запуске программы мы используем. В таком случае надо не забыть о них.

5. Использовать дефолтные значения

Для важных параметров, влияющих на работу системы, предусмотреть во всех случаях дефолтные значения.

6. Провалидировать параметры конфигурации

Это безумно важный шаг. Именно тут мы закладываем безопасность для наших параметров.

Например:

- проверка, что порт в допустимом диапазоне;
- таймауты положительные;
- URL и пути хотя бы не пустые, а лучше всего можно проверить, что они отличают определённому паттерну.

Приложение **должно** упасть как можно раньше, если с ним что-то не то. В данном случае мы должны упасть, если данные некорректны. Лучше завершиться с ошибкой, чем работать неправильно и подводить клиентов.

## Хранение секретов

У приложения есть обычные настройки, например: порт; лог-уровень; адрес; количество данных, хранящихся во внутреннем кеше. А есть так называемые секреты. Это токены, пароли, ключи доступа. И они тоже часть конфигурации. Но только обращаться с ними нужно осторожнее.

Как минимум:

- не хардкодить внутри исходного кода программы;
- не класть в git;
- не писать в лог.

И хранить их необходимо иначе. Для этого можно использовать разные программы. Мы уже упоминали про `Vault`. Это инструмент для безопасного хранения и выдачи секретов. Его используют только для чувствительных данных, храня их централизованно для всех приложений компании. Об этом стоит знать, но в наших учебных проектах его мы использовать, конечно, не будем.

## Полезная библиотека

Для работы с файлами разных форматов и `.env` одновременно можно использовать удобную библиотеку `spf13/viper`. Она позволяет работать с большим диапазоном форматов и источников параметров конфигураций из коробки, тем самым облегчая жизнь разработчикам.

Для нас это тоже не обязательный инструмент на курсе. Но он позволяет не писать часть обвязки руками.
