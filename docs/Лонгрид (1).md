# Составные типы данных. Знакомство с исходным кодом Go


## Массивы

**Массив** в Golang, как и во многих других языках, является неким контейнером, имеющим фиксированный размер и хранящим элементы одного типа. В памяти массив представлен последовательной областью ячеек, в которых и хранятся значения массива.

Размер массива задаётся во время его объявления и является частью типа. Это означает, что массивы разной длины являются разными типами и, несмотря на то что сам по себе массив является сравниваемым типом (с ним можно проводить операции сравнения *==* и *!=*), сравнить массивы с разной длиной ты не сможешь:

```go
var numbers3 [3]int  // Массив из трёх целых чисел
var numbers5 [5]int  // Массив из пяти целых чисел

numbers3 == numbers5 // Ошибка компиляции: разные типы
```

Важно и то, что подобный составной характер типизации массивов не позволяет нам приводить массив одной длины к другому типу массива с отличной длиной. А ещё длиной массива не может быть переменная: так как она — часть типа, то компилятор должен знать это значение на этапе компиляции.

Выше размещён один из примеров объявления массива. В случае использования ключевого слова *var* созданный массив будет содержать столько `zero value` значений типа (помнишь из первой лекции?), сколько было указано в квадратных скобках.

```go
var temperatures [7]float64  // Семь температур на неделю
var scores [5]string         // Пять оценок студента

fmt.Println(temperatures)   // [0 0 0 0 0 0 0] — нулевые значения для float64
fmt.Println(scores)         // ["" "" "" "" ""] — нулевые значения для string
```

Но есть и другие способы. Мы можем инициализировать все значения массива сразу, либо заполнять его частично, либо создавать так называемые разреженные массивы — когда мы назначаем значения в его конкретных индексах:

```go
// Все значения сразу
weekdays := [7]string{
    "Понедельник", "Вторник", "Среда", "Четверг",
    "Пятница", "Суббота", "Воскресенье",
}

// Частичная инициализация значений
grades := [5]int{95, 87, 92} // [95 87 92 0 0] — остальные нули

// Разреженный массив
scores := [4]int{1, 2:14} // [1 0 14 0] — мы указали, что в третьем индексе (нумерация с 0) должно быть значение 14
```

Также существует опция не указывать длину массива. В таком случае вместо числа в квадратных скобках указывается троеточие и компилятор вычисляет длину массива самостоятельно, что полезно, когда мы не можем точно знать, какой длины нам потребуется массив:

```go
// Автоматическое определение длины массива
months := [...]string{
    "Январь", "Февраль", "Март", "Апрель",
    "Май", "Июнь", "Июль", "Август",
    "Сентябрь", "Октябрь", "Ноябрь", "Декабрь",
}
```

### Длина и ёмкость массива

Для того чтобы узнать параметры массива, такие как длина и ёмкость, существуют встроенные в язык функции:

- `len` — [функция](https://pkg.go.dev/builtin#len), которая возвращает длину значения, то есть количество элементов в типе, которые он в себе непосредственно содержит;
- `cap` — [функция](https://pkg.go.dev/builtin#cap), которая возвращает ёмкость значения, то есть максимальное количество элементов, которое тип может содержать. Эта функция ориентируется на выделенное место в памяти для этого типа и подсказывает, на сколько ещё элементов осталось памяти у типа.

В случае с массивом мы имеем соответствие этих функций `len(x) == cap(x)` благодаря природе самого массива.

```go
cities := [5]string{"Москва", "Долгопрудный"}
fmt.Println(len(cities)) // 5
fmt.Println(cap(cities)) // 5
```

### Работа с массивом

Несмотря на то что массивы разных типов друг с другом не взаимодействуют, мы можем копировать значения из массива в массив, если они имеют один и тот же тип. В таком случае мы получаем независимые друг от друга переменные:

```go
a := [3]int{1, 2, 3}
b := a
b[0] = 99
fmt.Println(a) // [1 2 3]
fmt.Println(b) // [99 2 3]
```

В примере выше видно, как работать с отдельными значениями массива. Для этого используются **индексы** — порядковые номера значений в массиве, которые позволяют получить доступ к нужному элементу, начиная с `0`. Благодаря индексам мы можем читать отдельные данные и изменять их.

```go
cities := [3]string{"Москва", "СПб", "Казань"}

fmt.Println("Первый город:", cities[0]) // Москва
fmt.Println("Второй город:", cities[1]) // СПб
fmt.Println("Третий город:", cities[2]) // Казань

// Изменение элемента
cities[1] = "Санкт-Петербург"
fmt.Println(cities) // [Москва Санкт-Петербург Казань]
```

И если ты попытаешься обратиться к индексу за пределами длины массива, то получишь ошибку:

```go
fmt.Println(cities[5]) // panic: runtime error: index out of range [5] with length 3
```

В связи с тем, что массив — составной тип, по нему можно пройтись циклом. Классический вариант цикла по массиву использует счётчик от 0 до `len(x)-1` (так как длина показывает общее количество элементов, а индекс начинается с 0 - почему так: [Why numbering should start at zero](https://www.cs.utexas.edu/~EWD/transcriptions/EWD08xx/EWD831.html)):

```go
prices := [4]float64{19.99, 25.50, 12.30, 45.00}

for i := 0; i < len(prices); i++ {
    fmt.Printf("Товар %d стоит %.2f руб.\n", i+1, prices[i])
}
```

Этот способ опасен тем, что мы можем указать неверный индекс окончания итерирования, который будет больше диапазона индексов массива, что приведёт к ошибке в рантайме. Поэтому более идиоматичный способ — использование `for-range`:

```go
fruits := [4]string{"яблоко", "банан", "апельсин", "груша"}

// С индексом и значением
for index, fruit := range fruits {
    fmt.Printf("%d. %s\n", index+1, fruit)
}

// Только значения (если индекс не нужен)
for _, fruit := range fruits {
    fmt.Println("Фрукт:", fruit)
}

// Только индексы (если значение не нужно)
for index := range fruits {
    fmt.Printf("Индекс %d\n", index)
}
```

Символ `_` — это пустой идентификатор, который сообщает компилятору, что требуется игнорировать значение. Переменные же в цикле являются копиями тех значений, что хранит массив, поэтому их изменение не приведёт к изменению значения в самом массиве. **Всегда используй** для изменения значений массива индексы.

### Ограничения массивов

- **Фиксированный размер**: после объявления массива изменить его размер нельзя.
- **Размер — часть типа**: `[10]int` и `[20]int` несовместимы.
- **Копирование при передаче**: дорого для больших массивов.

```go
func processNumbers(numbers [100]int) { // Что, если нужно 200 чисел?
    // Обработка 100 чисел
}

func processMoreNumbers(numbers [200]int) { // Другая функция!
    // Обработка 200 чисел
}

// Нужны две разные функции для разных размеров массивов!
```

Именно поэтому массивы в Go редко находят практическое применение в чистом виде.


## Слайсы: гибкая альтернатива массивам

**Слайсы**, или **срезы**, — очень похожи на массивы на первый взгляд:
- слайсы создаются/объявляются так же, как массивы (почти);
- обращение к элементам слайсов также происходит через индексы;
- они имеют точно такой же механизм итерирования, что и массивы;
- выход за пределы диапазона индексов также приводит к ошибке.

Но то, что они очень похожи, не значит, что у них нет серьезных отличий. **Отличие слайсов от массивов №1**: слайсы имеют динамический размер. А это уже означает очень многое. Во-первых, что слайс не имеет длины, как часть типа, и объявляется без числа в квадратных скобках:

```go
var numbers []int  // Пустой слайс целых чисел
var names []string // Пустой слайс строк
```

Такой слайс не содержит элементов и является не инициализированным (под его значения не будет выделена память), то есть не будет иметь элементов вообще, имея значение `nil` (к нему мы вернёмся). Истинно пустой массив можно создать иначе:

```go
empty := []int{}
```

Но для слайса появляется новый вариант создания — через встроенную [функцию](https://pkg.go.dev/builtin#make) `make`, являющуюся идиоматичным способом объявления слайсов. Она принимает на вход тип слайса, который надо создать, количество элементов этого слайса и его общую ёмкость.

```go
// make([]T, length) - создать слайс с заданной длиной
numbers := make([]int, 5) // [0 0 0 0 0], len=5, cap=5

// make([]T, length, capacity) - создать слайс с заданной длиной и ёмкостью
buffer := make([]int, 0, 10) // [], len=0, cap=10

fmt.Printf("numbers: len=%d, cap=%d, %v\n", len(numbers), cap(numbers), numbers)
// numbers: len=5, cap=5, [0 0 0 0 0]
fmt.Printf("buffer: len=%d, cap=%d, %v\n", len(buffer), cap(buffer), buffer)
// buffer: len=0, cap=10, []
```

Как ты видишь, в первом случае мы создаем слайс из пяти элементов значения по умолчанию, так как не определяем эти значения сразу — нам нужен просто контейнер с начальными значениями, чтобы начать с ним работу. Во втором случае мы получаем пустой слайс, но с аллоцированной памятью на 10 элементов, которую мы можем начать заполнять. И как видишь, функции `len` и `cap` работают иначе, чем у массива: `len` будет возвращать количество элементов в слайсе (если грубо, то `len` вернёт количество заполненных индексов в диапазоне ёмкости), а `cap` — количество индексов, которые мы зарезервировали для использования (сумма уже заполненных и незаполненных индексов).

Соответственно, раз слайс имеет динамический размер, мы можем наполнять его значениями до того момента, пока не кончится оперативная память. Но как это сделать? Тут на сцену выходит встроенная [функция](https://pkg.go.dev/builtin#append) для увеличения слайса — `append`. Она принимает первым аргументом исходный слайс, а далее — произвольное количество элементов, которые мы хотим добавить:

```go
var shopping []string
fmt.Println("Список покупок:", shopping) // []
fmt.Println(len(shopping)) // 0

shopping = append(shopping, "хлеб")
fmt.Println("Добавили хлеб:", shopping) // [хлеб]
fmt.Println(len(shopping)) // 1

shopping = append(shopping, "молоко", "яйца")
fmt.Println("Добавили еще:", shopping) // [хлеб молоко яйца]
fmt.Println(len(shopping)) // 3
```

Но **нужна внимательность**! `append` возвращает *новый* слайс! Поэтому результат его вычисления нужно присваивать обратно переменной, которую мы расширяем (как это сделано в примере). Но что происходит под капотом? `append` работает по одному из двух сценариев:

1. Если в слайсе есть свободное место (`len < cap`), слайс не расширяется и `append` дописывает элемент в слайс, увеличивая длину.
2. Если места нет (`len == cap`), создаётся новый слайс, туда копируются старые элементы и добавляется новый — именно поэтому надо присваивать результат выполнения `append` той же переменной, которая содержала слайс, над которым производится работа.

Тут встает закономерный вопрос: на сколько **слайс вырастает** во втором случае? Как он выбирает конечное значение ёмкости? В исходном коде Golang реализован метод *growSlice* ([ссылка на GitHub](https://github.com/golang/go/blob/go1.25.6/src/runtime/slice.go#L177)), который вызывается под капотом во время вызова `append` в теле твоей программы. Именно в этом методе реализована оптимизация: слайсу при росте выдаётся больше памяти, чем ему требуется, дабы не обращаться к аллокатору за новой памятью постоянно и тратить на это время. В методе *nextslicecap* ([ссылка на GitHub](https://github.com/golang/go/blob/go1.25.6/src/runtime/slice.go#L289)) выбирается новое значение ёмкости, которое потом уточняется в *growSlice*. В общем виде этот алгоритм претерпевает изменения раз в несколько версий языка, так что об этом стоит знать, но не полагаться на реализацию.

На самом деле **тебе будет полезно** в целом ознакомиться с документом [slice.go на GitHub](https://github.com/golang/go/blob/go1.25.6/src/runtime/slice.go) — в нём содержится вся логика работы со слайсом во время выполнения программ.

**Отличие слайсов от массивов №2**: слайсы — это надстройка над массивами! И это снова видно в исходном коде [на GitHub](https://github.com/golang/go/blob/go1.25.6/src/runtime/slice.go#L15):

```go
// Упрощённая концепция представления слайса выглядит так:
type slice struct {
    array unsafe.Pointer // Указатель на первый элемент
    len   int            // Длина
    cap   int            // Ёмкость
}
```

Начнём с того, что слайс под капотом Go — это структура. Ты познакомишься с ней чуть позже, но пока нам важно понимать, что слайс — это совсем не массив внутри самого языка. Слайс — это два его параметра: длина и ёмкость, — и указатель (ты скоро поймёшь, что это, но упрощенно: указатель указывает на ячейку в памяти, в которой лежит значение) на первый элемент базового массива слайса. То есть данные у слайса лежат в массиве, на который слайс ссылается напрямую в память. И тут мы можем углубить знание работы `append`: когда количество элементов становится больше `cap`, то создаётся новый массив в новой области памяти, туда переносятся данные, а старый массив удаляется.

**Отличие слайсов от массивов №3**: слайсы — несравниваемый тип, в отличие от массивов. Ни один слайс ты не сможешь сравнить ни с чем другим через *==* и *!=*.

**Отличие слайсов от массивов №4**: массив остаётся массивом, слайс/срез можно брать от массива и от слайса. Этот процесс можно назвать реслайсом — когда мы вместо создания базового массива в памяти указываем языку, что именно ему считать за базовый массив слайса.

В данном примере мы специально добавили первым элементом пустую строку, чтобы индекс внутри слайса соответствовал порядковому номеру месяца в году и было проще ориентироваться:

```go
months := [12 + 1]string{
    "", "January", "February", "March", "April", "May", "June",
    "July", "August", "September", "October", "November", "December",
}

var (
    allMonths  = months[:]       // Все элементы
    firstHalf  = months[1 : 6+1] // Первые 6 месяцев года
    secondHalf = months[7:]      // Последние 6 месяцев года
    summer     = months[6 : 8+1] // Июнь, июль, август
)
fmt.Println("Лето:", summer)   // [June July August]
fmt.Println("Первая половина:", firstHalf) // [January February Match April May June]
```

Синтаксис среза `[start:end]` означает «взять элементы с индекса `start` до индекса `end`, не включая `end`». Если `start` опущен — берём с начала, если `end` опущен — до конца. Все эти слайсы ссылаются на **один и тот же** базовый массив! И имеют доступ к тем элементам, которые являются элементами слайсов. У каждого слайса могут быть и общие, и различные элементы, которые они «видят». И это открывает важную возможность слайсов: если брать слайс от массива/слайса, то в случае изменения значения элемента, который присутствует и там, и там, его изменение будет видно везде, где он присутствует.

Но срез можно взять чуть более сложно через синтаксис `[start:end:maxCap]`, где `maxCap` - это ограничение ёмкости нового базового массива на основе среза. Под капотом действует такая формула: `newCap = start - maxCap` - так и определяется ёмкость для нового среза. Если `maxCap` не указано (как в примере выше), то в таких случаях ёмкость считается по обычной формуле: `len(slice/array)-start`.

```go
summer[0] = "ИЮНЬ"                              // Изменяем слайс
fmt.Println("Исходный массив:", months[6])      // ИЮНЬ - массив изменился!
fmt.Println("Первая половина:", firstHalf[6-1]) // ИЮНЬ - и тут изменилось!
```

Это показывает, что если изменить элемент через один слайс, то это сразу повлияет на базовый массив и станет видно во всех остальных слайсах! Такое поведение называется ссылочной семантикой — слайс лишь указывает на данные в памяти, а не копирует их. В этом же примере мы так же можем наглядно наблюдать, как работает capacity — это длина от первого элемента слайса до конца базового массива внутри слайса.

### Ещё важное о слайсах

Иногда тебе потребуется создать независимый слайс от же существующего. Очевидный выход — скопировать значения. Самый примитивный — пройтись по одному слайсу итеративно, копируя значения по одному в новый слайс. Но за нас уже подумала core-команда Golang и создала встроенную функцию `copy`, которая копирует данные из одного слайса в другой:

```go
func copyExample() {
    original := []string{"a", "b", "c", "d", "e"}

    // Создаём слайс-получатель нужного размера
    destination := make([]string, len(original))

    // Копируем элементы
    n := copy(destination, original)
    fmt.Printf("Скопировано %d элементов\n", n) // 5

    // Изменяем копию — оригинал не изменится
    destination[0] = "MODIFIED"

    fmt.Println("Оригинал:", original)     // [a b c d e]
    fmt.Println("Копия:   ", destination)  // [MODIFIED b c d e]
}
```

Функция `copy` безопасно копирует `min(len(src), len(dst))` элементов. Это значит, что она скопирует столько элементов, сколько поместится в слайс-получатель.

Мы ещё не поговорили про удаление элемента из слайса. Такой функции в Golang нет, но можно выполнить удаление самостоятельно, «склеив» с помощью `append`  части слайса, не включающие в себя удаляемый элемент/элементы:

```go
func removeElement(slice []string, index int) []string {
    // Объединяем части до и после удаляемого элемента
    return append(slice[:index], slice[index+1:]...)
}

func main() {
    fruits := []string{"яблоко", "банан", "апельсин", "груша", "киви"}
    fmt.Println("До удаления:", fruits)

    // Удаляем элемент с индексом 2 (апельсин)
    fruits = removeElement(fruits, 2)
    fmt.Println("После удаления:", fruits) // [яблоко банан груша киви]
}
```

Что происходит в функции `removeElement`:
1. `slice[:index]` — все элементы до удаляемого (не включая его).
2. `slice[index+1:]` — все элементы после удаляемого.
3. `append` склеивает эти две части.

Оператор `...` в данном случае распаковывает слайс в вариадический параметр: он «раскрывает» для рантайма слайс поэлементно. Такой вот синтаксический «сахар».

И последнее: многие функции, которые нужны для работы со слайсами, уже лежат в стандартной библиотеке языка в пакете `slices` ([ссылка на GitHub](https://github.com/golang/go/tree/master/src/slices)). Изучи его самостоятельно.

### Немного о nil

`nil` — специальное нулевое значение для _ссылочных типов_. Оно означает «ничего», или «пустой указатель». Можно думать о `nil` как о «несуществующем значении». Концепцию `nil` мы подробно изучим далее, когда будем говорить про указатели и интерфейсы. Слайс со значением `nil` имеет `len` и `cap`, равные `0`. Такой слайс также может быть передан в функцию `append`.


## Мапы: встроенные хеш-таблицы

Мапа — это ещё один встроенный составной тип данных, который предназначен для хранения ассоциативных данных в виде пары ключ — значение, где и ключ, и значение могут быть разными типами. Единственное ограничение — ключи должны быть сравниваемыми.

```go
var m map[int]string
```

В примере выше **важно то**, что мапа хоть и объявлена, но **не инициализирована**. То есть она имеет значение `nil`, и в случае попытки записи в такую мапу ты получишь ошибку. Как с ней работать, мы посмотрим далее.

### Сравниваемые типы

Сейчас самое время отвлечься и вкратце рассказать про сравниваемость типов. Этой темы мы уже касались, но давай формализуем её. Итак, сравниваемые переменные можно сравнивать друг с другом, если их типы сравниваемы. То есть могут использовать операторы `==` и `!=`.

Из типов, что нам известны, сравниваемыми являются (и могут являться ключами мапы):
- bool;
- string;
- int, int8, int16, int32, int64;
- uint, uint8, uint16, uint32, uint64, uintptr;
- float32, float64;
- complex64, complex128;
- массивы.

Несравниваемые типы, о которых мы знаем: слайсы и мапы. По ходу курса мы расширим наши знания об этих двух группах типов.

### Работа с мапами

Ты уже знаешь, как создать неинициализированную мапу. Но нам надо понять, как создавать инициализированную. Для этого используются два подхода. Первый работает с уже известной нам функцией `make`, и с её помощью мы можем создать и пустую мапу, и мапу, у которой уже будет инициализировано некоторое количество памяти под данные. По факту для мапы задаётся только её начальная емкость. Второй способ — напрямую заполнить мапу при её создании:

```go
m1 := make(map[int]bool)     // Пустая инициализированная мапа
m2 := make(map[bool]int, 10) // Инициализированная мапа с аллоцированной памятью на 10 значений
m3 := map[float32]bool{1.2: true, 1.5: false} // Инициализированная мапа с фактическими значениями
```

Чтение и запись осуществляются тривиально — через обращение по ключу. Удаление ключа происходит через встроенную [функцию](https://pkg.go.dev/builtin#delete) `delete`. Но что, если мы хотим проверить существование ключа? Тут существует хитрость: если у мапы запросить ключ, который она не содержит, то она вернёт `zero value` типа, который существует у её значения. Чтобы не попасть в такую неприятную ситуацию, можно проверить существование ключа через нотацию `value-ok`, где ok — это булевое значение, которое говорит нам о том, есть ли в мапе проверяемый ключ или нет. А ещё можно проверить количество элементов в мапе через уже знакомую функцию `len()`. `cap()` для мапы не работает:

```go
m := make(map[int]bool)
m[1] = true       // Запись
fmt.Println(m[1]) // Чтение: true
delete(m, 1)      // Удаление

v, ok := m[2] // v - это всегда значение по ключу: zero value или фактическое существующее
if !ok {
    fmt.Println("ключа нет!")  // Если такого ключа нет (а его нет), то пишем об этом
}

var m1 map[int]int

fmt.Println(len(m1), len(m)) // Мы получим в обоих случаях 0. Это потому, что неициализированная мапа (по факту nil) тоже имеет размер 0. Так что хорошей практикой является проверять мапу не на nil, а на длину
```

Ещё один фокус с мапами проявляется тогда, когда нам необходимо обойти её в цикле. Мапы устроены так, что при их обходе мы каждый раз будем получать рандомный (недетерминированный) порядок данных. Исключение — если ты положишь мапу аргументом в функцию `fmt.Println`, внутри функции реализована сортировка ключей, поэтому ты всегда будешь получать один и тот же вывод в консоль. Попробуй!

```go
for k, v := range m {  // k - ключ, v - значение. Порядок получения рандомен
    fmt.Println(k, v)
}
fmt.Println(m) // Всегда один и тот же порядок данных
```

### Внутреннее устройство

Тебе повезло. На данном этапе развития языка мапы являются удивительным примером его эволюции. Совсем недавно на собеседованиях спрашивали про внутреннее устройство мап и все зубрили ответы, но в версии 1.24 мапа изменилась! Нет, она осталась в коде такой же, какой и была. Даже ответ, что мапа — это хеш-таблица, у которой существует хеш-функция, подбирающая подходящее место для данных внутри мапы, остался корректным. Поэтому большую часть информации о внутреннем устройстве мап лучше изучить самому — ссылки мы приложили в конце материала.

Самое важное, что надо помнить: мапа — это структура. Как и слайсы. И вообще, очень многое в коде Go есть структуры, с которыми мы познакомим тебя очень скоро. Мапы тут не исключение, и теперь, чтобы посмотреть на неё в исходном коде, надо смотреть в два места (*по крайней мере до версии 1.25*): [новая версия](https://github.com/golang/go/blob/go1.25.6/src/runtime/map_swiss.go) и [старая реализация](https://github.com/golang/go/blob/go1.25.6/src/runtime/map_noswiss.go). Давай всё же вкратце опишем устройство мап до версии 1.24 и после неё. Но не забудь изучить самостоятельно!

#### Мапа до версии 1.24

До версии Go 1.24 встроенный тип map реализовывался как классическая хеш-таблица с бакетами. В основе структуры лежал массив бакетов, каждый из которых содержал фиксированное количество слотов для хранения пар ключ — значение. Один бакет вмещал до восьми элементов. Если при вставке все слоты бакета оказывались заняты, создавался дополнительный overflow-бакет, который связывался с основным через указатель. Таким образом, коллизии разрешались с помощью хеширования и цепочек overflow-бакетов, образующих связанный список.

Для ускорения операций поиска и вставки использовался механизм tophash. Для каждого элемента в бакете хранился дополнительный байт, представляющий собой старшие биты хеша ключа. При поиске сначала сравнивался именно этот байт, и только при совпадении выполнялось полноценное сравнение ключей. Поскольку вероятность совпадения tophash составляет 1/256, большинство неподходящих элементов отсеивалось на раннем этапе, что существенно снижало число дорогих операций сравнения ключей.

Количество бакетов в таблице всегда выбиралось равным степени двойки. Это позволяло вычислять индекс бакета с помощью побитового маскирования, избегая операции взятия остатка. По мере роста числа элементов карта расширялась: при достижении коэффициента загрузки примерно 81% или  появлении слишком длинных цепочек overflow-бакетов запускался процесс увеличения таблицы. Новый массив бакетов имел вдвое больший размер, однако перенос элементов выполнялся инкрементально. При каждой последующей операции над map переносилась лишь небольшая часть бакетов, что исключало
длительные паузы и обеспечивало предсказуемое время выполнения операций.

Дополнительной особенностью реализации было использование случайного зерна хеширования. При создании каждой map выбирался случайный seed, который участвовал в вычислении хешей ключей. Это делало распределение элементов по бакетам непредсказуемым и защищало программы от атак типа hash-flooding, при которых злоумышленник пытается намеренно создать
большое число коллизий.

#### Мапа после версии 1.24

Начиная с версии Go 1.24, тип реализован на основе структуры данных Swiss Table, представляющей собой вариант хеш-таблицы с открытой адресацией. Эта реализация существенно отличается от прежней бакетной модели и направлена
на снижение накладных расходов памяти и повышение производительности операций доступа.

В основе Swiss Table лежит разбиение таблицы на группы по восемь слотов. Каждая такая группа управляется 64-битным контрольным словом, в котором каждый байт соответствует одному слоту. В этом байте закодировано состояние слота (пустой, занятый, удалённый) и семибитный фрагмент хеша ключа, обычно обозначаемый как h2. Все элементы мапы размещаются в одном
непрерывном массиве памяти, без использования цепочек указателей. При возникновении коллизий используется линейное пробирование: если целевой слот занят, поиск или вставка продолжаются в следующих слотах и
группах.

Контрольное слово играет ключевую роль в ускорении поиска. Оно позволяет за одну операцию сравнить h2 сразу для восьми слотов. На современных архитектурах это реализуется с использованием SIMD-инструкций, что даёт возможность мгновенно отфильтровать неподходящие слоты и свести число полных сравнений ключей к минимуму. Даже на платформах без SIMD
поддержка реализована через оптимизированный цикл по байтам, который остаётся существенно быстрее последовательной проверки каждого элемента. Почему реализовано именно так? Всё дело в кэш-линиях процессоров - блоках фиксированного размера, в который надо укладывать данные так, чтобы они в эту линию попадали полностью. Это влияет на скорость получения информации и полноту её хранения в кэше.

Новая структура позволяет значительно увеличить допустимую плотность хранения элементов мапы. Коэффициент заполнения может достигать примерно 87,5% без заметного ухудшения производительности.

Для сохранения латентности операций в Swiss Table используется сегментированный рост. Внутренне map состоит из набора независимых подтаблиц, каждая из которых имеет ограниченный размер — порядка тысячи элементов. При увеличении мапы перераспределение затрагивает только одну подтаблицу, а не всю структуру целиком. Такой подход сохраняет инкрементальный характер расширения и гарантирует, что время выполнения операций вставки и обновления остаётся ограниченным даже для очень больших мап.

### Последнее о мапах

Как в случае со слайсами и строками, мапы имеют собственный пакет, который предоставляет возможности для работы с этим типом данных. Называется он — неожиданно — `maps` ([ссылка на GitHub](https://github.com/golang/go/tree/master/src/maps)). Советуем изучить его, чтобы быть во всеоружии.


## Указатели

В мире программирования существуют две похожие с первого взгляда друг на друга вещи: указатели и ссылки. Кажется, что это синонимы. Но указатель — это переменная, которая хранит адрес в памяти другой переменной. А ссылка — это псевдоним/алиас существующего объекта, который нельзя инициализировать, как `null`/`nil`, в отличие от указателя. И вот ссылок в Go **не существует**.

**Зачем же нужны указатели**

- Эффективность: избежание копирования больших структур данных.
- Изменение значений: возможность изменять переменные внутри функций.
- Работа с `nil`: представление отсутствующего значения.
- Разделяемое состояние: несколько переменных могут ссылаться на одни и те же данные.

Синтаксис указателей очень прост. Он состоит из двух операторов:

- `&` — оператор взятия адреса (когда мы хотим работать именно с адресом в памяти, по которому лежит какое-то значение);
- `*` — оператор разыменования (когда мы хотим получить значение, хранящееся по адресу).

```go
package main

import "fmt"

func main() {
    x := 42
    p := &x  // p - указатель на переменную x, хранит адрес x
    
    fmt.Println("Значение x:", x)      // 42
    fmt.Println("Адрес x:", &x)        // 0xc000012078 (пример)
    fmt.Println("Значение p:", p)      // 0xc000012078 (тот же адрес)
    fmt.Println("Значение *p:", *p)    // 42 (разыменование)
    
    *p = 100  // Изменяем значение через указатель
    fmt.Println("Новое значение x:", x) // 100

    var pi *int       // Указатель на int
    var ps *string    // Указатель на string
    var pb *bool      // Указатель на bool

    fmt.Println(pi == nil) // true
}

```

Нулевое значение, значение по умолчанию у указателя — это `nil`.

### Как ещё создать указатель

В Golang присутствует специальная функция `new`, которая выделяет память для значения типа `T`, инициализирует его нулевым значением и возвращает указатель `*T`:

```go
p := new(int)
fmt.Println(*p)  // 0

*p = 42
fmt.Println(*p)  // 42

```

`new` работает с любым типом. Но не путай её с `make`, которая с указателями не работает совсем и используется лишь для трех типов: слайсов, мап и третьего, пока нами не изученного.

## Неявные указатели

В Go некоторые типы данных содержат указатели внутри себя, даже если это не очевидно из синтаксиса. Эти типы часто называют ссылочными типами (reference types), хотя это неформальное название. Забавно, что такие типы не называют указательными.

Прежде чем мы продолжим, стоит сказать, что в мире программирования есть две концепции работы с аргументами функций. Мы можем передавать их в функцию по указателю или по значению. Если в языке аргументы передаются по указателю, то это означает, что передаются не сами данные, а их адрес. Поэтому если переменную изменить внутри функции, то мы увидим это изменение и вне её. А если мы говорим о передаче данных по значению, то это означает, что аргумент копируется в функцию, являясь самодостаточной сущностью, а потому если его вдруг потребуется изменить внутри функции, то, чтобы это изменение увидеть, нам потребуется явно вернуть переменную из функции. Звучит пока не очень просто, поэтому предлагаем перейти к примерам.

В Golang есть несколько ссылочных типов, и два из них тебе уже знакомы: это слайс и мапа. Внутри себя они содержат указатель. Массив при этом не ссылочный тип данных!

```go
func main() {
    m := map[int]int{3: 4}
    sl := make([]int, 1, 2)
    ms := [2]int{1, 2}

    addToMap(m)
    fmt.Println(m) // map[1:2 3:4] - изменения видны вне функции

    addToSlice(sl)
    fmt.Println(sl) // [1] - изменения видны вне функции

    addToMassive(ms)
    fmt.Println(ms) // [1 2] - измнения не видны вне функции
}

func addToMap(m map[int]int) {
    m[1] = 2
}

func addToSlice(m []int) {
    m[0] = 1
}

func addToMassive(m [2]int) {
    m[1] = 3
}
```

Поэтому соблюдай предельную осторожность, когда передаёшь в функцию слайс или мапу. Правилом хорошего тона считается возвращать явно из функции слайс или мапу, если переменная этого типа изменяется. Потому что со слайсом зачастую может случиться такая оказия:

```go
func main() {
    sl := make([]int, 1, 2)

    addToSlice(sl)
    fmt.Println(sl) // [1] - изменения не видны вне функции - append перезаписал переменную
}

func addToSlice(m []int) {
    m[0] = 1
    m = append(m, 2)
    fmt.Println(m) // [1 2] - все правильно, слайс расширен
}
```

Все потому, что `append` перезаписывает переменную, область видимости которой из-за её передачи по значению теперь ограничивается телом функции. В таком случае необходимо явно вернуть слайс через `return`.
