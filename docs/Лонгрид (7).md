# Работа с файлами, i/o-операции

Работа с файлами в любом языке — один из важных навыков программирования. Наравне с сетевым взаимодействием (тема следующего модуля) передача данных в файлах также довольно активно используется. Это могут быть и классические текстовые файлы, и csv, excel-файлы, pdf при передаче данных клиенту на фронтенд сайта или в отдельную админку. Не забывай про изображения, которые порой необходимо сжать или изменить их разрешение. Вариантов взаимодействия много. Именно поэтому работать с этим так необходимо. 

В этом лонгриде мы сфокусируемся на работе с текстовыми файлами, так как, поняв суть работы с ними, ты сможешь самостоятельно начать работать с другими типами файлов.

## Работа с файлами

Очевидно, что при работе с файлами мы можем использовать следующие основные операции:

- открытие файлов;
- чтение файла;
- запись в файл;
- создание файлов;
- проверка их существования.

Начнём по порядку, но сначала познакомимся с пакетом `os`, который предлагает функционал для работы с файлами. И весь этот функционал работает с [представлением файлов](https://github.com/golang/go/blob/go1.25.6/src/os/types.go#L18) в контексте исходного кода языка:

```go
type File struct {
	*file // os specific
}
```

Здесь структура `File` содержит неэкспортируемое представление системной информации о файле, позволяя тем самым оборачивать его, предоставляя функционал через методы экспортируемой структуры, имеющие доступ к низкоуровневой информации. Именно с этим нам и предстоит работать далее.

**Важное замечание**: как говорится в документации к `File`, все его методы являются [конкурентно безопасными](https://github.com/golang/go/blob/go1.25.6/src/os/file.go#L37).

Начнём с операции открытия файла. Для этого в библиотеке имеются методы `os.Open()` и `os.OpenFile()`, которые позволяют вытащить дескриптор файла, но по-разному, хотя и возвращают одно и то же. 

Первый метод позволяет открыть файл только для чтения, и если такого файла не существует в директории, то вернётся ошибка типа `type PathError = fs.PathError`, которую можно проверить в твоем коде через `errors.As()`. Данный метод открывает файл **только** для его чтения. Это видно в исходном коде, так как под капотом `os.Open()` [вызывает](https://github.com/golang/go/blob/go1.25.6/src/os/file.go#L389) функцию `os.OpenFile()` с флагом `O_RDONLY`, который и говорит системе о том, что права для этого дескриптора исключительно читающие. 

Вот так это выглядит в коде:

```go
file, err := os.Open("input.txt")
if err != nil {
	return err
}

defer file.Close()
```

Получив `file`, можно работать с ним для дальнейших операций. Заметь, что файл закрывается методом `Close` внутри `defer` — если не закрыть файл, то мы получим утечку ресурсов во время дальнейшей работы программы. Как ты помнишь, `defer` позволяет обработать закрытие функции в любом случае — даже во время паники. Таким образом сбрасываются все внутренние буферы, задействованные при работе с файлом. 

Однако правильнее вызывать закрытие всё же с обработкой ошибки, хотя этим зачастую пренебрегают:

```go
defer func() {
	if err := file.Close(); err != nil {
		fmt.Println(err)
	}
}()
```

А вот с помощью `os.OpenFile()` можно открыть файл и для чтения, и для записи, и создавая его перед использованием. Для этого надо использовать [определённые флаги](https://github.com/golang/go/blob/go1.25.6/src/os/file.go#L79) вторым аргументом, а также уровни доступа к этому файлу:

```go
file, err := os.OpenFile("notes.txt", os.O_RDWR|os.O_CREATE, 0644)
if err != nil {
	log.Fatal(err)
}

if err := file.Close(); err != nil {
	log.Fatal(err)
}
```

Дальнейшее чтение с помощью переменной `file` можно осуществлять через метод `Read`, который позволяет читать конкретное количество байтов в переданный в качестве аргумента слайс (не более `len(b)` этого слайса, чтобы возможный `append` не перезаписал слайс в памяти). В случае если файл заканчивается, тут и в других методах будет получена ошибка `io.EOF`, которая говорит о том, что файл закончен. Имеет смысл всегда её обрабатывать:

```go
b := make([]byte, 1024)

for {
	n, err := file.Read(b) // Возвращает количество прочитанных байтов (можно опустить) и ошибку
	if err != nil {
		if errors.Is(err, io.EOF) {
			return nil
		}

		return fmt.Errorf("Warning: %w", err)
	}
}
```

Но чтение файла, как и его создание, можно реализовать с помощью отдельных функций пакета. Для чтения у нас в арсенале имеется `os.ReadFile`, который читает весь файл сразу в память, возвращая слайс байтов:

```go
data, err := os.ReadFile("input.txt")
if err != nil {
	log.Fatal(err)
}

fmt.Print(string(data))
```

Это бывает удобно, но не с большими файлами, так как вызов функции блокирует выполнение до полного чтения файла, занимая большой объём памяти. Поэтому в случаях работы с файлами уже в сотни мегабайтов лучше подумать о других подходах, которые мы затронем позже.

Создать нужный файл можно не только с помощью `os.OpenFile()` с нужным флагом, но и с целевой функцией `os.Create()`:

```go
// Открытие дескриптора файла
    file, err := os.Create("large_file.txt")
    if err != nil {
        fmt.Printf("Ошибка открытия файла: %v\n", err)
        return
    }
    defer file.Close()

    var buffer = make([]byte, 4096) // Буфер размером 4 КБ
    for {
        // Чтение следующих 4 КБ файла
        n, err := file.Read(buffer)
        if err == io.EOF {
            break
        }
        // Обработка данных из буфера
    }
```

А в случае если тебе не надо создавать файл и ты хочешь понять, лежит ли интересующая тебя сущность в директории, можно воспользоваться функцией `os.Stat` с проверкой ошибки `fs.ErrNotExist`:

```go
func fileExists(filename string) (bool, error) {
    _, err := os.Stat(filename)
    if err == nil {
        return true, nil
    }

    if errors.Is(err, fs.ErrNotExist) {
        return false, nil
    }

    return false, err
}
```

Для того чтобы писать в файл, у тебя снова есть несколько возможностей. Это и функция `os.WriteFile()`, и методы у дескриптора `file.Write()` и `file.WriteString()`. Функция `os.WriteFile()` наподобие остальных функций пакета `os` позволяет загрузить в файл конкретные данные с возможностью создать этот файл, если он не существует:

```go
err := os.WriteFile("testdata/hello", []byte("Hello, Gophers!"), 0666)
if err != nil {
	log.Fatal(err)
}
```

Методы дескриптора дают возможность записать в файл конкретное количество байтов (`file.Write()`) или конкретную строку (`file.WriteString()`).

Возможность удаления файла мы предоставляем тебе найти самостоятельно.

## Пакет io

Здесь стоит кратко остановиться и обсудить работу с вводом-выводом через интерфейсы пакета `io` (input/output). Этот пакет содержит не только рабочие функции, которые ты можешь использовать при взаимодействии с файлами. Эти самые функции также используют то, на чём нам стоит сфокусировать внимание, когда мы говорим про `io` — интерфейсы.

Ты уже знаешь, что такое интерфейсы, для чего они и как с ними можно работать. Но на самом деле `io` — максимально понятный пример того, как можно и нужно использовать интерфейсы в библиотеках и самом языке. Если [посмотреть](https://github.com/golang/go/blob/go1.25.6/src/io/io.go) в сам пакет, то в глаза бросятся именно интерфейсы, которые по тому самому канону, заложенному в язык (помнишь, что чем меньше интерфейс, тем сильнее абстракция?), дают возможность работать с одним поведением во всём нашем коде. 

К примеру, стоит обратить внимание на следующие интерфейсы пакета и запомнить их:

- `io.Reader` — описывает типы, читающие данные методом `Read(p []byte) (int, error)`. А мы ведь его видели в пакете `os`!
- `io.Writer` — описывает типы, записывающие данные методом `Write(p []byte) (int, error)`.
- `io.Closer` — описывает типы, освобождающие ресурсы методом `Close() error`.
- `io.ReadWriter` — объединяет интерфейсы `io.Reader` и `io.Writer`.
- `io.ReadCloser`, `WriteCloser` — объединяют интерфейсы `io.Reader/io.Writer` и `io.Closer` и так далее.

Получается, что мы, как разработчики, можем заимствовать эти интерфейсы как есть, затягивая в свои файлы только одну зависимость в виде пакета `io`, а что именно его реализует — уже неважно. Даже сами функции пакета используют его собственные интерфейсы как параметры, от которых необходимо лишь поведение, а не информация о том, с чем именно работает пакет. Это отличный пример сильных абстракций и универсального кода, который позволяет переиспользовать поведение.

Особое внимание всё-таки стоит уделить функции `io.Copy()`, которая копирует данные из `Reader` в `Writer`, не спрашивая, а что именно за источник и приёмник к ней пришли. Ведь контракты уже в интерфейсе описаны, а строгая типизация позволяет с точностью быть уверенными в том, что подобное поведение сработает в проде.

## Работа с большими файлами

Как уже было сказано при описании метода `os.ReadFile()`, бывает неудобно и неправильно читать весь файл в память сразу из-за его объёма. Да, можно использовать метод `file.Read()`, но в таком случае велика вероятность через ручной буфер в виде слайса вычитывать данные некрасиво (например, останавливаясь в середине слова). Для более корректной работы с большими объёмами данных стоит использовать уже готовые решения из пакета `bufio` (buffer input/output).

Ты можешь через этот пакет делать всё: читать, писать, перезаписывать данные. Такая сущность, как `bufio.Scanner`, представляет из себя высокоуровневый инструмент для последовательного чтения данных. Чаще всего текстовых:

```go
file, err := os.Open("log.txt")
if err != nil {
	log.Fatal(err)
}

defer file.Close()

scanner := bufio.NewScanner(file)
for scanner.Scan() {
	fmt.Println(scanner.Text())
}

if err := scanner.Err(); err != nil {
	log.Fatalf("Ошибка чтения: %v", err)
}
```

Здесь мы получаем дескриптор файла для чтения и передаём его в конструктор. Конструктору всё равно, что в него передают, — главное, чтобы параметр соответствовал интерфейсу `io.Reader` (а `file` имеет метод данного интерфейса). И далее `scanner` сам возвращает нам построчный вывод, разделяя поток данных по нужным токенам (например, по `\n`). 

Это отличный инструмент для чтения данных, но и он имеет ограничения. Например, его буфер ограничен 64 Кб и предназначен для текстовых, а не бинарных блоков. Он также не даёт полного контроля над чтением. Да, мы можем перед `scanner.Scan()` вызвать метод `scanner.Split(bufio.ScanWords)`, заменяя разбиение строки на пословное с построчной. Или читать каждую руну отдельно.

Однако в случаях длинных строк, бинарных данных и желания управлять чтением серьезнее, лучше посмотреть в сторону `bufio.NewReader(io.Reader)` (предлагаем тебе самостоятельно изучить эту сущность и её возможности).

Логично, что если мы можем получить доступ последовательно, то также можем считывать данные из произвольного места. Именно для этого нам нужен интерфейс `io.Seeker`, который помогает вычитывать данные с конкретного места в файле. 

Например, через тот же дескриптор мы можем вызвать:

```go
file.Seek(0, io.SeekStart)
```

Подобная запись говорит нам о том, что мы хотим вычитать файл с нулевого байта, отсчитывая с начала этого файла. Константа `io.SeekStart` как раз задаёт точку отсчёта. Но в пакете [есть](https://github.com/golang/go/blob/go1.25.6/src/io/io.go#L21) пара других констант.

## Обход файловой системы

Работая с файлами и данными на диске, нельзя обойти стороной инструменты работы с твоей файловой системой: с папками, файлами, бинарями. 

Работа с системой — это одна из базовых задач программиста. В Go для этого используются всё тот же `os` и `path/filepath` для работы с путями и обходом директорий. Мы дадим несколько базовых функций из этих пакетов, **остальные тебе нужно изучить самостоятельно**.

### Создание директорий

Как создать файл, ты уже знаешь. Создание директорий происходит через `os.Mkdir()`, когда надо создать одну директорию в конкретном месте, или через `os.MkdirAll()`, которая создаёт всю цепочку директорий, если её не существует:

```go
err := os.MkdirAll("data/logs", 0755)
if err != nil {
	log.Fatal(err)
}
```

### Чтение содержимого директорий

Порой требуется понять, что именно лежит в конкретном каталоге. Для его просмотра используется `os.ReadDir()`, а также вспомогательные методы сущностей, которые функция возвращает. Например, часто необходима проверка того, является ли сущность директорией или какое имя/размер соответствует ей:

```go
entries, err := os.ReadDir("./data")
if err != nil {
	log.Fatal(err)
}

for _, entry := range entries {
if entry.IsDir() {
  	fmt.Println("Каталог:", entry.Name())
  } else {
  	fmt.Println("Файл:", entry.Name())
  }
}
```

### Рекурсивный обход

Да, ты можешь обходить систему вручную, но в Go уже есть сущность, которая делает рекурсивный обход автоматически. Ты можешь использовать для этих целей `filepath.Walk`, но предпочтительнее пользоваться `filepath.WalkDir`, который работает с `DirEntry` (сущностью директории) вместо `FileInfo`, не открывает файлы без необходимости, избегает лишних системных вызовов, за счёт чего более эффективен.

```go
filepath.WalkDir(".", func (path string, d fs.DirEntry, err error) error {
  if err != nil {
    return err
  }

  if !d.IsDir() {
    fmt.Println(path)
  }

  return nil
})
```

`WalkDir` принимает точку входа для обхода и функцию, которая называется *обработчиком обхода*. Это функция, которая вызывается для каждого элемента файловой системы (каталога или файла) во время рекурсивного обхода, и именно в ней размещается вся пользовательская логика обработки всего процесса обхода.

### Аргументы и флаги

При запуске программ часто требуется настраивать их поведение. Например, передать имя файла или указать порт прослушивания. Для этого используются аргументы командной строки. Go предоставляет несколько способов для работы с ними.

Самый простой способ получить аргументы командной строки — это использовать переменную `os.Args`. Это глобальная [переменная](https://github.com/golang/go/blob/go1.25.6/src/os/proc.go#L16) типа слайса строк, которая при запуске программы считывает все аргументы командной строки запуска в [функции](https://github.com/golang/go/blob/go1.25.6/src/os/proc.go#L18) `init()` (то есть при самом запуске), начиная с названия программы (пути к ней) в первом индексе.

```go
package main

import (
    "fmt"
    "os"
)

func main() {
    fmt.Println("Количество аргументов:", len(os.Args)-1)
    fmt.Println("Название программы:", os.Args[0])
    
    if len(os.Args) > 1 {
        fmt.Println("Аргументы:")
        for i, arg := range os.Args[1:] {
            fmt.Printf("  %d: %s\n", i+1, arg)
        }
    }
}
```

```bash
go run main.go hello world 123
```

```
Количество аргументов: 3
Название программы: /tmp/go-build123456/b001/main
Аргументы:
  1: hello
  2: world
  3: 123
```

**Что здесь важно:** заметь, что в примере выше мы посчитали количество аргументов как `len(os.Args)-1`. `os.Args[0]` содержит имя программы таким, каким его передал родительский процесс. Обычно это путь к исполняемому файлу (при `go run` путь к временному бинарнику во временной директории, потому что эта команда под капотом компилирует, запускает и удаляет бинарник). Этот элемент не является пользовательским и присутствует всегда, даже когда никакие аргументы не переданы. Пользовательские аргументы начинаются с `os.Args[1]`.

Но зачастую нам требуется более сложная обработка входных параметров в программу. И здесь нам помогает пакет `flag`. Он предоставляет удобный способ определения именованных параметров (флагов) с типизацией, значениями по умолчанию и автоматической генерацией справки. Пакет `flag` автоматически обрабатывает различные форматы передачи флагов (-name=value, -name value, --name=value, --name value). Булевы флаги можно указывать без значения (-verbose то же самое что и -verbose=true).

```go
package main

import (
    "flag"
    "fmt"
)

func main() {
    // Определение флагов
    var (
        name    = flag.String("name", "Мир", "имя для приветствия")
        count   = flag.Int("count", 1, "количество повторений")
        verbose = flag.Bool("verbose", false, "подробный вывод")
    )
    
    // Парсинг аргументов (должен быть вызван после объявления всех флагов)
    flag.Parse()
    
    if *verbose {
        fmt.Printf("Выполняется приветствие с параметрами: name=%s, count=%d\n", *name, *count)
    }
    
    n := *count
    for range n {
        fmt.Printf("Привет, %s!\n", *name)
    }
}
```

```bash
go run main.go -name="Алиса" -count=3 -verbose
```

```
Выполняется приветствие с параметрами: name=Алиса, count=3
Привет, Алиса!
Привет, Алиса!
Привет, Алиса!
```

**Но и здесь присутствуют важные моменты, о которых надо помнить.** В пакете `flag` нет прямой функции чтобы узнать, был ли указан флаг пользователем, поэтому для флагов требуется всегда указывать значение по умолчанию. При стиле определения флагов выше переменные являются указателями, поэтому их придётся дополнительно разыменовывать. Эти переменные всегда инициализируются пакетом `flag`, поэтому дополнительных проверок на `nil` не потребуется по причине предыдущего пункта. 

Если тебе знакома командная строка на MacOS или Linux, стоит обратить внимание на то, что пакет `flag` не придерживается стиля GNU при парсинге флагов: -flag и --flag являются идентичными. В силу этого в экосистеме Go принято придерживаться стиля авторов и использовать флаги с одним дефисом (это применимо только к программам, использующим пакет `flag`).

А ещё мы можем определять неименованные аргументы после флагов. Они используются, когда программе нужно обработать список элементов (файлов, URL, имен пользователей и так далее), при этом сохраняя возможность настройки поведения через флаги. Это распространённый паттерн в command-line-утилитах Unix/Linux (например, ls -l file1 file2, grep -n "pattern" file1.txt file2.txt file3.txt):

```go
package main

import (
    "flag"
    "fmt"
)

func main() {
    var prefix = flag.String("prefix", "", "префикс для файлов")
    flag.Parse()
    
    files := flag.Args() // Получаем неименованные аргументы
    
    if len(files) == 0 {
        fmt.Println("Не указаны файлы для обработки")
        return
    }
    
    fmt.Printf("Обрабатываем %d файлов:\n", len(files))
    for _, file := range files {
        fmt.Printf("- %s%s\n", *prefix, file)
    }
}
```
