# Дженерики. Ошибки и их обработка. Паники и правила работы с ними


## Обобщение кода через дженерики

*Дженерики (generics)* — это способ сделать код более универсальным, чем он есть, через написание функций и типов, работающих с разными типа данных, сохраняя типобезопасность. Дженерики появились относительно недавно (в версии go1.18 от марта 2022 года), хотя сообщество просило о них продолжительное время. Почему? Посмотрим на примере, какую проблему решили дженерики:

```go
// Функция для поиска максимума int
func MaxInt(a, b int) int {
    if a > b {
        return a
    }
    return b
}

// Функция для поиска максимума float64
func MaxFloat64(a, b float64) float64 {
    if a > b {
        return a
    }
    return b
}

// Функция для поиска максимума string
func MaxString(a, b string) string {
    if a > b {
        return a
    }
    return b
}

// Дублирование кода!
```

Раньше нам приходилось писать одну и ту же функцию для разных типов данных и страдать от количества строк кода, которые делали одно и то же. Либо пользоваться [кодогенерацией](https://go.dev/blog/generate) - первоначально довольно сложной концепцией, которую в курсе мы затрагивать практически не будем, однако для общего представления с темой желательно ознакомиться самостоятельно. Но с дженериками проблемы просто пропали. Теперь эти три функции можно объединить в одну:

```go
// Одна функция для всех сравнимых типов!
func Max[T cmp.Ordered](a, b T) T {
    if a > b {
        return a
    }
    return b
}

func main() {
    fmt.Println(Max(10, 20))           // 20
    fmt.Println(Max(3.14, 2.71))       // 3.14
    fmt.Println(Max("hello", "world")) // world
}
```

То есть функция `Max` превратилась в обобщённую функцию, которая может работать с более чем одним типом. Описывается такая функция следующим синтаксисом:
- `[T cmp.Ordered]` — обобщённый параметр с наложенным ограничением на тип (чуть дальше поговорим об этом подробнее);
- `T` — имя параметра типа (может быть любой буквой, но обычно используются T, K, V);
- `cmp.Ordered` — ограничение на тип.

Иными словами, мы описываем для функции некий абстрактный тип, который так же имеет конкретные ограничения, но в рамках этого типа в нашем случае функция может принимать две переменные `a` и `b` одного и того же типа `T`, который, в свою очередь, должен отвечать требованиям ограничений. Кстати, таких обобщённых параметров у функции может быть больше одного:

```go
// Функция с двумя параметрами типов
func Map[T, U any](slice []T, fn func(T) U) []U {
    result := make([]U, len(slice))
    for i, v := range slice {
        result[i] = fn(v)
    }
    return result
}
```

Как видно, объявляя обобщённые параметры, мы вольны использовать их в сигнатуре по своему усмотрению.

Вот только всё имеет свою цену. Это в коде мы видим единую запись функции, которая подразумевает использование её с различными типами. А вот при компиляции кода в бинарном файле произойдет не очевидная, но в целом логичная штука: сгенерируется ровно столько функций, сколько типов функция-дженерик может использовать (чуть ниже расскажем про ограничение количества типов). И это приводит к закономерному результату - бинарный файл растёт в весе. Поэтому этот момент тоже стоит учитывать.

Но не функциями едиными. Ты можешь создавать ещё и обобщённые структуры:

```go
package main

import "fmt"

// Обобщённая структура для пары значений
type Pair[K, V any] struct {
    Key   K
    Value V
}

func main() {
    p1 := Pair[string, int]{Key: "age", Value: 30}
    p2 := Pair[int, string]{Key: 1, Value: "first"}
    
    fmt.Println(p1) // {age 30}
    fmt.Println(p2) // {1 first}
}
```

Подобное поведение открывает возможности к созданию обобщённых кешей, стеков и прочего.


### Ограничения параметров

Как уже было сказано выше, мы можем вручную ограничить обобщённые параметры. Это означает, что мы можем указывать, какие именно типы может представлять параметр в функции.
В Go, как обычно, подобные ограничения уже имеются у нас на руках из стандартной библиотеки:

- `any` — подобное ограничение используется в примере с функцией `Map` в предыдущем пункте лонгрида. `Any` означает, что для типа не существует никаких ограничений. Из лонгрида про интерфейсы ты помнишь, что `any` — это алиас на пустой интерфейс, а значит, при выборе этого ограничения мы никак не ограничиваем параметр в использовании типа.
- `comparable` — ограничение, которое позволяет работать только со сравниваемыми типами (которые работают с операторами `==` и `!=`).
- `cmp.Ordered` — даёт возможность работать только с упорядоченными типами (которые работают со знаками неравенства `>`, `<`, `>=`, `<=`).

Но не только встроенными ограничениями мы пользуемся. Golang позволяет нам создавать пользовательские ограничения для обобщённых функций и типов. И, неожиданно, ограничения дженериков работают на интерфейсах:

```go
// Ограничение: тип должен иметь метод String()
type Stringer interface {
    String() string
}

func PrintAll[T Stringer](items []T) {
    for _, item := range items {
        fmt.Println(item.String())
    }
}
```

Теперь любой тип, реализующий интерфейс, обобщается в параметре `T`.

Ещё в интерфейсе ты можешь указать конкретный набор типов, которые бы был готов видеть в обобщённом параметре. Например:

```go
// Ограничение: только int или float64
type Number interface {
    int | float64
}

func Sum[T Number](numbers []T) T {
    var sum T
    for _, n := range numbers {
        sum += n
    }
    return sum
}
```

Оператор `|` по сути является тем самым `И`, которое помогает компилятору понять, что перечень типов в интерфейсе является ограничением для дженерика.

Помнишь, мы говорили о том, что можно создавать кастомные типы на основе встроенных типов в Go? Что делать, если захотелось использовать в дженерике и встроенный тип, и кастомный тип, который создан на основе встроенного типа? Перечислять оба? А если таких типов создано множество? На помощь приходит оператор `~`, который является «приближённым ограничением» (approximation), означающим, что в дженерике может использоваться конкретный тип и все типы, для которых он является базовым:

```go
package main

import "fmt"

// Разрешает int и пользовательские типы на основе int
type Integer interface {
    ~int
}

type MyInt int

func Double[T Integer](value T) T {
    return value * 2
}

func main() {
    var x int = 5
    var y MyInt = 10
    
    fmt.Println(Double(x)) // 10
    fmt.Println(Double(y)) // 20
}
```

И это отлично работает! В целом работа с ограничениями дженериков достаточно гибкий механизм. Даже `cmp.Ordered` — это [набор](https://github.com/golang/go/blob/go1.25.6/src/cmp/cmp.go#L18) базовых типов:

```go
type Ordered interface {
	~int | ~int8 | ~int16 | ~int32 | ~int64 |
		~uint | ~uint8 | ~uint16 | ~uint32 | ~uint64 | ~uintptr |
		~float32 | ~float64 |
		~string
}
```


### Где дженериков пока нет

К сожалению, методы не могут содержать собственные обобщённые параметры. Зато могут содержать обобщённые параметры своих ресиверов:

```go
type Container[T any] struct {
    value T
}

// ✅ Метод использует T из структуры
func (c Container[T]) Get() T {
    return c.value
}

func (c *Container[T]) Set(value T) {
    c.value = value
}

// ❌ ОШИБКА: метод не может иметь свои параметры типов
// func (c Container[T]) Convert[U any](fn func(T) U) U {
//     return fn(c.value)
// }
```

На самом деле это проблема для разработчика. Иногда так хочется использовать метод с обобщёнными собственными типами... На закономерный вопрос «А почему бы всё-таки не позволить методам иметь обобщённые методы?» разработчики языка недвусмысленно заявляют:

> In Go, one of the main roles of methods is to permit types to implement interfaces. It is not clear whether it is reasonably possible to permit parameterized methods to implement interfaces.

Подробнее написано в [FAQ](https://go.dev/doc/faq#generic_methods) по языку отдельным пунктом и в [объяснении](https://go.googlesource.com/proposal/+/master/design/43651-type-parameters.md#no-parameterized-methods) дизайна языка. Однако, никогда не говори «никогда». Авторы в объяснении дизайна говорят о том, что подобную концепцию всё-таки можно пересмотреть. В том числе поэтому на горизонте замаячил proposal (предложение — механизм внесения изменений на обсуждение любым членом сообщества (даже тобой) в GitHub) от Роберта Гризмера (одного из отцов-основателей языка), в [котором](https://github.com/golang/go/issues/77273) он приводит техническое описание и обоснование возможности использования дженериков у типов. Поэтому, быть может, через год-два мы увидим в Go изменения в политике обобщённых методов.


### Использование дженериков

Дженерики лучше использовать в следующих случаях:

1. Для сложных структур данных — списков, стеков, деревьев, графов.
2. Для утилитных функций — Map, Filter, Reduce.
3. Для обобщённых алгоритмов — сортировка, поиск.
4. Когда избегаешь дублирования кода для разных типов.

А не использовать в таких ситуациях:

5. Когда достаточно интерфейсов: в случае если нужно поведение, а не тип.
6. Для простых функций, если тип известен.

В целом вырисовывается перечень правил работы с дженериками:

- Начинай с конкретных типов, обобщай позже.
- Используй понятные имена: T, K, V для простых случаев.
- Ограничивай типы.
- Предпочитай интерфейсы, когда нужно поведение.
- Не злоупотребляй, потому что простота важнее универсальности.


## Ошибки

Теперь переходим к одной из самых холиварных тем в Golang — к **ошибкам и их обработке**. В Go не существует *исключений*, как в других языках. Вместо них мы используем именно *ошибки*, которые возникают в месте их создания и затем обычно на самый верх до пользователя системы либо до места логирования ошибки. Это возможно благодаря простой концепции, которая лежит в основе языка: **errors as values**. Согласно ей, ошибки — это не просто условия для проверки, а **значения**, которые имеют свой тип, с которыми можно работать так же, как и с любыми другими данными. Возможно, тебе уже где-то встречался такой код, хотя пока мы его немного избегали:

```go
_, err := fmt.Scan(&x)
if err != nil {
	fmt.Println(err)
}
```

На семинарах мы использовали `fmt.Scan`, но вот такой блок кода с проверкой на ошибку — почти нет. А именно эта нотация, эта обработка ошибки после выполнения функции одновременно одна из самых важных вещей в языке и самых обсуждаемых на стороне тех, кто приходит в Golang из других языков. Самое важное, что мы должны всегда помнить: **обрабатывать ошибки нужно всегда**. Не пренебрегай этим: вовремя пойманная ошибка — залог успешной работы твоей программы.

Однако представь, сколько однотипного кода пишется go-разработчиками, когда они пытаются обработать ошибки во всех своих функциях и методах... И это то, что пугает новичков в языке. Возникает вопрос: можно ли как-то обойти подобное количество шаблонного кода? Ответ: почти нет. Почему ответ не твердое **нет**, объясним чуть позже. А пока стоит сказать, что для языка открывалось множество proposals, в которых комьюнити предлагало различные варианты избавления от пресловутого `err != nil`: [пример 1 на GitHub](https://github.com/golang/go/issues/32825), [пример 2 на GitHub](https://github.com/golang/go/issues/71528), [пример 3 на GitHub](https://github.com/golang/go/issues/71808). **Только вот** разработчики языка выпустили [пост](https://go.dev/blog/error-syntax) в своём блоге о том, чтобы никто больше не предлагал изменить существующий подход обработки ошибок: команда не изменит своего мнения о том, что `err != nil` — лучший подход для работы с ошибками в Go. Так что мы можем свыкнуться с этой мыслью и идти дальше.

Итак, что такое ошибка в Go? Это значение. Но ещё это **интерфейс**!

```go
type error interface {
    Error() string
}
```

То есть всякий тип, который удовлетворяет этому [интерфейсу](https://github.com/golang/go/blob/go1.25.6/src/builtin/builtin.go#L308), может быть ошибкой. Поэтому мы можем в целом говорить о том, что `error` в Go — это встроенный тип. А значит с ошибками можно работать по-разному:

1. Во-первых, мы можем добавлять ошибке контекст. Согласись, как вообще можно понять, что произошло в недрах программы, если везде возвращать ошибку так:

```go
func one() error {
	...
	if err != nil {
		return ошибка
	}
}

func two() error {
	err := one()
	if err != nil {
		return err // Поднимаем ошибку наверх, не понимая контекста
	}
}

func three() error {
	err := one()
	if err != nil {
		return err // Поднимаем ошибку наверх, не понимая контекста в другой функции
	}
}

func four() errror {
	err := two()
	if err != nil {
		return err // Поднимаем ошибку наверх, не понимая контекста уже два уровня выше
	}
}
```

Поэтому можно и нужно использовать возможность улучшать нашу ошибку, чтобы легче понимать причину возникшей проблемы. Ты можешь включить такие дополнительные параметры, как:
- параметры функции;
- значения переменных;
- дополнительные сообщения об ошибках.

Одним из способов обогатить ошибку является использование пакета `fmt`. Функция `Errorf` добавляет форматированное описание функции:

```go
return fmt.Errorf("ошибка при обработке данных пользователя %s: %w", userID, err)
```

Тут мы уже существующей ошибке `err` добавляем следующий уровень контекста в виде текста и некоего ID пользователя через использование `%w` — этот глагол форматирования позволяет в дальнейшем извлечь исходное сообщение об ошибке. То есть `fmt.Errorf` позволяет обернуть ошибку более понятным описанием и отправить её выше по стеку вызова.

Ещё можно использовать сторонние пакеты, например, `https://github.com/cockroachdb/errors`, который имеет метод `Wrap`, позволяющий делать, по сути, то же самое, что и `fmt.Errorf`.

2. Раз мы можем оборачивать ошибки, значит, мы можем и раскрывать их. Для этого используется пакет стандартной библиотеки `errors`. А именно метод `errors.Is`:

```go
if err != nil && errors.Is(err, `какая-то ошибка`){
	fmt.Println("тут ошибка")
}
```

Метод позволяет отыскать в цепочке ошибок ту самую нужную. Также можно использовать метод `errors.Unwrap`, чтобы извлекать оригинальную ошибку.

3. Раз уж заговорили об этом: что такое оригинальная ошибка? Все ошибки с чего-то начинаются, и ты можешь их создавать. Делается это просто, через тот же пакет `errors`:

```go
var ErrInvalidArgument = errors.New("invalid argument")
```

Всё, теперь эту ошибку можно возвращать в случае непрохождения проверки валидации какого-то параметра в функции:

```go
func checkArgument(a int) error {
	if a == 0 {
		return ErrInvalidArgument
	}

	return nil
}
```

Важно тут ещё и то, что подобные sentinel-ошибки по соглашению должны начинаться с префикса `Err...` или `err...` при объявлении таких переменных в виде ошибок.

4. Раз наши ошибки могут быть любым типом, которому только и нужно, что удовлетворять интерфейсу `Error`, ты можешь проверять существование конкретного типа у ошибки через метод `erros.As`:

```go
type MyError struct {
	Code int
	Msg  string
}

func (e *MyError) Error() string {
	return fmt.Sprintf("Ошибка %d: %s", e.Code, e.Msg)
}

func doSomething() error {
	return fmt.Errorf("дело дрянь: %w", &MyError{Code: 500, Msg: "Сервер недоступен"})
}

func main() {
	var targetErr *MyError

	err := doSomething()
	if errors.As(err, &targetErr) {
		fmt.Println("Код ошибки:", targetErr.Code, "Сообщение:", targetErr.Msg)
	} else {
		fmt.Println("Произошла неизвестная ошибка")
	}
}
```

В новой версии языка 1.26 подобный синтаксис упростили:

```go
pathError, ok := errors.AsType[*fs.PathError](err)
```


### Как избавиться от err!=nil

Способ всё-таки есть. Его используют безумно редко, так как он не является идиоматичным. Но да, можно выполнить абстрагирование ошибок:

```go
type errorHandler struct {
    SampleObject sampleObjectType
    Err          error
}

func (handler *errorHandler) Write(param ParamType) {
    if handler.Err != nil {
        return
    }
    _, handler.Err = handler.sampleObject.SomeAction(param)
}
```


## Отложенные вызовы

В Go существует отличное ключевое слово, которое позволяет отложить выполнение функций, пока не завершится выполнение функции, в чьём теле они существуют. И это слово `defer`. Используя это слово, можно значительно упростить себе жизнь. Например, писать о закрытии соединения рядом с его открытием.

В теле любой функции можно вызвать неограниченное количество `defer`. Как же они тогда будут выполняться? По принципу `LIFO` — последним пришёл, первым ушёл. После завершения выполнения функции все `defer` будут выполняться условно снизу вверх, то есть от самого последнего объявления к самому первому. При этом ты можешь работать как с нормальными, объявленными функциями, так и с анонимными функциями:

```go
func Example() {
    for i := 3; i > 0; i-- {
        defer fmt.Print(i, " ") // Используем существующую функцию + выведется ответ, обратный объявлению, то есть `1 2 3`
    }
}

func Example3() {
    for i := 3; i > 0; i-- {
        defer func(n int) {    // Анонимная функция
            fmt.Print(n, " ")
        }(i)
    }
}
```

Использование `defer`, по сути, гарантирует, что в случае возникновения ошибок, мы в любом случае выполним код, который есть у `defer`. То есть даже возникновение ошибок не влияет на то, выполнится ли этот блок кода или нет.


### Паники не ошибки

В Go есть свои аналоги необрабатываемых исключений (unhandled exception) из других языков. И имя этому явлению «паника», которая вызывается через `panic()`. Тебе уже встречалась паника во время написания кода на Go. По сути, это фатальная ошибка в твоём коде, которая делает дальнейшее выполнение программы невозможным/бессмысленным. Когда вызывается паника (вручную или же из рантайма языка), приложение падает, выдавая в стандартный вывод ошибок стек вызова с описанием местоположения паники. Очень удобно!

Но тут возникает сильный соблазн понатыкать паники вообще везде. Однако принцип работы с паникой в Go можно выразить словами [«Dont panic!»](https://go.dev/wiki/CodeReviewComments#dont-panic). Этот принцип прямо нам говорит о том, что паника нужна нам только в самых крайних случаях, когда выполнение программы просто невозможно или происходит что-то непредусмотренное. Паника не исключение из других ЯП!

```go
func foo() {
    panic("Something went wrong!")
}

func main() {
    fmt.Println("Start")
    foo()
    fmt.Println("End") // Эта строка не будет выполнена
}
```

Когда тебе может понадобиться паника:
- Если ошибка фатальна и продолжать работу невозможно
- Если ошибка нарушает логику работы программы и её нельзя исправить на месте.
- Если ошибку невозможно обработать.

Когда не нужно использовать панику:
- Почти всегда, когда ты ожидаешь в конкретном месте конкретную ошибку и знаешь, что она тут может возникнуть и она некритична.
- Если ошибку можно исправить/залогировать/передать дальше.
- Если программа может продолжить работать в случае получения какой-то ошибки.


### Ловец паник

А теперь можно объединить знания о `defer` и `panic`. Да, паники крашат код, но паники можно перехватить, обработать и продолжить выполнение программы как ни в чем не бывало. И это можно сделать с помощью встроенной функции `recover()`. Именно она восстанавливает программу после паники. **Даже после возникновения паники** управление передаётся в ближайший отложенный блок кода с `defer`. И вот он-то и может вызвать `recover` для поимки паники.

Собственно, `recover`  работает, только если она вызывается внутри `defer`. Иначе функция просто возвращает значение, переданное в `panic`, и завершает выполнение программы. Если же `recover` вызывается в коде, где нет никакой активной паники, то просто возвращается `nil` и программа продолжает работу.

```go
package main

import (
	"fmt"
)

func Example() {
	fmt.Println("Inside function")
	defer func() {
		if r := recover(); r != nil {
			fmt.Println("Recovered:", r)
		}
	}()
	panic("Panic!")
}

func main() {
	Example()
	fmt.Println("Main function continues")
}
```

Вывод тут будет следующим:

```go
Inside function
Recovered: Panic!
Main function continues
```

В разработке стоит избегать не только паник, но и восстановления после них. Это следует использовать только в самых крайних случаях. Dont panic!
